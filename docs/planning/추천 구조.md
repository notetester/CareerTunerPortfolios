# CareerTuner 개발 구조 및 환경 가이드

## 1. 문서 목적

이 문서는 CareerTuner의 모노레포 구성, 개발 도구 사용 방식, 프론트엔드·백엔드 경계, 로컬 실행 흐름, 기능별 코드 배치 원칙을 설명한다.

세부 기술 규약은 [`../ARCHITECTURE.md`](../ARCHITECTURE.md), 실제 담당 범위는 [`../TEAM_WORK_DISTRIBUTION.md`](../TEAM_WORK_DISTRIBUTION.md)를 함께 참고한다.

---

## 2. 구조 선택 요약

CareerTuner는 하나의 저장소 안에서 백엔드, 프론트엔드, 문서를 함께 관리하는 모노레포 구조를 사용한다.

```text
CareerTuner/
 ├─ backend/       Spring Boot REST API
 ├─ frontend/      사용자·관리자 React SPA
 ├─ docs/          제품 기획, 아키텍처, 업무 분담 문서
 ├─ AGENTS.md      공통 개발 지침
 └─ README.md      저장소 진입 안내
```

이 구조의 목적은 단순히 폴더를 한곳에 두는 데 있지 않다. 제품 기능 하나를 개발할 때 필요한 사용자 화면, 관리자 화면, API, 데이터 모델, 기획 문서를 같은 저장소에서 일관되게 변경하고 검토하기 위한 구성이다.

---

## 3. 기술 스택

| 영역 | 기술 |
| --- | --- |
| 백엔드 | Spring Boot 4.1.0, Java 21, Spring Security |
| 영속성 | MyBatis, MySQL 8 |
| API 문서 | springdoc-openapi, Swagger UI |
| 프론트엔드 | React 19, Vite 8, TypeScript 7 |
| UI | Tailwind CSS v4, shadcn/ui |
| 라우팅 | react-router 8 |
| 모바일 확장 | 반응형 웹 → PWA → Capacitor |

### 3.1 주요 기술 결정

- 영속성 계층은 MyBatis만 사용하며 JPA를 혼용하지 않는다.
- 백엔드 API는 모두 `/api/**` 경로 아래에 둔다.
- 백엔드 응답은 `ApiResponse<T>` 형식으로 통일한다.
- 사용자 화면과 관리자 화면은 같은 React 앱에서 관리한다.
- 관리자 라우트는 `/admin/**`, 관리자 코드는 `frontend/src/admin/` 아래에 둔다.
- 모바일은 별도 앱을 즉시 만들지 않고 현재 React 앱을 단계적으로 확장한다.

## 4. IntelliJ 기반 개발 환경

### 4.1 프로젝트 열기

IntelliJ IDEA Ultimate에서 `CareerTuner` 루트 폴더를 연다.

```text
File
→ Open
→ CareerTuner
```

루트 폴더를 열면 백엔드 Gradle 프로젝트, 프론트엔드 npm 프로젝트, 문서를 한 창에서 관리할 수 있다.

### 4.2 권장 실행 구성

| 실행 구성 | 대상 | 기본 포트 |
| --- | --- | --- |
| `backend dev` | Spring Boot 애플리케이션 | `8080` |
| `frontend dev` | Vite 개발 서버 | `5173` |
| `frontend typecheck` | TypeScript 타입 검사 | 해당 없음 |

IntelliJ 외의 편집기를 사용해도 되지만, 팀 문서와 실행 흐름은 루트 모노레포를 기준으로 설명한다.

### 4.3 데이터베이스 도구

MySQL은 IntelliJ Database 도구 또는 MySQL Workbench로 연결할 수 있다. 최초 개발 환경 구성 시 다음 순서로 스키마와 시드를 적용한다.

```text
backend/src/main/resources/db/schema.sql
→ backend/src/main/resources/db/data.sql
```

실제 연결 정보와 환경변수는 [`../../backend/README.md`](../../backend/README.md)를 기준으로 확인한다.

## 5. 로컬 실행 흐름

### 5.1 백엔드 실행

PowerShell:

```powershell
cd backend
.\gradlew.bat bootRun
```

macOS 또는 Linux:

```bash
cd backend
./gradlew bootRun
```

기본 주소:

```text
API 서버    http://localhost:8080
헬스 체크   http://localhost:8080/api/health
Swagger UI  http://localhost:8080/swagger-ui.html
```

### 5.2 프론트엔드 실행

```bash
cd frontend
npm install
npm run dev
```

기본 주소:

```text
http://localhost:5173
```

### 5.3 프론트엔드 검증 명령

```bash
npm run typecheck
npm run build
```

개발 중에는 `typecheck`로 타입 오류를 빠르게 확인하고, 기능 단위 완료 시 프로덕션 빌드까지 확인한다.

### 5.4 프론트엔드와 백엔드 연결

개발 환경에서는 브라우저가 Vite 개발 서버에 요청하고, `/api/*` 요청은 Vite 프록시가 Spring Boot로 전달한다.

```text
Browser
  │
  ├─ 화면·정적 자산 ─────────▶ Vite :5173
  │
  └─ /api/* ─▶ Vite proxy ─▶ Spring Boot :8080
```

프론트엔드에서는 개발·운영 환경에 따라 API 호스트를 직접 하드코딩하지 않고 상대 경로를 사용한다.

```ts
fetch("/api/health");
```

## 6. 백엔드 구조

### 6.1 기본 구조

```text
backend/
 ├─ src/main/java/com/careertuner/
 │   ├─ CareerTunerApplication.java
 │   ├─ common/
 │   ├─ auth/
 │   ├─ user/
 │   ├─ profile/
 │   ├─ applicationcase/
 │   ├─ jobposting/
 │   ├─ jobanalysis/
 │   ├─ companyanalysis/
 │   ├─ fitanalysis/
 │   ├─ interview/
 │   ├─ community/
 │   ├─ payment/
 │   ├─ notification/
 │   ├─ ai/
 │   └─ admin/
 ├─ src/main/resources/
 │   ├─ application.yaml
 │   ├─ db/
 │   └─ mapper/
 └─ build.gradle
```

### 6.2 도메인 계층

각 도메인은 다음 계층을 기본으로 한다.

```text
controller
→ service
→ mapper
→ domain
```

필요에 따라 `dto`를 분리한다.

| 계층 | 책임 |
| --- | --- |
| controller | HTTP 요청 검증, 인증 사용자 확인, 응답 형식 구성 |
| service | 비즈니스 규칙, 트랜잭션, 여러 매퍼·외부 서비스 조합 |
| mapper | MyBatis 데이터 접근 인터페이스 |
| domain | 저장 모델과 핵심 데이터 구조 |
| dto | API 요청·응답 전용 구조 |

### 6.3 MyBatis 규칙

```text
Java Mapper:
backend/src/main/java/com/careertuner/<domain>/mapper/*Mapper.java

XML Mapper:
backend/src/main/resources/mapper/<domain>/*Mapper.xml
```

- Java 매퍼와 XML 매퍼의 namespace를 일치시킨다.
- SQL과 결과 매핑은 XML에 명시한다.
- 공통 쿼리라는 이유만으로 여러 도메인의 SQL을 한 매퍼에 모으지 않는다.
- MySQL JSON 컬럼은 초기에는 문자열로 다룰 수 있으며, 구조화 조작이 늘어날 때 TypeHandler 도입을 검토한다.

### 6.4 공통 영역

`common/`은 여러 담당자가 공유하는 코드이므로 변경 영향이 크다.

```text
common/
 ├─ config
 ├─ security
 ├─ web
 └─ exception
```

대표 공통 요소:

- `ApiResponse<T>`
- `ErrorCode`
- `BusinessException`
- `GlobalExceptionHandler`
- `SecurityConfig`
- JWT 인증 필터

공통 영역 수정 전에는 변경 이유, 영향받는 도메인, 마이그레이션 필요 여부를 팀에 공유한다.

### 6.5 AI 코드 배치

AI 공통 인프라와 도메인별 기능을 분리한다.

```text
ai/
 ├─ common/       AI 클라이언트, 공통 응답 처리, 비용·사용량 기반
 └─ prompt/       공통 프롬프트 템플릿 기반

각 도메인/
 └─ service/      해당 도메인 AI 기능과 업무 흐름
```

공고 분석, 적합도 분석, 면접 질문 생성처럼 도메인 책임이 분명한 로직은 해당 도메인에 둔다. 이렇게 하면 공통 AI 폴더의 동시 수정과 결합도를 줄일 수 있다.

## 7. 프론트엔드 구조

### 7.1 기본 구조

```text
frontend/
 ├─ src/
 │   ├─ main.tsx
 │   ├─ app/
 │   │   ├─ App.tsx
 │   │   ├─ routes.ts
 │   │   ├─ pages/
 │   │   └─ components/
 │   ├─ features/
 │   ├─ admin/
 │   └─ styles/
 ├─ public/
 ├─ package.json
 ├─ vite.config.ts
 └─ tsconfig.json
```

### 7.2 사용자 기능 모듈

사용자 기능은 `frontend/src/features/<feature>` 아래에 둔다.

```text
features/applications/
 ├─ pages/
 ├─ components/
 ├─ api/
 ├─ hooks/
 └─ types/
```

| 폴더 | 책임 |
| --- | --- |
| pages | 라우트에 연결되는 화면 구성 |
| components | 기능 내부에서 재사용하는 UI |
| api | API 요청 함수와 응답 변환 |
| hooks | 상태와 비동기 흐름 조합 |
| types | 기능 전용 TypeScript 타입 |

### 7.3 관리자 기능 모듈

관리자 기능은 같은 React 앱의 `frontend/src/admin/` 아래에 둔다.

```text
admin/
 ├─ routes.ts
 ├─ pages/
 ├─ components/
 ├─ hooks/
 ├─ lib/
 └─ features/
     └─ <feature>/
         ├─ pages/
         ├─ components/
         ├─ api/
         ├─ hooks/
         └─ types/
```

사용자 기능과 관리자 기능은 같은 데이터 도메인을 다룰 수 있지만, 화면 목적과 권한이 다르므로 컴포넌트를 무리하게 공유하지 않는다. 순수한 API 타입이나 디자인 시스템 프리미티브처럼 책임이 명확한 요소만 공유한다.

### 7.4 앱 공통 영역

```text
app/
 ├─ routes.ts
 ├─ layouts/
 ├─ providers/
 └─ components/
     ├─ ui/
     └─ layout/
```

라우트, 인증 Provider, 전역 오류 처리, 공통 레이아웃은 앱 공통 영역에 둔다. 기능별 비즈니스 로직을 `app/`에 추가하지 않는다.

### 7.5 스타일과 디자인 시스템

- Tailwind CSS v4와 디자인 토큰을 사용한다.
- shadcn/ui 프리미티브는 공통 UI 기반으로 사용한다.
- 기능별 화면은 공통 토큰을 사용하되, 기능 고유 컴포넌트는 해당 기능 폴더에 둔다.
- 지원 건 분석 결과처럼 제품 고유의 반복 UI는 공통화 필요성과 소유권을 검토한 뒤 배치한다.

## 8. 기능 단위 개발 흐름

기능 하나를 개발할 때는 다음 순서를 권장한다.

```text
1. 기획과 완료 조건 확인
2. 데이터 모델과 API 계약 정의
3. 백엔드 service·mapper·controller 구현
4. API 응답과 오류 케이스 검증
5. 프론트엔드 api·types 구현
6. 화면과 상태 처리 구현
7. 로딩·빈 상태·오류·권한 처리
8. 데스크톱·모바일 확인
9. 관련 문서 갱신
```

기능 구현은 사용자 프론트, 백엔드, 관리자 기능을 하나의 수직 범위로 관리한다. 담당 범위를 벗어나 공통 파일이나 다른 담당자의 도메인을 수정해야 하는 경우 먼저 영향 범위를 공유한다.

## 9. 환경변수와 설정

### 9.1 백엔드

백엔드 민감 설정은 환경변수로 주입한다.

```text
DB_HOST
DB_PORT
DB_NAME
DB_USERNAME
DB_PASSWORD
JWT_SECRET
OAUTH_*_CLIENT_ID
OAUTH_*_CLIENT_SECRET
MAIL_*
APP_FRONTEND_URL
APP_API_BASE_URL
```

개발 기본값과 실제 배포 값의 관리 방법은 `backend/README.md`를 따른다. 실제 운영 비밀값은 저장소에 커밋하지 않는다.

### 9.2 프론트엔드

프론트엔드에 포함되는 환경변수는 브라우저에서 확인할 수 있다는 점을 전제로 한다. 비밀키와 서버 전용 자격 증명은 프론트엔드 환경변수에 넣지 않는다.

## 11. 검증 기준

### 11.1 백엔드

- 애플리케이션이 정상 부팅되는가?
- 관련 API가 `/api/**` 경로와 `ApiResponse<T>` 규약을 지키는가?
- 인증과 소유권 검증이 적용되는가?
- MyBatis 매퍼와 SQL이 정상 동작하는가?
- 오류가 공통 예외 형식으로 반환되는가?

### 11.2 프론트엔드

- `npm run typecheck`가 통과하는가?
- `npm run build`가 통과하는가?
- 로딩, 빈 상태, 오류 상태가 처리되는가?
- 데스크톱과 모바일에서 주요 액션을 수행할 수 있는가?
- 권한과 인증 상태에 맞는 화면이 표시되는가?

### 11.3 통합

- Vite 프록시를 통해 프론트엔드와 백엔드가 연결되는가?
- API 계약과 TypeScript 타입이 일치하는가?
- 사용자 본인의 데이터만 조회·수정할 수 있는가?
- 핵심 사용자 흐름을 처음부터 끝까지 완료할 수 있는가?

```text
초기
CareerTuner/
 ├─ backend/
 ├─ frontend/
 └─ docs/

Capacitor 도입 후
CareerTuner/
 ├─ backend/
 ├─ frontend/
 │   ├─ android/
 │   ├─ ios/
 │   └─ capacitor.config.json
 └─ docs/
```

모바일 확장 기준과 단계별 완료 조건은 [`모바일 고려.md`](모바일%20고려.md)를 따른다.

## 13. 구조 변경 의사결정 기준

새 폴더, 공통 모듈, 별도 앱을 추가하기 전에 다음을 확인한다.

1. 기존 기능 모듈에 둘 수 없는 명확한 책임이 있는가?
2. 두 곳 이상에서 실제로 반복되는 복잡성을 줄이는가?
3. 소유권과 변경 영향이 명확한가?
4. 배포 또는 보안 경계를 분리해야 하는가?
5. 현재 MVP 개발 속도보다 장기 구조 이점이 큰가?

명확한 근거가 없는 구조 확장은 보류한다. 현재 저장소의 기능별 모듈과 단일 프론트엔드 앱 구성을 우선 활용한다.

