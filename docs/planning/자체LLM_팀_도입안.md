## 0. 용어 정리 (먼저 읽기)

"LLM을 직접 만든다"는 세 가지로 갈린다. 강사가 요구하는 건 거의 확실히 ②번이다.

| 방식 | 의미 | 우리에게 |
| --- | --- | --- |
| ① 사전학습(pre-training) | 빈 모델을 0부터 학습 (수조 토큰, GPU 수백 대, 수억원) | **불가능** (학생팀) |
| ② **파인튜닝(fine-tuning)** | 공개 오픈모델을 우리 데이터로 추가 학습해 "우리 모델"로 | **이것이 우리 계획** |
| ③ API/프롬프트 | 남의 모델 그대로 호출 | "직접 만든 것" 아님 (현재 상태) |

→ 본 문서의 "자체 LLM" = **②파인튜닝**. 오픈모델(gemma/Qwen 등)을 우리 데이터로 LoRA 파인튜닝한다.

## 1. 배경과 목표

### 1-1. 왜 자체 LLM인가 (문제의식)
현재 34개 AI 기능(`TEAM_WORK_DISTRIBUTION.md` 3장)이 **전부 외부 API 호출**이다.
= 위 표의 ③상태. 평가/포트폴리오에서 "직접 만든 모델이 없다"가 최대 약점.
→ 각 담당이 자기 도메인 모델을 직접 파인튜닝해 ②로 끌어올린다.

### 1-2. 핵심 원칙과 *그 이유*
| 원칙 | 왜 |
| --- | --- |
| 목적 = "직접 만든 증거" (품질 1등 아님) | 소형 모델이 GPT급 품질을 낼 수 없음. 무리하면 일정·비용 폭발. "만들어 붙였다"가 평가 포인트 |
| 폴백 필수 (외부 API로 자동 전환) | 자체 모델이 죽거나 헛소리할 때 데모가 멈추면 안 됨. 안정성 보장 |
| 무료 우선 (Colab 학습 + 로컬 서빙) | 학생 예산. 유료 GPU 상시 운영은 비현실 |
| 워크플로우 통일 | 6명이 같은 절차·스크립트를 쓰면 학습 비용이 1/6로 분산되고 서로 도울 수 있음 |

### 1-3. 한 줄 목표
> "6개 도메인 각각에 직접 파인튜닝한 소형 자체 모델을 붙이고, 외부 API는 폴백으로 둔다."

## 2. 공통 워크플로우 (모든 담당 동일)

```text
[1] 데이터   합성(Gemini 선생) → JSONL   /   실사용 데이터 누적 시 export JSONL
        ↓     ※ DB 안 거침. 합성은 파일로만 (운영 DB 오염 방지)
[2] 학습     Colab 무료 T4 → LoRA 파인튜닝(공통 베이스) → 1~2시간
        ↓
[3] 변환     LoRA 머지 → GGUF (Ollama 서빙용)
        ↓
[4] 서빙     로컬 Ollama (F 인프라를 ai/common 으로 승격)
        ↓
[5] 연결     백엔드 AiClient 추상화 (provider = oss | openai)
        ↓
[6] 폴백     자체 모델 실패·저품질 → 외부 API 자동 전환
```

**왜 "합성 데이터 증류"부터?** 학습 데이터가 지금 0건이라서. 실사용 데이터가 쌓일 때까지 기다리면
1주에 못 끝난다. 큰 모델(선생)이 만든 예시로 작은 모델(학생)을 가르치는 게(distillation) 유일한 콜드스타트 길.

## 3. 기술 선택 — 이유와 장단점 (★논의 핵심★)

> 각 항목: **선택지 비교 → 추천 → 이유 → [팀 결정]**. 추천은 "이렇게 제안한다"이지 확정이 아니다.

### 3-1. 선생 모델 (합성 데이터 생성용)

합성 데이터를 누가 만들지. **싼데 품질 적당한** 큰 모델이면 된다(데이터 생성은 대량 호출이라 단가가 핵심).

| 후보 | 단가(in/out $/1M) | 장점 | 단점 |
| --- | --- | --- | --- |
| **Gemini 2.5 Flash-Lite** | 0.10 / 0.40 | **최저가**, 무료 티어 有, 1M 컨텍스트 | 품질이 최상은 아님(소형 선생) |
| Gemini 2.5 Flash | 0.30 / 2.50 | 품질↑, 여전히 저렴 | Flash-Lite보다 비쌈 |
| Claude Haiku 4.5 | 1.00 / 5.00 | 안정적 품질 | Flash 계열보다 비쌈 |
| (참고) Gemini 3.5 Flash | 1.50 / 9.00 | 고품질 | **오히려 Haiku보다 비쌈 → 데이터생성엔 부적합** |

- **추천: 기본 Gemini 2.5 Flash-Lite.** 이유 = **싸서**(데이터 수천 건 생성에 단가가 곧 총비용). 무료 티어로 0원도 가능.
- **단, 품질이 중요한 task(채점·첨삭)는 Gemini 2.5 Flash로 상향** 검토 — 선생 품질이 학생의 천장이라서.
- 합성 데이터는 가짜라 프라이버시 부담 없음 → 무료 티어 적극 활용.

> **[팀 결정]** ① 선생 모델 기본값(Flash-Lite vs Flash) ② 까다로운 task만 상향할지 ③ 무료 티어 vs Batch 유료

### 3-2. 학생 베이스 모델 (우리가 파인튜닝할 오픈모델) ★가장 중요한 결정★

우리 서비스는 **한국어**다. 그래서 "한국어 품질"과 "모델 크기(서빙 부담)"의 **트레이드오프**가 핵심.
또 발표/배포를 생각하면 **라이선스**도 봐야 한다.

| 후보 | 크기 | 한국어 | 라이선스 | 장점 | 단점 |
| --- | --- | --- | --- | --- | --- |
| **Qwen2.5-3B-Instruct** | 3B | 양호(다국어 강함) | **Qwen Research License** | 한국어+크기 균형 | 비상업 연구·평가 중심, 상용 사용 전 별도 허가 필요 |
| **gemma-2-2b-it** | 2B | 보통~양호 | Gemma Terms(대체로 자유) | **가장 가벼움, F가 이미 셋팅** | 한국어 살짝 아쉬움, 라이선스 조항 有 |
| EXAONE 3.5 2.4B | 2.4B | **최상(한국 특화)** | EXAONE License(**상업 제한 주의**) | 한국어 1등급 | 라이선스가 상업·배포에 제약 → 확인 필수 |
| Llama 3.2 3B Instruct | 3B | 보통(영어 위주) | Llama Community(조항 有) | 생태계 큼 | 한국어 약함, 라이선스 조항 |
| Phi-3.5-mini | 3.8B | 약함 | MIT(자유) | 영어·추론 강함 | **한국어 약함 → 우리엔 부적합** |
| Qwen2.5-1.5B | 1.5B | 보통 | Apache 2.0 | 초경량, CPU 서빙 빠름 | 품질↓(복잡 task 한계) |

**핵심 트레이드오프 두 개:**
1. **한국어 vs 크기** — 한국어 서비스라 한국어가 중요. 단 클수록 로컬 서빙(CPU/내장GPU)이 느려짐.
   → Qwen2.5-3B가 "한국어 양호 + 로컬 서빙 가능"의 균형점.
2. **F의 gemma 2B 유지 vs Qwen으로 통일** —
   · F는 검열에 gemma를 **프롬프트로** 쓰는 중(파인튜닝 아님). 즉 *베이스를 바꿔도 F 작업 손실 거의 없음*.
   · 통일하면 학습 노트북·GGUF 변환·서빙 절차를 6명이 그대로 복사 → 비용 급감.
   · gemma 유지하면 F는 익숙하지만, 한국어/라이선스에서 Qwen이 우위.

- **당시 추천: 공통 베이스 = Qwen2.5-3B-Instruct.** 이유 = 한국어 양호 + 3B(로컬 서빙 가능).
  **2026-07-14 정정:** 3B 모델은 Apache 2.0이 아니라 Qwen Research License다. 포트폴리오 연구·평가와
  상용 배포를 구분하고, 상용 사용 전 별도 허가 또는 상업 친화 베이스 재학습을 게이트로 둔다.
  EXAONE이 한국어는 최강이나 **라이선스 제약** 때문에 발표·배포 안전성에서 Qwen을 앞세움.
- **대안: gemma 2B 통일** — 서빙이 제일 가볍고 F가 익숙. 한국어가 아쉬우면 데이터로 보완.

> **[팀 결정]** ① 공통 베이스 1종 통일 여부 ② Qwen2.5-3B vs gemma 2B vs (EXAONE 라이선스 확인 후 검토)
> ③ 통일 시 F의 검열은 기존 gemma 유지할지 / 베이스로 옮길지

### 3-3. 서빙 방식 (학습된 모델을 어떻게 켜놓나)

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| **Ollama** | 설치 쉬움, **F가 이미 구축**, CPU/로컬 OK, GGUF 사용 | 고성능·동시처리 약함 |
| vLLM | 빠름, OpenAI 호환 API, 동시처리 강함 | **GPU 필요**, 셋업 어려움, 상시 GPU 비용 |

- **추천: Ollama.** 이유 = 로컬·무료로 돌고, **F 인프라가 이미 있어서** 재활용하면 됨. 데모/발표엔 충분.
- vLLM은 상시 GPU가 필요해 무료 원칙과 충돌 → 제외(필요 시 후순위).

> **[팀 결정]** Ollama로 확정해도 되는지 (이견 없으면 확정)

### 3-4. 학습 환경 (어디서 파인튜닝하나)

| 후보 | 비용 | 장점 | 단점 |
| --- | --- | --- | --- |
| **Colab 무료(T4 16GB)** | 0원 | 공짜, 3B 4bit LoRA 가능 | 세션 12h·유휴90분 끊김, GPU 미배정 가능 |

### 3-5. 모델 구성 방식 (모델을 몇 개 만드나)

| 방식 | 장점 | 단점 |
| --- | --- | --- |
| **담당자별 1모델 (6개)** | 발표 그림 명확("각자 자체 모델"), 도메인 격리 | Ollama에 6개 등록(요청 시 로드로 해결) |
| 팀 공통 1모델 (전 task) | 관리 단순 | 도메인 충돌, 한 모델이 다 잘하기 어려움 |

  공통 베이스로 통일하면 6개라도 절차는 동일(복사).

> **[팀 결정]** 담당자별 1모델로 갈지 (이견 없으면 확정)

### 3-6. 데이터 전략

- **콜드스타트 = 합성 데이터(Gemini 선생)**. 실사용 데이터 0건이라 합성부터.
- 운영 DB(team1_db, 원격 공유)에 **합성 데이터를 넣지 않는다** → JSONL 파일로만. (가짜 데이터로 운영 DB·타 팀원 화면 오염 방지)
- 실사용 데이터가 쌓이면 export 해서 합성분과 합쳐 **재학습**(품질 향상).

> **[팀 결정]** 합성 데이터 JSONL 보관 위치 통일(공유 폴더/repo 경로)

## 4. 담당자별 자체 모델 설계

> AI 기능 34개(`TEAM_WORK_DISTRIBUTION.md` 3장)를 task 성격으로 묶어 매핑.
> "적합도" = 소형 모델 + 합성데이터로 학습 시 기대 성과(추출/분류/요약 高 · 생성/첨삭 中 · 추론/추천 中下).

### A — 회원/프로필 (이력서·스펙 추출) · 1~5
요약(1)·기술스택 추출(2)·키워드 추출(3,4)·완성도 진단(5). **추출/요약 도메인 = 적합도 최상.**
- 1순위: 기술스택 추출(2). 콜드스타트 모범 사례로 적합(입출력 명확).

### B — 공고/기업 분석 · 6~11
공고 분석(6)·조건 추출(7,8)·업무 요약(9)·기업 요약(10)·면접 포인트(11). **구조화 추출 = 적합.**
- 1순위: 공고문 문장 분류 + 필수/우대 조건 추출(7,8).

B의 자체 AI 적용은 단일 소형 LLM으로 공고 분석 전체를 바로 대체하는 방식이 아니라, 공고문 텍스트를 단계적으로
구조화하는 하이브리드 방식으로 진행한다.

```text
OCR 또는 PDF/URL 텍스트 추출
→ 텍스트 정제와 문장 분리
→ KLUE-RoBERTa 기반 문장 분류
→ Qwen/gemma 계열 소형 모델로 공고 분석 JSON 생성 시도
→ 실패 또는 저품질이면 기존 대형 LLM으로 폴백
```

1차 목표는 공고문 문장을 필수 조건, 우대 조건, 담당 업무, 기술 스택 등으로 분류하고, 그 결과를 기존 대형 LLM
공고 분석 입력 보조 데이터로 쓰는 것이다. KLUE-RoBERTa는 공개 배포된 한국어 사전학습 모델이며, CC BY-SA 4.0
조건을 확인해 출처를 명시하고 문장 분류 실험에 활용한다.

### C — 홈/취업 분석 (커리어 전략 추천) · 12~18
적합도 분석(12)·부족역량/로드맵/자격증 추천(13,14,15)·경향 분석(16,17)·대시보드 요약(18).
- 1순위: 대시보드 요약(18, 요약이라 적합). 추천 계열은 **RAG(자격증/로드맵 DB) 결합** 권장.

### D — 가상 면접 (AI 면접관) · 19~23  ★상세안 별도★
6 task 멀티태스크 + 자율 에이전트 Planner. QGEN·PROBE·TALK·EVAL·REPORT·**PLAN**.
- 상세: `면접 자율 에이전트 로드맵.md` + `interview-llm-plan.md`(OneDrive 공유).
- 1순위: 채점(22). **차별점**: Planner(다음 행동 결정)까지 자체 모델.

### E — 첨삭 (답변·자소서 첨삭) · 24~28  ★추천안★
**"텍스트 첨삭/개선 전용 자체 모델"** — 면접답변(24)·자소서(25)·이력서(26)·포폴(27) 멀티태스크.
- 합성 데이터: Gemini가 **(약점 있는 원문 → 개선문 + 이유)** before/after 쌍 생성.
- **D(채점)↔E(첨삭) 연계**: 평가 모델이 약점 진단 → 첨삭 모델이 수정. 한 흐름으로 발표 강화.
- 28(요금제 추천)은 수치 기반 → LLM 말고 규칙엔진 권장(자체모델 제외).
- 적합도: 첨삭은 생성 품질 중요 → 폴백 비중↑. 단 "직접 학습 증거"는 확보.

E도 팀 공통 베이스인 Qwen2.5-3B-Instruct를 사용한다. 자기소개서, 면접 답변, 이력서, 포트폴리오 설명을 하나의 E LoRA 모델로 학습하며, Qwen3-8B와 Qwen3-4B 도메인 예외는 사용하지 않는다.
첨삭 품질은 모델 크기를 늘리는 대신 장문 데이터 보강, strict JSON Schema, 동일 3B repair 1회, 서버 검증과 외부 API fallback으로 보완한다.
운영 모델 태그는 `careertuner-e-correction-3b:latest`로 통일하고, 출력은 원문 보존 여부, 변경 이유, 위험 표현, 허위 사실 추가 여부를 JSON으로 검증한다.

### F — 커뮤니티/고객센터 · 29~34  ★진행 최다★
후기요약(29)·태그추천(30)·질문추출(31)·게시글추천(32)·**부적절 분류(33, 구현완료)**·문의답변(34).
- 현황: Ollama+gemma 검열 **구현 완료**(`community/moderation/`, 현재 프롬프트 기반).
  → 파인튜닝(33 분류)으로 승격하면 "자체 모델" 기준 충족. **F 인프라가 팀 공통 서빙의 출발점.**
- 1순위: 분류(33) → 후기요약(29)/태그(30)/질문추출(31). 추출·분류라 적합도 높음.

**콜드스타트 유리 순서**: 추출·분류·요약 도메인(**A·B·F**)부터 성공시켜 워크플로우 검증 →
생성·첨삭·추론(**C·D·E**)은 폴백 두텁게 깔고 진행.

## 9. 멘토 발표 포인트

1. **문제의식**: 34개 AI가 전부 외부 API → 직접 만든 모델이 없음을 자각.
2. **방법론**: 6명이 동일 파이프라인(합성 증류→Colab 무료 LoRA→로컬 Ollama→폴백)으로 각자 모델 제작.
3. **선택의 근거**: 모델·도구를 *왜* 골랐는지(가격/한국어/라이선스/서빙) 표로 설명.
4. **증거**: 외부 API vs 자체 모델 **나란히 비교표**(도메인별 점수·일관성·비용).
5. **비용 효율**: GPU 임대 없이 총 5천원 안팎에 6개 모델.
6. **심화(D)**: Planner까지 자체 모델 = 자율 에이전트 두뇌.
7. **연계(D↔E)**: 자체 모델이 채점하고 자체 모델이 첨삭.
