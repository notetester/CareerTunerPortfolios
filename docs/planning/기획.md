## 2. 제품 정의

### 2.1 한 문장 정의

> CareerTuner는 사용자가 등록한 채용공고와 개인 경력 정보를 AI가 비교·분석하여, 공고별 지원 전략 수립부터 모의면접과 답변 개선까지 지원하는 취업 준비 플랫폼이다.

### 2.2 해결하려는 문제

취업 준비 과정에서 사용자는 다음 문제를 반복적으로 경험한다.

1. 채용공고의 필수 조건과 우대 조건을 읽어도 실제 준비 우선순위를 정하기 어렵다.
2. 자신의 경력과 기술이 해당 공고에 얼마나 부합하는지 객관적으로 판단하기 어렵다.
3. 이력서, 자기소개서, 포트폴리오, 면접 답변이 서로 분리되어 관리된다.
4. 예상 질문을 준비하더라도 답변의 구체성, 논리성, 직무 연관성을 스스로 평가하기 어렵다.
5. 여러 공고에 지원하며 반복적으로 나타나는 부족 역량과 개선 추세를 파악하기 어렵다.

CareerTuner는 이 과정을 하나의 연속된 준비 흐름으로 연결한다.

```text
채용공고 등록
→ 공고 요구사항 구조화
→ 사용자 프로필과 비교
→ 부족 역량과 준비 전략 도출
→ 예상 질문 생성
→ 모의면접 진행
→ 답변 평가 및 첨삭
→ 장기 취업 경향 분석에 반영
```

### 2.3 제품 포지셔닝

CareerTuner는 채용공고를 수집하거나 기업 지원을 중개하는 구인구직 플랫폼이 아니다. 사용자가 선택한 공고를 바탕으로 지원 준비의 품질을 높이는 **AI 취업 전략 플랫폼**이다.

| 구분 | 기존 구인구직 플랫폼 | CareerTuner |
| --- | --- | --- |
| 주요 목적 | 공고 탐색과 입사 지원 | 선택한 공고에 대한 준비 최적화 |
| 핵심 데이터 | 채용공고 | 지원 건과 준비 이력 |
| 사용자 행동 | 검색, 저장, 지원 | 분석, 비교, 연습, 개선 |
| AI 활용 | 공고 추천, 서류 보조 | 공고 분석, 전략 수립, 면접 평가, 장기 추세 분석 |
| 성과 기준 | 지원 완료 | 준비도와 답변 품질 향상 |

### 2.4 제품명과 브랜드 메시지

- 제품명: **CareerTuner**
- 핵심 메시지: **채용공고에 맞춰 내 스펙과 면접 답변을 조정하는 AI 취업 전략 플랫폼**
- 랜딩페이지 핵심 문구: **공고를 등록하면, AI가 지원 전략을 설계합니다.**

제품 문구는 합격을 보장하거나 채용 결과를 예측한다고 표현하지 않는다. 서비스의 역할은 사용자가 자신의 경험을 더 정확히 정리하고 효과적으로 전달하도록 돕는 데 있다.

## 3. 제품 목표와 비목표

### 3.1 제품 목표

1. 사용자가 채용공고의 핵심 요구사항을 빠르게 이해하도록 돕는다.
2. 사용자 프로필과 공고의 차이를 설명 가능한 형태로 보여준다.
3. 부족 역량을 실제 실행 가능한 준비 과제로 전환한다.
4. 공고와 사용자 경험을 근거로 예상 질문과 모의면접을 제공한다.
5. 면접 답변의 논리성, 구체성, 직무 연관성을 반복적으로 개선한다.
6. 여러 지원 건의 데이터를 종합하여 장기적인 취업 준비 방향을 제시한다.

### 3.2 비목표

초기 제품은 다음 기능을 목표로 하지 않는다.

- 외부 채용 플랫폼의 공고를 대량 수집하거나 재배포하는 기능
- 기업을 대신해 지원자를 평가하거나 자동 탈락시키는 기능
- AI 점수를 근거로 합격 가능성을 보장하거나 단정하는 기능
- 얼굴, 표정, 음성 특성을 근거로 성격이나 직무 적합성을 판정하는 기능
- 모든 직군에 대해 전문가 수준의 자격·법률·노무 판단을 제공하는 기능

## 4. 목표 사용자

### 4.1 핵심 사용자군

| 사용자군 | 주요 상황 | 핵심 요구 |
| --- | --- | --- |
| 신규 취업 준비생 | 경험을 직무 요구사항과 연결하는 데 어려움 | 공고 해석, 부족 역량 파악, 답변 구조화 |
| 경력 이직 준비자 | 기존 경험을 새로운 직무와 기업에 맞게 재구성해야 함 | 경험 매칭, 이직 사유와 성과 설명, 심화 질문 대비 |
| 직무 전환 준비자 | 보유 역량 중 전환 가능한 부분과 부족한 부분을 구분해야 함 | 전환 가능 역량 설명, 학습 우선순위, 포트폴리오 전략 |
| 반복 면접 준비자 | 여러 번 연습하지만 개선 지점을 객관화하기 어려움 | 점수 변화, 반복 약점, 답변 첨삭 |

### 4.2 대표 사용자 시나리오

**시나리오 A: 신규 지원 건 준비**

사용자는 관심 있는 채용공고를 텍스트로 붙여넣고 새 지원 건을 생성한다. CareerTuner는 필수 기술, 우대 조건, 담당 업무를 추출하고 사용자 프로필과 비교한다. 사용자는 강점과 부족 역량을 확인한 뒤, 준비 과제와 예상 질문을 바탕으로 모의면접을 진행한다.

**시나리오 B: 기존 경험 재구성**

경력 이직자는 특정 공고에서 요구하는 문제 해결 경험과 성과 지표를 확인한다. CareerTuner는 사용자의 프로젝트 기록 중 연관성이 높은 경험을 찾아 답변 구성에 활용하도록 제안한다.

**시나리오 C: 장기 준비 방향 점검**

사용자는 여러 지원 건을 진행한 뒤 취업 분석 화면에서 반복적으로 부족한 기술, 자주 지원한 직무, 면접 점수 변화, 답변의 공통 약점을 확인한다. 분석 결과를 바탕으로 다음 학습 계획과 지원 방향을 조정한다.

## 5. 핵심 제품 원칙

### 5.1 지원 건 중심

CareerTuner의 핵심 단위는 채용공고 자체가 아니라 **지원 건(Application Case)** 이다.

지원 건은 특정 기업, 공고, 직무에 대한 하나의 독립된 준비 작업공간이다. 동일한 기업이라도 직무나 공고 시점이 다르면 별도 지원 건으로 관리한다.

```text
사용자
 ├─ A기업 / 프론트엔드 개발 / 2026 상반기
 ├─ A기업 / 서비스 기획 / 2026 상반기
 ├─ B기업 / 전산직 / 수시채용
 └─ C기업 / 마케팅 기획 / 2026 신입공채
```

지원 건 안에는 다음 정보가 연결된다.

| 영역 | 포함 정보 |
| --- | --- |
| 기본 정보 | 기업명, 직무명, 지원 상태, 마감일, 등록일 |
| 공고 원문 | 텍스트, 파일, 추출 텍스트, 출처 |
| 공고 분석 | 담당 업무, 필수 조건, 우대 조건, 핵심 역량 |
| 기업 분석 | 기업 개요, 주요 사업, 최근 이슈, 면접 활용 포인트 |
| 적합도 분석 | 강점, 부족 역량, 근거, 보완 우선순위 |
| 지원 전략 | 강조할 경험, 준비 과제, 학습 및 자격증 제안 |
| 면접 준비 | 예상 질문, 면접 세션, 답변, 평가 리포트 |
| 첨삭 기록 | 개선 전후 답변, 자기소개서·포트폴리오 설명 개선안 |

### 5.2 분석 결과는 설명 가능해야 함

적합도 점수나 추천 결과는 단독 숫자로 제공하지 않는다. 점수는 AI가 공고 요구 조건과 사용자 스펙을 비교해
산정하되, 사용자에게는 숫자 점수와 등급/구간을 함께 보여주는 하이브리드 형식으로 제공한다.
각 결과에는 공고 원문과 사용자 프로필에서 어떤 근거를 사용했는지 함께 제시한다.

```text
판단: REST API 연동 경험은 강점으로 분류
공고 근거: "외부 API 연동 및 데이터 처리 경험"
사용자 근거: "팀 프로젝트에서 결제 API 연동 담당"
보완 제안: 장애 처리와 재시도 전략을 답변에 포함
```

### 5.3 AI는 초안을 제시하고 사용자가 확정함

AI가 추출하거나 생성한 내용은 사용자가 확인하고 수정할 수 있어야 한다. 특히 공고 요구사항, 프로필 정보, 기업 정보, 첨삭 결과는 사실과 다를 수 있으므로 확정 전 검토 단계를 제공한다.

### 5.4 실행 가능한 결과를 제공함

분석은 단순 평가에서 끝나지 않고 다음 행동으로 연결되어야 한다.

- 부족 역량 → 학습 과제와 우선순위
- 약한 답변 → 보완해야 할 근거와 개선 답변
- 낮은 준비도 → 오늘 수행할 작업
- 반복 약점 → 장기 개선 계획

### 5.5 민감한 판단을 제한함

표정, 시선, 자세, 음성 분석은 면접 태도 개선 참고자료로만 사용한다. 이를 성격, 능력, 합격 가능성, 직무 적합성 판단에 사용하지 않는다.

## 6. 전체 사용자 흐름

### 6.1 최초 진입 흐름

```text
회원가입 또는 로그인
→ 기본 프로필 입력
→ 이력서·경력·프로젝트 등록
→ 첫 지원 건 생성
→ 공고문 등록
→ AI 분석 결과 검토
→ 준비 전략 확인
→ 예상 질문 생성
→ 모의면접 진행
→ 리포트 확인 및 답변 개선
```

최초 경험에서는 프로필 입력 부담을 최소화해야 한다. 사용자는 기본 정보와 핵심 경험만 입력해도 첫 분석을 시작할 수 있어야 하며, 부족한 프로필 항목은 분석 과정에서 추가 입력하도록 유도한다.

### 6.2 반복 사용 흐름

```text
대시보드에서 진행 중 지원 건 확인
→ 오늘의 준비 과제 수행
→ 모의면접 또는 첨삭 진행
→ 준비도와 약점 변화 확인
→ 다음 지원 건 또는 학습 계획에 반영
```

### 6.3 지원 건 생명주기

| 상태 | 설명 | 주요 액션 |
| --- | --- | --- |
| 초안 | 공고와 기본 정보만 등록된 상태 | 공고 내용 보완, 분석 시작 |
| 분석 중 | AI 분석이 진행 중인 상태 | 진행 상태 확인, 취소·재시도 |
| 준비 중 | 분석 완료 후 지원 준비를 수행하는 상태 | 전략 확인, 질문 생성, 면접 연습 |
| 지원 완료 | 실제 지원을 완료한 상태 | 면접 일정·결과 기록 |
| 종료 | 채용 절차가 종료된 상태 | 결과 회고, 장기 분석 반영 |

지원 상태는 진행 상태만 표현한다. 보관과 삭제는 상태가 아니라 별도 시각 컬럼으로 관리한다.

| 속성 | 의미 | 목록 규칙 |
| --- | --- | --- |
| `archived_at` | 보관 여부와 보관 시점 | 보관함은 `archived_at IS NOT NULL AND deleted_at IS NULL` |
| `deleted_at` | 소프트 삭제 여부와 삭제 시점 | 삭제된 지원 건은 `deleted_at IS NOT NULL` |

기본 목록은 `archived_at IS NULL AND deleted_at IS NULL`인 지원 건만 표시한다.
삭제는 DB row 삭제가 아니라 소프트 삭제이며, 일반 기능 구현 중 삭제 정책을 임의로 변경하지 않는다.

```text
 ├─ 분석 리포트    하위: 적합도 분석, 장기 취업 분석, 대시보드 AI 분석 결과
 ├─ 커뮤니티      하위: 후기, 질문 공유, 신고/정책
 ├─ 요금제        하위: 플랜, 크레딧, AI 사용량, 결제 내역
 └─ 사용자 메뉴    프로필, 알림, 결제/구독, 설정, 로그아웃

관리자 영역
 ├─ 운영 대시보드
 ├─ 사용자·동의 관리
 ├─ AI 사용량·프롬프트 관리
 ├─ 결제·요금제 관리
 ├─ 커뮤니티·신고 관리
 └─ 고객지원·공지 관리
```

지원 건과 직접 연결되는 첨삭 기능은 지원 건 상세에서 우선 제공한다. 공통 첨삭 도구가 필요한 경우에만 별도 진입점을 제공한다.

### 8.5 기업 분석

기업 분석은 지원 전략의 품질을 높이는 핵심 기능이다. 기본 정책은 웹 검색, 외부 API, 수동 입력 출처를
함께 허용하는 것이다. 가능하다면 실제 기업의 공개 정보를 분석하여 기업 요구사항과 직무 맥락을 추출하고,
그 결과를 지원 전략과 면접 준비에 반영한다.

사용자가 명시적으로 요청한 경우에는 사용자 입력 자료만 사용한 제한 분석도 제공할 수 있다. 다만 이 방식은
기업 최신성, 사업 맥락, 직무 연결점이 부족할 수 있으므로 기본 권장 방식은 아니다.

| 출력 영역 | 설명 |
| --- | --- |
| 기업 개요 | 주요 사업, 제품·서비스, 산업군 |
| 최근 동향 | 최근 공개 정보와 주요 변화 |
| 직무 연결점 | 해당 직무가 기업 사업에 기여할 수 있는 지점 |
| 면접 활용 포인트 | 지원동기와 질문에 활용할 수 있는 사실 |
| 출처 | 확인 시점과 참고한 공개 출처 |

기업 정보는 최신성이 중요하므로 생성 시점과 출처를 함께 표시하고, 확인되지 않은 추론은 사실과 구분한다.
재조회가 필요한 경우 사용자는 기존 분석을 보존한 채 새 분석을 실행할 수 있어야 한다.

적합도 점수는 절대적인 합격 확률이 아니다. 점수는 AI가 공고 요구 조건과 사용자 스펙을 비교해 산정하는

```text

적합도 64점 / 보완 필요
적합도 41점 / 준비 부족
```

점수 구간과 등급명은 운영 중 조정될 수 있지만, 숫자와 구간을 함께 보여주는 하이브리드 형식은 기본 원칙으로 둔다.

### 8.8 첨삭

첨삭은 사용자의 경험을 허위로 만들거나 과장하는 것이 아니라, 이미 보유한 경험을 더 명확하고 관련성 있게 전달하도록 돕는 기능이다.

| 첨삭 대상 | 주요 개선 요소 |
| --- | --- |
| 면접 답변 | 질문 대응성, STAR 구조, 직무 연결, 구체적 결과 |
| 자기소개서 | 문항 의도, 근거 경험, 문단 구조, 중복 표현 |
| 이력서 | 성과 중심 표현, 가독성, 공고 키워드 연결 |
| 포트폴리오 설명 | 역할, 기술 선택 이유, 문제 해결, 결과 |

모든 개선안은 원문, 문제점, 개선 원칙, 수정 예시를 함께 제공한다. 사용자가 사실과 다른 내용을 추가하지 않도록 확인 안내를 표시한다.

### 8.9 장기 취업 경향 분석

장기 분석은 여러 지원 건과 면접 기록을 종합한다.

```text
지원 건별 분석
 ├─ 특정 공고의 강점과 부족 역량
 └─ 해당 지원 건의 면접·첨삭 결과

사용자 전체 분석
 ├─ 자주 지원하는 직무와 산업
 ├─ 반복적으로 부족한 역량
 ├─ 자주 활용되는 강점 경험
 ├─ 면접 점수와 답변 품질 변화
 └─ 다음 지원 및 학습 방향
```

장기 분석은 처음부터 시작할 수 있다. 다만 데이터가 부족하면 빈 차트나 단정적 결론을 보여주지 않고,
어떤 데이터가 있어야 분석 가능한지 안내한다.

예:

```text
아직 면접 기록이 없어 면접 점수 변화는 분석할 수 없습니다.
지원 건 1개가 더 쌓이면 반복 부족 역량을 더 안정적으로 비교할 수 있습니다.
첨삭 기록이 없어 답변 품질 변화는 표시하지 않습니다.
```

| 영역 | 내용 |

| 진행 현황 | 활성 지원 건, 지원 완료, 예정된 면접 |
| 준비도 | 평균 준비도, 최근 변화, 주요 강점과 약점 |
| 오늘의 할 일 | 마감일과 준비 우선순위를 반영한 작업 |
| 최근 활동 | 분석, 첨삭, 면접 기록 |
| 사용량 | 보유 크레딧, 최근 AI 사용 내역 |

### 8.11 커뮤니티

커뮤니티는 핵심 분석 기능을 보조하는 영역이다.

```text
취업 후기
면접 후기
직무별 질문 공유
포트폴리오 피드백
자격증 후기
자유게시판
```

면접 후기에는 기업명 공개 여부와 익명 여부를 선택할 수 있어야 한다. 개인정보, 기밀 정보, 허위 정보, 혐오·비방 콘텐츠를 관리하기 위한 신고와 운영 정책이 필요하다.

### 8.12 결제와 크레딧

AI 기능은 기능별 비용 차이가 크므로 구독과 크레딧을 혼합할 수 있다.
아래 정책은 현재 제품 정책 초안이며, 실제 AI 모델, 토큰 비용, 영상 처리 비용, 회의 결과에 따라 변경될 수 있다.

| 과금 단위 | 적용 예시 |
| --- | --- |
| 무료 제공량 | 제한된 무료 크레딧, 공고 분석 보관 1건, 제한된 예상 질문, 체험 면접 |
| 구독 플랜 | 월별 분석·면접 제공량, 고급 리포트 접근 |
| 크레딧 | 기업 조사, 긴 문서 분석, 음성·영상 분석 등 고비용 기능 |

실제 결제·크레딧 기능을 활성화하는 2차 단계 전에는 크레딧 장부와 사용량 기록을 먼저 구현한다.
무제한 표현은 사용하지 않으며, 플랜별 제공량과 차감 기준을 명확히 표시한다.

기본 정책 초안:

- 무료 사용자는 제한된 무료 크레딧을 받는다.
- 무료 사용자는 공고 분석 보관을 1건으로 제한한다.
- AI 기능별 크레딧 차감량은 다르게 책정한다.
- AI 영상 면접은 음성·텍스트 기능보다 훨씬 많은 크레딧을 소비하는 고비용 기능으로 분류한다.
- AI 요청이 시스템 오류, 모델 오류, 네트워크 실패 등으로 정상 결과를 반환하지 못하면 기본적으로 차감하지 않는다.
- 결제 요청 직후가 아니라 구매 확정 또는 결제 성공 검증 후 크레딧을 지급한다.
- 구매 확정 후에는 기본적으로 환불 불가로 운영한다. 단, 법적 의무, 중복 결제, 시스템 오류 등 예외는 별도 환불 정책으로 처리한다.
- 정확한 제공량, 차감량, 환불 예외는 AI 모델과 운영 비용이 확정되기 전까지 회의로 변경될 수 있다.

### 8.13 관리자 기능

관리자 기능은 운영 안정성과 AI 품질 개선을 지원한다.
사용자 기능이 완료될 때 관련 관리자 화면과 관리자 API도 함께 완료한다. 관리자 기능은 빈 메뉴가 아니라
운영자가 데이터 목록, 상세, 상태, 비용, 실패, 신고, 동의, 변경 이력을 확인하고 필요한 조치를 할 수 있는 수준을 뜻한다.

```text
회원과 권한 관리
결제·환불·요금제 관리
AI 사용량과 비용 모니터링
프롬프트 템플릿과 버전 관리
분석 실패 및 재시도 관리
커뮤니티 게시글·댓글·신고 관리
공지사항·FAQ·고객문의 관리
약관·동의 버전 관리
```

관리자 권한은 `SUPER_ADMIN` 여부 하나로만 나누지 않는다. 일반 `ADMIN`은 회원, 보안, 결제,
콘텐츠, AI 운영, 정책처럼 담당 영역별로 조회·생성·수정·삭제 권한을 따로 부여받는다. 조회 권한이
없는 영역은 메뉴와 화면을 노출하지 않고, 생성·수정·삭제 버튼도 각각의 행위 권한이 있을 때만
표시한다. `SUPER_ADMIN`은 권한 체계를 개발·복구·운영할 수 있도록 세부 권한과 무관하게 전체 기능을
사용하며, 다른 관리자에게 개별 권한이나 역할별 권한 템플릿을 배정·회수할 수 있다. 감사 로그는
변조 대상이 아니므로 조회 권한만 제공한다.

관리자 삭제 동작은 운영 데이터를 물리 삭제하지 않고 삭제 시각과 변경 이력을 남기는 소프트 삭제를
기본으로 한다. 인증 토큰처럼 즉시 폐기가 보안상 필요한 단기 자격 증명은 이 원칙의 예외로 별도
수명주기를 적용한다.

프롬프트 변경은 기존 결과와 비용에 영향을 줄 수 있으므로 버전, 변경자, 변경 사유, 적용 시점을 기록한다.

## 9. AI 기능 설계

### 9.1 역할별 AI 구성

하나의 거대한 요청으로 모든 결과를 생성하지 않고, 책임이 분명한 기능 단위로 분리한다.

```text
공고문 분석
기업 정보 정리
프로필 구조화

준비 전략 생성

답변 평가
```

아래 AI 기능은 모두 구현해야 할 제품 기능이다. 공통 AI 클라이언트와 프롬프트 엔진은 팀장 Owner의
공통 영역으로 관리하고, 각 기능 담당자가 자기 도메인 AI 서비스와 기능별 프롬프트 하위 폴더를 구현한다.

| 담당 | 구현 AI 기능 |

| B | 공고문 분석, 필수/우대 조건 추출, 담당 업무 요약, 기업 현황 요약, 면접 포인트 추출 |
| C | 공고-스펙 적합도 분석, 부족 역량 추천, 학습/자격증 추천, 장기 취업 경향 분석, 다음 지원 방향/커리어 전략 추천, 대시보드 AI 분석 결과 출력 |

### 9.2 공통 출력 원칙

AI 결과에는 가능한 경우 다음 정보를 포함한다.

- 결과 요약
- 판단 근거
- 신뢰 수준 또는 확인 필요 여부
- 사용자가 수정할 수 있는 구조화 데이터
- 다음 행동 제안
- 생성 시점과 사용한 데이터 범위

### 9.3 실패와 재시도

AI 요청은 지연, 형식 오류, 입력 부족, 비용 제한으로 실패할 수 있다. 사용자는 실패 이유와 재시도 방법을 확인할 수 있어야 하며, 동일 요청의 중복 과금을 방지해야 한다.

### 9.4 사용량 기록

AI 요청마다 사용자, 기능 유형, 지원 건, 프롬프트 버전, 토큰 또는 비용, 크레딧 차감량, 성공 여부를 기록한다. 이 기록은 사용자 사용량 조회, 비용 관리, 장애 분석, 품질 개선에 활용한다.

## 10. 신뢰·안전·개인정보 원칙

### 10.1 개인정보 최소 수집

분석에 필요하지 않은 민감정보는 수집하지 않는다. 사용자가 업로드한 이력서와 면접 기록은 목적, 보관 기간, 삭제 방법을 명확히 안내한다.
현재 정책 초안은 기본 영구 보관이다. 사용자가 직접 삭제하거나 탈퇴·삭제 요청을 하면 정책에 따라 삭제 또는 비식별화한다.
사용자가 원본 파일, 음성, 영상, 답변 원문을 삭제하면 해당 원본을 필요로 하는 추가 분석이나 재분석은 제공하지 않는다.

### 10.2 사용자 통제

사용자는 다음을 제어할 수 있어야 한다.

- AI 분석에 사용할 프로필과 문서 범위
- 음성·영상 원본 저장과 삭제
- 장기 경향 분석 반영 여부
- 데이터 보관과 삭제
- 기업명과 면접 후기 공개 여부

### 10.3 편향과 과도한 판단 방지

- 적합도 점수는 합격 확률로 표현하지 않는다.
- 성별, 나이, 외모, 장애, 억양 등 보호되거나 민감한 특성을 평가에 사용하지 않는다.
- 표정·시선·자세 피드백은 선택 기능으로 제공하며 태도 개선 목적에 한정한다.
- AI의 추론과 확인된 사실을 시각적으로 구분한다.

### 10.4 저작권과 외부 정보

공고 원문은 사용자 준비를 위한 범위에서 저장·분석하며 재배포하지 않는다. 기업 분석은 공개 출처와 확인 시점을 표시하고, 외부 서비스의 이용약관과 접근 제한을 준수한다.
기업 분석에는 웹 검색, 외부 API, 사용자가 직접 입력한 수동 출처를 사용할 수 있다. 출처 URL, 확인 시점,
재조회 시점, 추론과 확인된 사실의 구분을 기록한다. 사용자가 요청하면 사용자 입력 자료만 사용한 제한 분석도
가능하지만, 기본 권장 방식은 공개 출처를 포함한 실제 기업 분석이다.

- 지원 건 상세에서 준비 기록이 연결되어 보인다.
- 1차 범위에 포함된 사용자 기능은 관련 관리자 목록·상세·상태 확인 API와 화면을 갖춘다.

## 15. 기획 의사결정 기준

새 기능을 추가하거나 우선순위를 변경할 때 다음 질문을 순서대로 검토한다.

1. 이 기능이 지원 건 중심의 준비 흐름을 강화하는가?
2. 사용자가 다음 행동을 더 잘 결정하도록 돕는가?
3. 결과의 근거와 제한사항을 설명할 수 있는가?
4. 개인정보와 공정성 위험을 통제할 수 있는가?
5. MVP 핵심 흐름을 지연시키지 않는가?
6. 운영 비용과 유지보수 책임이 명확한가?
7. 성공 여부를 측정할 지표가 있는가?

이 기준을 충족하지 못하는 기능은 별도 실험으로 분리하거나 후속 단계로 미룬다.

## 16. 현재 확정 사항과 정책 초안

### 확정 사항

- 제품명은 CareerTuner를 사용한다.
- 제품의 핵심 단위는 지원 건이다.
- 1차 MVP는 텍스트 기반 분석과 모의면접을 중심으로 한다.
- 웹은 반응형 SPA로 개발하고 PWA, Capacitor 순으로 확장한다.
- 표정·시선·자세 분석은 합격 판단이 아닌 선택형 태도 개선 피드백으로 제한한다.
- AI 결과에는 근거와 사용자 수정 경로를 제공한다.
- 전역 내비게이션은 핵심 메뉴 + 큰 기능 드롭다운의 하이브리드 구조를 사용한다.
- URL은 리소스와 주요 작업을 path로, 목록 필터·정렬을 query로 표현한다.
- 사용자 기능 완료 시 관련 관리자 화면과 API도 함께 완료한다.
- 지원 건 보관과 소프트 삭제는 `status`가 아니라 `archived_at`, `deleted_at`으로 분리한다.
- 프로필 버전과 분석 실행 단위는 별도 엔터티로, 분석 항목별 결과와 근거는 구조화 JSON으로 관리한다.
- 하나의 지원 건은 특정 기업·직무·공고 조합이며, 공고문 변경은 같은 공고의 revision으로 저장한다.
- 공통 영역 Owner는 팀장이다.

### 현재 정책 초안

아래 항목은 현재 제품 방향으로 채택한 정책 초안이다. 개발 중 회의, 실제 AI 모델 선택, 운영 비용,
개인정보 검토, 사용자 테스트 결과에 따라 얼마든지 변경될 수 있다.

- 적합도는 AI가 공고 요구 조건과 사용자 스펙을 비교해 산정하고, 점수와 등급/구간을 함께 표기한다.
- 무료 사용자는 제한된 무료 크레딧을 제공받고, 공고 분석 보관도 1건으로 제한한다.
- AI 기능별 크레딧 차감량은 다르게 책정하며, AI 영상 면접은 고비용 기능으로 높은 차감량을 적용한다.
- 실패한 AI 요청은 기본적으로 크레딧을 차감하지 않는다.
- 크레딧은 결제 요청 직후가 아니라 구매 확정 또는 결제 성공 검증 후 지급한다.
- 구매 확정 후에는 기본적으로 환불 불가이나, 법적 의무·중복 결제·시스템 오류는 예외 정책으로 처리한다.
- 기업 분석은 기본적으로 웹 검색, 외부 API, 수동 출처를 허용하고, 출처·확인 시점·재조회 정책을 둔다.
- 사용자의 명시적 요청이 있으면 사용자 입력 자료만 쓰는 제한 분석도 가능하지만 기본 권장 방식은 아니다.
- 사용자 업로드 파일, 분석 결과, 원본 기록의 기본 보관 기간은 영구 보관으로 둔다.
- 음성·영상 원본은 저장하되 사용자가 삭제할 수 있다. 삭제 후에는 원본 기반 추가 분석을 제공하지 않는다.
- PWA는 정적 리소스 중심으로 캐시하고, 지원 건·프로필·AI 분석·첨삭·면접 답변·결제 정보는 장기 캐시하지 않는다.
- 로그아웃 시 사용자 관련 캐시를 삭제하고, 오프라인 상태에서는 민감 기능 접근을 제한한다.
- 장기 분석은 처음부터 시작하되, 데이터가 부족한 항목은 분석 불가 사유와 필요한 데이터를 안내한다.
