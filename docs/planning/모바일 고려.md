# CareerTuner 모바일 확장 전략

## 1. 문서 목적

이 문서는 CareerTuner를 데스크톱 웹에서 시작해 모바일 웹, PWA, Android·iOS 앱으로 확장하는 전략과 설계 기준을 정의한다.

CareerTuner의 초기 핵심 가치는 공고 분석, 스펙 비교, 지원 전략, 모의면접, 답변 첨삭에 있다. 따라서 초기부터 별도 네이티브 앱을 동시에 개발하기보다, 반응형 웹에서 핵심 흐름을 검증한 뒤 모바일 기능을 단계적으로 확장한다.

---

## 2. 전략 요약

CareerTuner의 모바일 전략은 다음 세 단계로 구성한다.

```text
1단계: React 반응형 웹
2단계: PWA
3단계: Capacitor 기반 Android·iOS 패키징
```

현재 모노레포 구조는 모바일 앱을 제외한 구조가 아니라, 하나의 React 코드베이스를 모바일 앱까지 확장하기 위한 웹 우선 구조다.

```text
CareerTuner/
 ├─ backend/      Spring Boot REST API
 ├─ frontend/     데스크톱 웹 + 모바일 웹 + PWA + 앱 WebView 원본
 └─ docs/
```

Capacitor를 도입하는 단계에서는 네이티브 프로젝트를 `frontend/android`, `frontend/ios` 아래에 관리한다.

---

## 3. 기술 선택

### 3.1 선택안 비교

| 기준 | React 반응형 웹 + PWA + Capacitor | React Native 별도 앱 |
| --- | --- | --- |
| 웹 화면 재사용 | 높음 | 낮음 |
| 코드베이스 수 | 하나 중심 | 웹과 앱 분리 |
| 초기 개발 부담 | 낮음 | 높음 |
| 앱스토어 배포 | Capacitor 단계에서 가능 | 가능 |
| 카메라·마이크·푸시 | 웹 제한이 있으나 Capacitor 플러그인으로 확장 가능 | 네이티브 접근에 강함 |
| 네이티브 UI 정밀도 | 중간 | 높음 |
| CareerTuner MVP 적합성 | 높음 | 낮음 |

### 3.2 현재 선택

CareerTuner는 **React 반응형 웹 → PWA → Capacitor** 방식을 기본 전략으로 채택한다.

선택 이유는 다음과 같다.

1. 핵심 기능 대부분이 문서 입력, 분석 결과 확인, 텍스트·음성 상호작용으로 구성되어 웹에서 구현 가능하다.
2. 데스크톱과 모바일에서 동일한 지원 건 데이터와 API를 사용한다.
3. 하나의 프론트엔드 코드베이스로 초기 개발과 유지보수 부담을 줄일 수 있다.
4. 사용자 가치가 검증되기 전에 별도 네이티브 앱을 개발하는 위험을 줄인다.

### 3.3 React Native 재검토 조건

다음 조건이 반복적으로 발생하면 별도 네이티브 앱 또는 React Native 도입을 재검토한다.

- WebView 성능으로 면접 녹화·실시간 처리가 안정적으로 동작하지 않음
- 백그라운드 오디오, 장시간 녹화, 복잡한 푸시 흐름이 핵심 기능이 됨
- 앱 전용 UI와 제스처가 제품 경쟁력의 핵심이 됨
- 네이티브 플랫폼별 기능 차이가 커져 공통 코드 유지가 더 어려워짐
- 모바일 사용량과 매출 비중이 별도 앱 투자를 정당화함

## 4. 모바일 제품 목표

모바일 환경은 데스크톱의 모든 정보를 축소해 제공하는 화면이 아니다. 이동 중에도 수행 가치가 높은 작업을 빠르게 완료하도록 설계한다.

### 4.1 모바일 핵심 과업

1. 진행 중인 지원 건과 오늘의 할 일을 확인한다.
2. 공고문을 붙여넣거나 파일로 등록한다.
3. 분석 결과의 핵심 강점과 부족 역량을 확인한다.
4. 예상 질문을 확인하고 짧은 답변 연습을 진행한다.
5. 음성 기반 모의면접을 수행한다.
6. 면접 리포트에서 우선 개선점을 확인한다.
7. 마감일과 준비 과제 알림을 확인한다.

### 4.2 데스크톱 우선 과업

다음 작업은 모바일에서도 접근할 수 있지만, 긴 입력과 비교가 필요한 특성상 데스크톱 경험을 우선 최적화한다.

- 긴 이력서·자기소개서 작성과 전체 편집
- 공고 원문과 프로필의 상세 비교
- 다수 지원 건을 한 화면에서 비교
- 복잡한 장기 분석 차트 확인
- 관리자 대량 데이터 관리

## 5. 아키텍처 원칙

### 5.1 API 중심 경계

모든 핵심 기능은 특정 화면이나 플랫폼에 종속되지 않는 백엔드 API로 제공한다.

```text
Spring Boot API
 ├─ 인증과 사용자 데이터
 ├─ 지원 건과 공고문
 ├─ 공고·기업·적합도 분석
 ├─ 면접 세션과 답변
 ├─ 첨삭과 리포트
 ├─ 알림
 └─ 결제·크레딧

React Frontend
 ├─ 데스크톱 웹
 ├─ 모바일 웹
 ├─ PWA
 └─ Capacitor 앱 화면
```

대표 API는 다음과 같다.

```text
POST /api/application-cases
POST /api/application-cases/{id}/job-posting
POST /api/application-cases/{id}/analysis
POST /api/application-cases/{id}/interview-sessions
POST /api/interview-sessions/{id}/answers
GET  /api/interview-sessions/{id}/report
```

플랫폼별 차이는 파일 선택, 카메라, 마이크, 알림처럼 장치 접근이 필요한 경계에서만 처리한다.

### 5.2 기능 감지와 점진적 향상

카메라, 마이크, 파일 공유, 푸시 알림은 브라우저와 OS마다 지원 수준이 다르다. 사용자 에이전트 문자열에 의존하기보다 실제 기능 지원 여부를 감지하고 대체 흐름을 제공한다.

```text
마이크 사용 가능
→ 음성 면접 제공

마이크 사용 불가 또는 권한 거부
→ 텍스트 면접으로 전환
```

### 5.3 공통 코드와 플랫폼 코드 분리

```text
frontend/src/
 ├─ features/              공통 제품 기능
 ├─ app/                   공통 라우트와 레이아웃
 ├─ platform/
 │   ├─ files/             웹·Capacitor 파일 선택 어댑터
 │   ├─ media/             마이크·카메라 어댑터
 │   ├─ notifications/     웹·네이티브 알림 어댑터
 │   └─ storage/           플랫폼 저장소 어댑터
 └─ admin/
```

플랫폼 분기 코드를 화면 컴포넌트에 흩어놓지 않고 어댑터 계층에 모은다.

### 5.4 네이티브 OAuth 반환 경계

네이티브 OAuth의 성공·실패 반환은 OS 소유권 검증이 가능한
`https://careertuner.kro.kr/auth/callback` Android App Link/iOS Universal Link로만 처리한다.
일반 앱 화면 이동에 쓰는 `careertuner://` 커스텀 스킴은 OAuth callback을 허용하지 않는다.

- Android manifest는 `android:autoVerify="true"`, host `careertuner.kro.kr`,
  exact path `/auth/callback`으로 범위를 제한한다.
- 로그인된 사용자의 소셜 계정 연결 결과도 exact path `/profile/detail` App Link로 제한한다.
  성공 제공자·mock 여부 또는 제한된 오류 코드만 허용하며, 임의 query와 커스텀 스킴 반환은 거부한다.
- 운영 웹 배포는 release 인증서 SHA-256 지문으로 `/.well-known/assetlinks.json`을 생성한다.
- 지문이 없거나 형식이 틀리면 운영 배포가 실패해야 하며 debug 지문을 운영 도메인에 게시하지 않는다.
- iOS 재생성 프로젝트에는 `applinks:careertuner.kro.kr` entitlement와 Xcode target 연결을 자동 적용한다.
  웹 배포는 Apple Team ID variable로 exact path AASA를 만들며, 미등록 상태에서는 빈 details로 모두 거부한다.
- iOS 완료 판정은 실제 서명 앱의 entitlement와 공개 AASA가 같은 application identifier를 사용하고
  실기기에서 Universal Link가 열리는지 확인한 뒤에만 내린다.
- 웹 브라우저 OAuth의 최종 화면은 `/auth/browser-callback`으로 분리해 설치된 앱이 웹 로그인을 가로채지 않게 한다.
- App Link의 `error` query는 제3자도 열 수 있으므로 표시만 하고 pending PKCE verifier를 삭제하지 않는다.
- debug APK의 명시적 intent 라우팅 성공과 release App Link의 OS verified 상태를 별도 검증한다.

## 6. 반응형 화면 설계

### 6.1 공통 원칙

- 모바일에서는 한 번에 하나의 주요 과업에 집중한다.
- 3열·2열 레이아웃은 의미 있는 순서의 단일 열로 재배치한다.
- 주요 액션은 손가락이 닿기 쉬운 하단 영역에 배치할 수 있다.
- 표와 차트는 카드 요약을 우선 제공하고 상세 보기에서 확장한다.
- 가로 스크롤은 데이터 표처럼 불가피한 경우에만 사용한다.
- 터치 영역과 요소 간 간격을 충분히 확보한다.

### 6.2 전역 내비게이션

#### 데스크톱

```text
상단 메뉴 + 사용자 메뉴 + 알림
```

#### 모바일

```text
상단: 화면 제목 + 알림 + 사용자 메뉴
하단 또는 Drawer: 대시보드 / 지원 건 / 면접 / 분석
```

하단 내비게이션을 도입할 경우 핵심 메뉴를 4~5개로 제한한다. 프로필, 결제, 설정, 고객센터는 사용자 메뉴 또는 Drawer에 둔다.

### 6.3 지원 건 목록

데스크톱에서는 목록과 필터를 넓게 제공할 수 있지만, 모바일에서는 카드형 목록을 기본으로 한다.

```text
기업명 · 직무명
상태 · 마감일
현재 준비도
다음 할 일
[이어서 준비]
```

필터는 화면 상단의 간단한 상태 필터와 전체 필터 Bottom Sheet로 구분한다.

### 6.4 지원 건 상세

데스크톱의 3열 구조를 모바일에서는 다음과 같이 변환한다.

```text
상단
 ├─ 기업명과 직무명
 ├─ 상태와 준비도
 └─ 지원 건 전환 버튼

가로 탭 또는 섹션 메뉴
 ├─ 개요
 ├─ 공고
 ├─ 비교
 ├─ 전략
 ├─ 질문
 ├─ 면접
 └─ 리포트

본문
 └─ 선택한 섹션

하단 고정 주요 액션
 └─ 예: 면접 시작, 분석 시작, 다음 과제 수행
```

- 지원 건 목록은 Drawer에서 연다.
- 데스크톱 우측 요약 패널은 상단 요약 카드 또는 별도 개요 탭으로 이동한다.
- 탭 수가 많으면 중요 탭을 우선 노출하고 전체 메뉴에서 나머지를 선택하게 한다.

### 6.5 공고 등록

모바일에서는 다른 앱에서 복사한 공고를 바로 붙여넣는 흐름이 중요하다.

```text
새 지원 건
→ 기업·직무 입력
→ 공고문 붙여넣기 또는 파일 선택
→ 추출 내용 간단 확인
→ 분석 시작
```

Capacitor 앱 단계에서는 공유 시트에서 CareerTuner로 텍스트나 URL을 전달하는 기능을 검토한다.

### 6.6 AI 면접

면접 진행 화면은 가장 높은 집중도와 장치 안정성이 필요한 화면이다.

#### 모바일 화면 구조

```text
상단: 진행률, 남은 시간, 종료
중앙: 현재 질문, 필요 시 아바타
하단: 녹음 시작·중지, 답변 완료
보조: 질문 다시 듣기, 일시 정지
```

#### 설계 원칙

- 진행 중에는 평가 점수와 복잡한 탐색 메뉴를 숨긴다.
- 화면 꺼짐과 앱 전환이 면접 세션에 미치는 영향을 안내한다.
- 네트워크 불안정 시 답변 임시 보관과 재전송을 지원한다.
- 권한이 없거나 장치 오류가 발생하면 텍스트 입력으로 전환한다.
- 영상은 선택 기능으로 제공하며, 사용 전 목적과 저장 여부를 안내한다.

### 6.7 분석 리포트

모바일에서는 요약 카드와 우선순위 중심으로 보여준다.

```text
현재 준비도
가장 강한 역량
우선 보완 역량 3개
이번 면접의 핵심 개선점
다시 연습할 질문
```

세부 근거와 전체 차트는 펼치기 또는 별도 상세 화면에서 제공한다.

## 7. PWA 단계

### 7.1 목표

PWA 단계의 목적은 앱스토어 배포 전에 설치형 경험, 안정적인 재방문, 제한적인 오프라인 사용 가능성을 검증하는 것이다.

### 7.2 적용 범위

| 항목 | 적용 방향 |
| --- | --- |
| Web App Manifest | 이름, 아이콘, 시작 URL, 테마 색상 |
| 설치 안내 | 사용자가 필요성을 느끼는 시점에 선택적으로 제공 |
| 서비스 워커 | 정적 자산 중심 캐시, 민감 데이터 장기 캐시 금지 |
| 오프라인 화면 | 네트워크 상태와 사용 가능한 기능 안내 |
| 업데이트 안내 | 새 버전 준비 시 사용자에게 새로고침 안내 |
| 웹 푸시 | 브라우저 지원과 사용자 동의를 확인한 뒤 제공 |

### 7.3 캐시 원칙

이력서, 면접 답변, 분석 결과처럼 개인정보가 포함된 데이터는 장치 공유와 유출 위험이 있다. 기본적으로 민감한 API 응답을 장기 캐시하지 않으며, 오프라인 저장이 필요한 경우 사용자 동의와 암호화·삭제 정책을 별도로 설계한다.
현재 정책 초안은 정적 리소스를 캐시해 모바일 로딩 속도를 개선하되, 사용자 프로필, 지원 건, AI 분석 결과,
면접 답변, 첨삭 결과, 결제 정보 등 개인·민감 데이터는 장기 캐시하지 않는 것이다.

캐시 정책:

| 데이터 | 정책 |
| --- | --- |
| 정적 파일, 앱 셸, 아이콘, 폰트, CSS/JS 번들 | 캐시 허용 |
| 공지사항, 요금제, 공개 서비스 정보 | 짧은 캐시 허용 |
| 지원 건, 프로필, AI 분석, 첨삭, 면접 답변 | 기본적으로 서버 우선 조회 |
| 결제 정보, 크레딧 장부, 인증 관련 데이터 | 장기 캐시 금지 |
| 로그아웃 시 | 사용자 관련 캐시 삭제 |
| 오프라인 상태 | 민감 데이터는 보여주지 않고 "네트워크 연결 필요" 안내 |

피해야 할 기본 캐시:

- 인증 토큰
- 음성·영상 원본
- 이력서와 자기소개서 원문
- 상세 개인정보
- 지원 건 상세와 AI 분석 결과
- 첨삭 결과와 결제 정보

### 7.4 PWA 완료 기준

- 홈 화면 설치 후 주요 화면이 정상 실행된다.
- 새 버전 배포 후 업데이트 흐름이 명확하다.
- 오프라인 또는 느린 네트워크에서 오류 대신 안내 화면을 제공한다.
- 인증과 민감 데이터가 캐시로 노출되지 않는다.
- 로그아웃 시 사용자 관련 캐시가 삭제된다.
- 오프라인 상태에서 민감 기능은 "네트워크 연결 필요" 안내를 제공한다.
- 모바일 브라우저의 마이크·파일 업로드 흐름이 검증되었다.

## 8. Capacitor 앱 단계

### 8.1 도입 목표

Capacitor는 검증된 React 웹앱을 Android·iOS 앱으로 패키징하고, 웹에서 제한되는 네이티브 기능을 보완하기 위해 도입한다.

### 8.2 예상 구조

```text
frontend/
 ├─ src/
 ├─ public/
 ├─ android/
 ├─ ios/
 ├─ capacitor.config.json
 └─ package.json
```

### 8.3 우선 검토할 네이티브 기능

| 기능 | 목적 |
| --- | --- |
| Push Notifications | 마감일, 준비 과제, 리포트 완료 알림 |
| Camera / Microphone | 음성·영상 면접 품질 개선 |
| Filesystem / File Picker | 공고문과 이력서 파일 선택 |
| Share Target | 다른 앱의 공고 텍스트·URL 전달 |
| Secure Storage | 플랫폼 정책에 맞는 민감 설정 저장 |
| App Lifecycle | 앱 전환·중단 시 면접 세션 보호 |

### 8.4 앱 도입 전 확인 조건

- 반응형 웹의 핵심 흐름이 모바일에서 충분히 검증되었다.
- PWA 또는 모바일 웹 사용 데이터에서 앱 수요가 확인되었다.
- 앱에서만 해결 가능한 기능 요구가 명확하다.
- 개인정보 처리와 앱스토어 정책 검토가 완료되었다.
- Android와 iOS 빌드·배포를 담당할 운영 책임이 정해졌다.

### 8.5 앱스토어 배포 고려사항

- 카메라, 마이크, 알림 권한 요청 사유를 명확히 작성한다.
- 권한 거부 상태에서도 핵심 기능의 대체 흐름을 제공한다.
- 계정 삭제와 개인정보 삭제 경로를 앱 안에서 제공한다.
- 외부 결제와 디지털 상품 정책을 플랫폼별로 검토한다.
- 개인정보처리방침과 데이터 수집 항목을 실제 동작과 일치시킨다.

## 9. 권한과 미디어 처리

### 9.1 권한 요청 원칙

앱 시작 시 모든 권한을 한 번에 요청하지 않는다. 사용자가 기능을 선택한 시점에 목적과 사용 범위를 설명한 뒤 요청한다.

```text
사용자가 음성 면접 시작 선택
→ 마이크 사용 목적과 저장 여부 안내
→ 권한 요청
→ 장치 테스트
→ 면접 시작
```

### 9.2 권한 거부 대응

| 권한 | 대체 흐름 |
| --- | --- |
| 마이크 | 텍스트 면접 |
| 카메라 | 음성 또는 텍스트 면접 |
| 알림 | 앱 내 알림함과 이메일 안내 |
| 파일 접근 | 텍스트 붙여넣기 또는 수동 입력 |

### 9.3 음성·영상 저장

- 저장 여부를 면접 시작 전에 알린다.
- 원본은 기본 저장한다.
- 사용자는 저장된 원본을 삭제할 수 있다.
- 사용자가 원본을 삭제하면 해당 원본을 필요로 하는 추가 분석과 재분석은 제공하지 않는다.
- 기본 보관 기간은 영구 보관으로 두되, 사용자 삭제·탈퇴·법적 요구에 따른 삭제 정책을 제공한다.
- 영상 기반 피드백은 명시적 동의가 있을 때만 수행한다.
- 원본 파일과 분석 결과의 접근 권한을 분리한다.

## 10. 네트워크와 세션 안정성

모바일 네트워크는 자주 변경되고 중단될 수 있다. 특히 면접 답변과 파일 업로드는 재시도 가능한 구조로 설계한다.

### 10.1 필수 대응

- 요청 중복 방지를 위한 식별자 사용
- 업로드 진행률 표시
- 일시적 실패 시 재시도
- 면접 답변의 로컬 임시 보관
- 연결 복구 후 재전송
- 장시간 분석 작업의 비동기 상태 조회

### 10.2 사용자 안내

```text
답변을 기기에 임시 저장했습니다.
연결이 복구되면 자동으로 업로드합니다.
```

네트워크 오류를 단순 실패로 처리하지 않고, 현재 저장 상태와 사용자가 할 수 있는 행동을 구체적으로 표시한다.

## 11. 성능 기준

### 11.1 기본 원칙

- 초기 번들 크기를 줄이기 위해 화면 단위 코드 분할을 적용한다.
- 큰 차트와 면접 관련 모듈은 필요할 때 로드한다.
- 긴 지원 건 목록은 페이지네이션 또는 가상화를 검토한다.
- 이미지와 영상은 적절한 크기로 압축한다.
- 분석 진행 중에는 전체 화면을 차단하지 않는다.

### 11.2 측정 항목

| 항목 | 확인 내용 |
| --- | --- |
| 초기 로드 | 모바일 네트워크에서 첫 화면 표시 시간 |
| 입력 반응 | 탭, 입력, Drawer 동작 지연 |
| 업로드 | 공고·이력서·음성 파일 업로드 성공률 |
| 면접 안정성 | 중단, 재연결, 답변 유실 발생률 |
| 배터리·발열 | 장시간 음성·영상 면접 시 영향 |

## 12. 보안과 개인정보

- 모바일 저장소에 인증정보와 개인정보를 불필요하게 저장하지 않는다.
- 토큰 저장 방식은 웹과 앱 환경의 위협 모델을 구분해 결정한다.
- 앱 전환 화면에서 민감정보가 노출되는 문제를 검토한다.
- 공용 장치에서 로그아웃과 데이터 삭제가 정상 동작해야 한다.
- 분실 장치에서 원격 세션 종료가 가능하도록 서버 세션 관리 방식을 유지한다.
- 로그와 오류 보고에 이력서, 답변 원문, 토큰이 포함되지 않게 한다.

## 14. 단계별 로드맵과 완료 기준

### 14.1 1단계: 반응형 웹

**범위**

- 핵심 사용자 화면 반응형 적용
- 지원 건 상세 1열 전환
- 모바일 파일 업로드와 텍스트 입력
- 텍스트 기반 면접
- 기본 모바일 접근성

**완료 기준**

- 모바일 브라우저에서 첫 지원 건 분석을 완료할 수 있다.
- 핵심 화면에 불필요한 가로 스크롤이 없다.
- 주요 액션이 터치 환경에서 명확하게 동작한다.

### 14.2 2단계: PWA

**범위**

- 설치 가능한 앱 셸
- 서비스 워커와 안전한 캐시
- 오프라인 안내
- 웹 푸시 검토
- 음성 면접 품질 개선

**완료 기준**

- 설치, 업데이트, 오프라인 안내가 안정적으로 동작한다.
- 민감정보가 의도치 않게 캐시되지 않는다.
- 모바일 재방문과 면접 사용성이 측정 가능하다.

### 14.3 3단계: Capacitor 앱

**범위**

- Android·iOS 패키징
- 네이티브 푸시
- 미디어·파일 플러그인
- 앱 수명주기 대응
- 스토어 배포 준비

**완료 기준**

- 핵심 사용자 흐름이 양 플랫폼에서 안정적으로 동작한다.
- 권한 거부 시 대체 흐름이 제공된다.
- 개인정보와 스토어 정책 검토가 완료된다.
- 앱 운영과 릴리즈 책임이 정의되어 있다.

## 15. 최종 원칙

CareerTuner의 모바일 전략은 앱 패키징 자체가 목적이 아니다. 사용자가 언제 어디서든 지원 준비를 이어갈 수 있도록 핵심 흐름을 안정적으로 제공하는 것이 목적이다.

따라서 개발 순서는 다음 원칙을 유지한다.

1. 반응형 웹에서 핵심 사용자 가치를 먼저 검증한다.
2. PWA로 설치와 재방문 경험을 검증한다.
3. 명확한 네이티브 요구와 사용자 수요가 확인되면 Capacitor 앱으로 확장한다.
4. 모든 단계에서 개인정보, 장치 권한, 면접 답변 유실 방지를 우선한다.
