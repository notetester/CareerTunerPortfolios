# 모바일 고려 — 반응형 웹 → PWA → Capacitor 전략

맞아. **그 구조만 놓고 보면 1차 결과물은 PC 웹/모바일 웹**이다.
그래서 “모바일앱”까지 대응하려면 처음부터 구조를 이렇게 잡아야 한다.

```text
1단계: React 반응형 웹
2단계: PWA 적용
3단계: Capacitor로 Android/iOS 앱 패키징
```

즉, 지금 말한 `backend/ + frontend/` 구조는 **모바일앱을 포기하는 구조가 아니라**, 모바일앱으로 확장하기 위한 **웹 우선 구조**다.

---

# 1. 가장 추천하는 모바일 대응 방식

네 프로젝트에는 **React Native를 바로 별도로 만드는 방식**보다, 아래 방식이 더 적합하다.

```text
gonggo-fitter/
 ├─ backend/      ← Spring Boot API 서버
 ├─ frontend/     ← React + Vite 웹앱
 ├─ mobile/       ← 나중에 Capacitor Android/iOS 프로젝트
 └─ docs/
```

처음에는 `frontend`만 만든다.
그 React 화면을 **PC 웹 + 모바일 웹 반응형**으로 만든다.

그다음 어느 정도 완성되면 `frontend`를 Capacitor로 감싸서 앱처럼 만든다.

Capacitor는 JavaScript, HTML, CSS로 만든 웹 앱을 Android, iOS, PWA로 배포할 수 있게 해주는 네이티브 런타임이다. 공식 설명에서도 웹 기술 기반으로 iOS, Android, PWA 앱을 만들 수 있다고 설명한다. ([Capacitor][1])

---

# 2. 왜 React Native를 바로 안 쓰는 게 좋냐

모바일앱을 제대로 하려면 선택지가 두 개다.

## 선택지 A: React 웹 + Capacitor

```text
React 웹 하나를 만든다
→ 모바일 반응형으로 대응한다
→ Capacitor로 Android/iOS 앱으로 감싼다
```

장점:

```text
웹 화면 재사용 가능
React 코드 대부분 유지
PC 웹, 모바일 웹, 앱을 한 코드베이스로 관리 가능
개발 부담이 작음
현재 디자인 초안 재사용 가능
```

단점:

```text
완전 네이티브 앱보다 세밀한 모바일 UI/성능은 약할 수 있음
```

---

## 선택지 B: React Native 별도 앱

```text
frontend/  ← PC 웹용 React
mobile/    ← 앱용 React Native
```

장점:

```text
진짜 네이티브 앱에 가까운 UX
카메라, 마이크, 푸시, 앱 권한 처리에 강함
```

단점:

```text
화면을 거의 따로 만들어야 함
React 웹 컴포넌트를 그대로 쓰기 어려움
프로젝트가 2배로 커짐
```

React Native 공식 문서도 React 지식을 활용해 Android/iOS 네이티브 앱을 만들 수 있다고 설명하지만, 환경 설정에는 Android Studio와 Xcode 같은 네이티브 개발 환경이 필요하다. ([React Native][2])

지금 프로젝트는 아직 MVP 시작 단계다. 그래서 React Native를 바로 만들면 **백엔드, 웹, 앱, AI, DB를 동시에 다 만드는 상태**가 된다. 이건 부담이 너무 크다.

# 3. 이 프로젝트에는 어떤 방식이 맞나?

나는 **React 웹 + PWA + Capacitor** 방식을 추천한다.

이유는 명확하다.

이 서비스의 핵심 기능은 모바일 네이티브 기능이 아니라 다음이다.

```text
공고 업로드
공고 분석
스펙 비교
예상 질문 생성
AI 면접
답변 첨삭
지원 건 관리
분석 리포트
```

이건 대부분 웹으로 충분히 구현 가능하다.

그리고 모바일에서 필요한 기능도 웹 기반으로 어느 정도 가능하다.

| 기능         | 모바일 웹/PWA | Capacitor 앱 | React Native |
| ---------- | --------: | ----------: | -----------: |
| 로그인        |        가능 |          가능 |           가능 |
| 공고문 붙여넣기   |        가능 |          가능 |           가능 |
| PDF 업로드    |        가능 |          가능 |           가능 |
| 마이크 녹음     |        가능 |          가능 |           가능 |
| 카메라 접근     |        가능 |          가능 |           가능 |
| AI 면접 화면   |        가능 |          가능 |           가능 |
| 푸시 알림      |     제한 있음 |          가능 |           가능 |
| 앱스토어 등록    |        불가 |          가능 |           가능 |
| 네이티브 수준 UX |        보통 |          중간 |           높음 |
| 개발 부담      |        낮음 |          중간 |           높음 |

PWA는 웹 기술로 만들지만 설치 가능하고, 단일 코드베이스로 여러 플랫폼과 장치에서 실행될 수 있다는 점이 장점이다. MDN도 PWA를 웹 플랫폼 기술로 만들되 플랫폼 전용 앱과 비슷한 사용자 경험을 제공할 수 있는 앱으로 설명한다. ([MDN][3])

# 4. 그러면 폴더 구조는 어떻게 해야 하냐

처음부터 이렇게 잡으면 된다.

```text
gonggo-fitter/
 ├─ backend/
 │   └─ Spring Boot API 서버
 │
 ├─ frontend/
 │   └─ React + Vite + TypeScript
 │      ├─ PC 웹
 │      ├─ 모바일 웹
 │      └─ PWA 기반
 │
 ├─ mobile/
 │   └─ Capacitor Android/iOS 프로젝트
 │      ※ 처음에는 비워둠
 │      ※ 2차 단계에서 생성
 │
 └─ docs/
```

처음부터 `mobile/` 폴더를 만들 필요는 없다.
하지만 문서상으로는 **나중에 `mobile`을 둘 자리**를 정해두면 된다.

실제로는 이런 순서다.

```text
초기:
gonggo-fitter/
 ├─ backend/
 ├─ frontend/
 └─ docs/

나중:
gonggo-fitter/
 ├─ backend/
 ├─ frontend/
 ├─ mobile/
 └─ docs/
```

# 5. 중요한 설계 원칙: “화면”은 반응형으로, “기능”은 API 중심으로

모바일앱 대응을 위해 가장 중요한 건 앱을 당장 만드는 게 아니다.

**백엔드와 프론트의 경계를 API로 정확히 나누는 것**이다.

```text
Spring Boot = 데이터와 AI 기능 제공
React Web = 화면
Capacitor App = React 화면을 앱 안에 담는 껍데기
```

즉, 모든 핵심 기능은 백엔드 API로 제공되어야 한다.

```text
POST /api/application-cases
POST /api/application-cases/{id}/job-posting
POST /api/application-cases/{id}/analysis
POST /api/application-cases/{id}/interview-sessions
POST /api/interview-sessions/{id}/answers
GET  /api/interview-sessions/{id}/report
```

그러면 나중에 모바일앱이 생겨도 같은 API를 호출하면 된다.

# 6. React 화면은 처음부터 모바일 대응 CSS로 짜야 한다

이게 핵심이다.

`frontend`는 PC 웹만 만드는 폴더가 아니다.

```text
frontend = PC 웹 + 모바일 웹 + 앱 WebView 화면의 원본
```

따라서 처음부터 반응형 기준으로 컴포넌트를 만들어야 한다.

예를 들어 지원 건 상세 화면은 PC에서는 이렇게 보인다.

```text
[지원 건 목록] [상세 분석 화면] [준비도 패널]
```

모바일에서는 이렇게 바뀌어야 한다.

```text
상단: 지원 건 제목
탭: 공고 분석 / 스펙 비교 / 면접 / 리포트
본문: 선택된 탭 내용
하단: 주요 액션 버튼
```

즉, PC의 3열 레이아웃을 모바일에서는 1열로 접어야 한다.

```text
PC:
sidebar + main + right panel

Mobile:
main only
sidebar는 drawer
right panel은 접이식 summary card
```

# 7. 모바일앱 기능은 언제 붙이나?

1차 MVP에서는 앱을 만들지 않아도 된다.

## 1차 MVP

```text
PC 웹
모바일 반응형 웹
Spring Boot API
MySQL
지원 건 관리
공고문 텍스트 업로드
AI mock 분석
텍스트 기반 면접
답변 첨삭
```

## 2차

```text
PWA 적용
모바일 홈 화면 설치
오프라인 캐시 일부 적용
모바일 UI 개선
마이크 녹음
카메라 권한 테스트
```

## 3차

```text
Capacitor 적용
Android 앱 빌드
iOS 앱 빌드
푸시 알림
앱스토어/플레이스토어 배포 검토
```

Capacitor는 PWA 지원도 제공하고, 웹 앱을 iOS/Android 네이티브 앱과 모바일 웹/PWA로 실행하는 흐름을 지원한다고 설명한다. ([Capacitor][4])

# 8. IntelliJ 기준으로 모바일까지 생각한 개발 흐름

IntelliJ Ultimate에서 이렇게 열면 된다.

```text
gonggo-fitter/
 ├─ backend/
 ├─ frontend/
 └─ docs/
```

실행 구성은 처음에는 두 개다.

```text
backend dev   → Spring Boot
frontend dev  → npm run dev
```

나중에 Capacitor를 붙이면 실행 구성이 늘어난다.

```text
backend dev
frontend dev
android sync
android run
```

Capacitor를 붙이는 시점에는 대략 이런 명령을 쓰게 된다.

```bash
cd frontend
npm install @capacitor/core @capacitor/cli
npx cap init
npm run build
npx cap add android
npx cap sync
npx cap open android
```

그러면 Android Studio가 열리고, 거기서 Android 앱 빌드가 가능해진다.

즉, 모바일앱은 처음부터 `React Native 프로젝트`를 따로 만드는 게 아니라, **완성된 React 앱을 Capacitor로 앱화하는 방식**으로 간다.

# 9. 이 서비스에서 모바일앱 대응 시 특히 조심할 화면

모바일에서 가장 어려운 화면은 이것들이다.

## 1) 지원 건 상세 화면

PC에서는 3열 구조가 좋지만 모바일에서는 복잡하다.

모바일 구조 추천:

```text
상단: 기업명 / 직무명 / 준비도 점수
가로 탭: 분석 / 스펙 / 전략 / 질문 / 면접 / 리포트
본문: 선택 탭
하단 고정 버튼: AI 면접 시작
```

---

## 2) AI 면접 화면

PC에서는 아바타, 질문, 답변, 평가 패널을 동시에 보여줄 수 있다.

모바일에서는 한 화면에 다 넣으면 답답하다.

모바일 구조 추천:

```text
상단: AI 면접관 아바타
중앙: 현재 질문
하단: 녹음 버튼 / 답변 종료 버튼
면접 후: 평가 리포트 화면으로 이동
```

---

## 3) 분석 리포트 화면

PC에서는 차트와 표를 많이 보여줄 수 있다.

모바일에서는 카드형이 낫다.

```text
적합도 72점
부족 역량 3개
추천 학습 4개
예상 질문 10개
```

이런 식으로 카드 단위로 접어야 한다.

# 10. 최종 권장 구조

따라서 지금은 이렇게 가면 된다.

```text
gonggo-fitter/
 ├─ backend/
 │   └─ Spring Boot REST API
 │
 ├─ frontend/
 │   └─ React + Vite + TypeScript
 │      ├─ Desktop Web
 │      ├─ Mobile Web
 │      └─ PWA-ready
 │
 └─ docs/
```

그리고 나중에:

```text
gonggo-fitter/
 ├─ backend/
 ├─ frontend/
 ├─ mobile/
 │   └─ Capacitor native shell
 └─ docs/
```

하지만 실제로 Capacitor는 `frontend` 안에서 관리해도 된다. 그러면 이렇게 될 수도 있다.

```text
frontend/
 ├─ src/
 ├─ public/
 ├─ android/
 ├─ ios/
 ├─ package.json
 └─ capacitor.config.ts
```

이 방식이 더 일반적이다.

# 한 줄 결론

**처음 구조가 PC 웹만 대응하는 것은 맞지만, 그게 문제는 아니다.**
처음 React를 **반응형 웹 + PWA-ready 구조**로 만들고, 나중에 **Capacitor로 Android/iOS 앱으로 감싸면** 된다.

그래서 지금 시작 구조는 여전히 이게 맞다.

```text
gonggo-fitter/
 ├─ backend/
 └─ frontend/
```

단, `frontend`를 “PC 웹 전용”으로 만들면 안 되고,

```text
frontend = PC 웹 + 모바일 웹 + 나중에 모바일앱으로 감쌀 원본
```

이라는 생각으로 만들어야 한다.
