# CareerTuner UX/UI 설계 기획서

## 1. 문서 목적

이 문서는 CareerTuner의 사용자 경험과 화면 설계 원칙을 정의한다. 초기 디자인 프로토타입의 장점을 유지하면서, 실제 제품 개발에 필요한 정보 구조, 화면별 역할, 반응형 규칙, 상태 표현, 접근성 기준을 구체화하는 것이 목적이다.

제품 기능의 범위와 우선순위는 [`기획.md`](기획.md)를 기준으로 하며, 이 문서는 해당 기능을 사용자가 이해하고 수행할 수 있는 화면 경험으로 변환한다.

---

## 2. UX 목표

CareerTuner의 화면은 사용자가 다음 세 질문에 빠르게 답할 수 있도록 설계한다.

1. **현재 무엇을 준비하고 있는가?**
2. **무엇이 부족하고 왜 부족한가?**
3. **지금 다음으로 무엇을 해야 하는가?**

이를 위해 모든 핵심 화면은 정보 제공에 그치지 않고 다음 행동으로 연결되어야 한다.

```text
분석 결과
→ 근거 확인
→ 사용자 수정 또는 확정
→ 준비 과제 선택
→ 면접·첨삭 실행
→ 변화 확인
```

### 2.1 핵심 UX 원칙

| 원칙 | 설명 |
| --- | --- |
| 지원 건 맥락 유지 | 사용자가 어느 기업·직무를 준비 중인지 모든 관련 화면에서 명확히 표시한다. |
| 설명 가능한 결과 | 점수와 추천에는 근거, 데이터 범위, 확인 필요 여부를 함께 보여준다. |
| 다음 행동 우선 | 화면마다 가장 중요한 다음 액션을 하나 이상 명확히 제시한다. |
| 점진적 정보 공개 | 처음에는 요약을 보여주고, 필요한 사용자가 근거와 세부 내용을 펼쳐본다. |
| 사용자 통제 | AI가 생성한 결과를 사용자가 수정·확정·재분석할 수 있게 한다. |
| 직군 중립성 | 특정 기술 직군에 편향되지 않는 용어와 예시를 사용한다. |
| 신뢰와 안전 | 합격 보장, 과도한 정밀도, 민감 특성 기반 판단처럼 오해될 표현을 피한다. |

## 3. 현재 디자인 방향 평가

초기 프로토타입은 홈, 대시보드, 지원 건 목록, 지원 건 상세, AI 면접, 분석, 커뮤니티, 요금제, 프로필, 로그인까지 제품의 주요 화면을 포함한다. 단순한 랜딩페이지가 아니라 서비스 전반의 흐름을 검증할 수 있는 구조이며, 특히 다음 두 요소는 제품 정체성을 효과적으로 보여준다.

### 3.1 유지해야 할 강점

**지원 건 상세의 3영역 구조**

```text
지원 건 탐색
→ 현재 지원 건의 분석·준비 작업
→ 준비도와 다음 행동 요약
```

이 구조는 여러 지원 건을 오가면서도 현재 작업 맥락을 유지하게 해준다. 데스크톱 핵심 화면의 기본 구조로 유지한다.

**대시보드의 행동 중심 요약**

대시보드는 활성 지원 건, 최근 분석, 준비도, 부족 역량, 오늘의 할 일을 한곳에서 보여준다. 사용자가 로그인 직후 다음 행동을 선택하는 데 적합하다.

### 3.2 개선이 필요한 영역

| 영역 | 현재 위험 | 개선 방향 |
| --- | --- | --- |
| 랜딩페이지 | 기능을 한 번에 너무 많이 설명하여 핵심 메시지가 흐려질 수 있음 | 첫 화면에서는 공고 등록 기반 지원 전략이라는 차별점을 우선 전달 |
| 상단 메뉴 | 기능 이름을 모두 노출하면 탐색 부담이 커짐 | 핵심 메뉴를 줄이고 프로필·설정은 사용자 메뉴로 이동 |
| 지원 건 생성 | 목록, 상세, 생성 경로가 섞일 수 있음 | `/applications/new`를 독립된 생성 흐름으로 정의 |
| AI 결과 | 점수만 강조하면 객관적 판정처럼 보일 수 있음 | 근거, 한계, 사용자 확인 상태를 함께 표시 |
| 예시 데이터 | 개발 직군 중심 예시는 범용 서비스 인식을 약화함 | 개발, 마케팅, 생산관리, 공공기관, 서비스 운영 등 다양한 직무 사용 |
| 모바일 | 데스크톱 3열 구조를 그대로 축소하면 사용성이 낮아짐 | 1열 구조, Drawer, 하단 주요 액션으로 재구성 |

## 4. 브랜드와 콘텐츠 방향

### 4.1 브랜드 포지셔닝

- 제품명: **CareerTuner**
- 브랜드 역할: 지원자의 경험과 준비 방향을 공고에 맞게 조정하는 조력자
- 핵심 가치: 명확성, 실행 가능성, 신뢰, 성장

### 4.2 핵심 메시지 체계

| 위치 | 권장 메시지 |
| --- | --- |
| 랜딩 히어로 | 공고를 등록하면, AI가 지원 전략을 설계합니다. |
| 서브 메시지 | 공고 분석, 스펙 비교, 예상 질문, 모의면접, 답변 개선을 하나의 지원 건에서 관리합니다. |
| 지원 건 생성 CTA | 새 지원 건 만들기 |
| 분석 시작 CTA | 공고 분석 시작 |
| 면접 시작 CTA | 이 지원 건으로 면접 연습 |
| 리포트 CTA | 개선 과제 확인 |

### 4.3 문체 원칙

제품 문구는 사용자를 평가하거나 명령하기보다, 관찰 가능한 사실과 선택 가능한 행동을 제시한다.

| 피해야 할 표현 | 권장 표현 |
| --- | --- |
| 이 공고에는 적합하지 않습니다. | 현재 프로필에서 확인되지 않은 요구사항이 3개 있습니다. |
| 합격 가능성이 낮습니다. | 지원 전 우선 보완하면 좋은 항목을 확인해 보세요. |
| 답변이 나쁩니다. | 질문에 대한 직접적인 답과 구체적인 결과가 부족합니다. |
| 표정이 어두워 직무 적합성이 낮습니다. | 답변 중 시선이 자주 아래로 향했습니다. 필요하면 시선 연습을 진행해 보세요. |

AI 결과에는 다음 표기를 구분해 사용한다.

- **확인됨**: 공고 원문 또는 사용자 입력에서 직접 확인된 내용
- **AI 제안**: 입력 내용을 바탕으로 생성한 추천
- **확인 필요**: 근거가 부족하거나 사용자의 검토가 필요한 내용

## 5. 전역 정보 구조와 내비게이션

### 5.1 공개 영역 내비게이션

```text
CareerTuner 로고
서비스 소개
요금제
고객센터
로그인
무료로 시작하기
```

공개 영역의 목적은 기능을 모두 설명하는 것이 아니라, 제품의 차별점과 신뢰 요소를 전달하고 첫 지원 건 생성으로 연결하는 것이다.

### 5.2 로그인 후 주요 내비게이션

로그인 후 전역 내비게이션은 핵심 메뉴를 간결하게 유지하되, 큰 기능의 드롭다운으로 하위 기능 전체에 접근할 수 있게 한다.

```text
대시보드
지원 건        목록, 새 지원 건, 보관함, 공고 분석, 적합도, 전략, 첨삭 기록
AI 면접        면접 시작, 세션 기록, 리포트
분석 리포트    적합도 분석, 장기 취업 분석, 대시보드 AI 분석 결과
커뮤니티      후기, 질문 공유, 신고/정책
요금제        플랜, 크레딧, AI 사용량, 결제 내역
사용자 메뉴    내 프로필, 알림, 결제/구독, 설정, 로그아웃
```

AI 첨삭은 특정 지원 건의 맥락에서 사용하는 경우가 많으므로 지원 건 상세에 우선 배치한다. 공통 첨삭 도구는 별도 페이지에서 제공할 수 있지만, 상단 핵심 메뉴에는 기본 노출하지 않는다.

### 5.3 URL 설계 원칙

아래 경로는 UX 관점의 목표 구조다. 현재 구현 경로는 `frontend/src/app/routes.ts`와
`frontend/README.md`에서 확인한다. 리소스와 주요 작업은 path로 표현하고, 검색·필터·정렬 같은 목록 상태만 query로 표현한다.

| 경로 | 역할 |
| --- | --- |
| `/` | 공개 홈 |
| `/dashboard` | 사용자 요약과 다음 행동 |
| `/applications` | 지원 건 목록 |
| `/applications/new` | 새 지원 건 생성 |
| `/applications/:id` | 지원 건 개요 |
| `/applications/:id/:section` | 공고 분석, 적합도, 전략, 질문 등 상세 영역 |
| `/interview` | 면접 기록과 빠른 면접 시작 |
| `/interview/:sessionId` | 진행 중 또는 완료된 면접 세션 |
| `/analysis` | 여러 지원 건을 종합한 장기 분석 |
| `/profile/:section` | 프로필 세부 관리 |
| `/pricing` | 요금제와 크레딧 |
| `/admin/**` | 관리자 영역 |

예시:

```text
/applications/123/fit-analysis
/applications/123/strategy
/profile/resume
/applications?status=READY&sort=deadline
```

새로고침과 링크 공유 후에도 같은 화면 상태가 유지되어야 한다.

## 6. 화면별 상세 설계

### 6.1 홈

#### 화면 목적

방문자가 3초 안에 CareerTuner가 무엇을 하는지 이해하고, 첫 지원 건 생성을 시작하도록 유도한다.

#### 권장 구성

```text
1. 히어로
   - 핵심 메시지
   - 공고 등록 CTA
   - 제품 화면 예시

2. 핵심 흐름 4단계
   - 공고 분석
   - 내 경험 비교
   - 예상 질문과 전략
   - 모의면접과 답변 개선

3. 지원 건 작업공간 예시
   - 한 공고 안에서 분석과 준비 기록이 연결되는 모습

4. 신뢰 요소
   - 근거 기반 분석
   - 사용자 수정 가능
   - 개인정보와 AI 사용 원칙

5. 요금제 요약

6. 최종 CTA
```

12단계 전체 프로세스, 모든 기능 목록, 세부 면접 모드는 별도 서비스 소개 페이지로 분리한다.

#### 주요 상태

- 비로그인: 무료로 시작하기, 데모 결과 보기
- 로그인 상태: 최근 지원 건 이어서 하기, 새 지원 건 만들기

### 6.2 로그인과 회원가입

#### 화면 목적

최소한의 입력으로 계정을 만들고 핵심 경험으로 진입하게 한다.

#### 설계 원칙

- 이메일과 소셜 로그인을 명확히 구분한다.
- 비밀번호 조건은 입력 전부터 표시한다.
- 오류는 필드 가까이에 구체적으로 표시한다.
- 로그인 성공 후 원래 접근하려던 경로로 복귀한다.
- 서비스 이용약관과 AI 데이터 이용 동의를 구분한다.

### 6.3 온보딩과 프로필

#### 화면 목적

첫 분석에 필요한 최소 프로필을 확보하고, 이후 사용자가 점진적으로 내용을 보완하게 한다.

#### 온보딩 단계

```text
희망 직무와 경력 구분
→ 대표 경험 또는 이력서 등록
→ 기술·역량 확인
→ 첫 지원 건 생성
```

모든 프로필 항목을 첫 진입에서 요구하지 않는다. 입력을 건너뛸 수 있게 하되, 분석 정확도에 미치는 영향을 설명한다.

#### 프로필 화면 구성

```text
프로필 완성도와 최근 갱신일
기본 정보
경력
프로젝트
기술·역량
학력·자격·어학
이력서·자기소개서·포트폴리오
AI 분석 사용 범위
```

### 6.4 대시보드

#### 화면 목적

현재 준비 상황을 요약하고 가장 중요한 다음 행동을 제시한다.

#### 우선순위 구조

1. 마감이 임박하거나 준비도가 낮은 지원 건
2. 오늘의 할 일
3. 최근 분석·면접 결과
4. 반복적으로 나타나는 부족 역량
5. 크레딧과 알림

#### 권장 카드

| 카드 | 표시 내용 | 주요 액션 |
| --- | --- | --- |
| 진행 중 지원 건 | 기업, 직무, 마감일, 준비도 | 이어서 준비 |
| 오늘의 할 일 | 우선순위, 예상 시간 | 완료 처리 |
| 최근 면접 | 점수 변화, 핵심 개선점 | 리포트 보기 |
| 반복 부족 역량 | 최근 지원 건에서 반복된 항목 | 학습 계획 보기 |
| AI 사용량 | 잔여 크레딧, 최근 사용 | 사용 내역 보기 |

카드 수가 많아질 경우 모든 정보를 한 화면에 펼치지 않고, 중요도와 최근성 기준으로 우선 노출한다.

### 6.5 지원 건 목록

#### 화면 목적

여러 지원 건을 빠르게 찾고, 상태를 파악하고, 다음 작업으로 이동하게 한다.

#### 필수 기능

- 상태 필터: 초안, 분석 중, 준비 중, 지원 완료, 종료, 보관
- 검색: 기업명, 직무명
- 정렬: 최근 활동, 마감일, 생성일, 준비도
- 즐겨찾기
- 새 지원 건 생성

기본 목록은 보관되지 않고 삭제되지 않은 지원 건만 보여준다. 보관함은 `archived_at`이 있는 지원 건을,
삭제 목록은 `deleted_at`이 있는 지원 건을 별도로 보여준다. 삭제는 소프트 삭제이며 일반 목록과 보관함에는 노출하지 않는다.

#### 목록 항목 정보

```text
기업명과 직무명
지원 건 상태
마감일
최근 활동
현재 준비도 요약
다음 할 일
```

준비도는 색상만으로 구분하지 않고 숫자 또는 텍스트 상태를 함께 제공한다.

### 6.6 새 지원 건 생성

#### 화면 목적

사용자가 부담 없이 공고를 등록하고 분석을 시작하도록 한다.

#### 권장 단계

```text
1. 기업·직무 기본 정보
2. 공고문 등록
3. 추출 내용 확인
4. 분석에 사용할 프로필 선택
5. 분석 시작
```

#### 공고문 입력 방식

- 텍스트 붙여넣기: P0
- PDF 업로드: P1
- 이미지 업로드와 OCR: P1
- URL 저장: P1
- 수동 입력: P1

분석 시작 전에 예상 크레딧 또는 제공량을 명확히 표시한다.

### 6.7 지원 건 상세

#### 화면 목적

특정 공고에 대한 분석, 준비, 면접, 첨삭을 한 공간에서 관리한다.

#### 데스크톱 레이아웃

```text
[지원 건 탐색] [현재 탭의 주요 작업] [준비도·다음 행동]
```

#### 중앙 영역 탭

| 탭 | 목적 |
| --- | --- |
| 개요 | 지원 건 상태와 핵심 요약 |
| 공고 분석 | 추출된 요구사항과 원문 근거 확인 |
| 기업 분석 | 공개 기업 정보와 면접 활용 포인트 |
| 내 스펙 비교 | 항목별 충족 상태와 근거 |
| 지원 전략 | 강조할 경험, 보완 과제, 준비 순서 |
| 예상 질문 | 질문 생성, 분류, 연습 상태 |
| 가상 면접 | 해당 지원 건 기반 면접 시작 |
| 면접 리포트 | 세션별 평가와 변화 |
| 첨삭 기록 | 답변과 문서의 개선 이력 |

#### 우측 요약 패널

- 현재 준비도와 산정 기준
- 가장 강한 역량
- 우선 보완할 역량
- 다음 할 일
- 최근 분석 시점
- AI 사용량

요약 패널은 사용자가 현재 탭의 세부 내용을 읽는 데 방해되지 않도록 고정 폭과 적절한 접기 기능을 제공한다.

### 6.8 분석 결과

#### 결과 표현 계층

```text
요약
→ 항목별 상태
→ 판단 근거
→ 사용자 확인·수정
→ 추천 행동
```

#### 상태 표현

| 상태 | 의미 | 표현 예시 |
| --- | --- | --- |
| 강점 | 요구사항을 뒷받침할 명확한 경험이 있음 | 강점 |
| 일부 충족 | 관련 경험은 있으나 깊이 또는 근거가 부족함 | 보완 필요 |
| 미확인 | 프로필에서 근거를 찾지 못함 | 확인 필요 |
| 해당 없음 | 사용자 또는 공고 상황에 적용되지 않음 | 해당 없음 |

적색과 녹색만으로 상태를 구분하지 않는다. 아이콘, 텍스트, 배지 형태를 함께 사용한다.

### 6.9 AI 면접

#### 면접 시작 화면

```text
지원 건 선택
면접 모드 선택
질문 수와 예상 시간
텍스트·음성·영상 사용 선택
사용할 프로필과 공고 범위 확인
시작 전 장치 테스트
```

#### 면접 진행 화면

진행 중에는 평가 정보나 많은 보조 기능을 노출하지 않는다. 사용자가 현재 질문과 답변에 집중할 수 있도록 구성한다.

```text
상단: 진행률과 남은 시간
중앙: 현재 질문과 필요한 보조 정보
하단: 답변 입력·녹음·종료 액션
보조: 질문 다시 듣기, 잠시 멈춤, 기술 문제 신고
```

#### 면접 종료 화면

- 전체 요약
- 잘된 점
- 우선 개선점
- 질문별 피드백
- 개선 답변 예시
- 다시 연습할 질문
- 다음 면접 추천

### 6.10 장기 분석

#### 화면 목적

여러 지원 건에서 반복되는 경향과 시간에 따른 변화를 보여준다.

#### 권장 구성

```text
분석 대상 기간과 데이터 수
지원 직무·산업 분포
반복 강점과 반복 부족 역량
면접 평가 변화
자주 개선이 필요한 답변 요소
다음 지원 방향과 학습 제안
```

데이터가 부족하면 빈 차트를 보여주지 않고, 분석을 생성하기 위해 필요한 활동을 안내한다.

### 6.11 커뮤니티

#### 화면 목적

사용자 경험과 질문을 공유하되, 개인정보와 기업 기밀 노출을 최소화한다.

#### 필수 UX

- 카테고리와 직무 필터
- 익명 여부 선택
- 기업명 공개 여부 선택
- 신고
- 운영 정책 안내
- 삭제·수정

게시글 작성 중 민감정보가 포함될 가능성이 있으면 확인 안내를 제공한다.

### 6.12 요금제와 크레딧

#### 화면 목적

사용자가 비용과 제공량을 이해하고 기능 사용 전에 예상 차감을 확인하도록 한다.

#### 표시 원칙

- 무제한 대신 명확한 월 제공량을 표시한다.
- 기능별 크레딧 차감 기준을 제공한다.
- 결제 전에 갱신일, 환불 기준, 자동 결제 여부를 표시한다.
- AI 요청 시작 전에 예상 차감량을 표시한다.
- 실패한 요청의 차감·환불 상태를 확인할 수 있게 한다.
- 무료 사용자의 제한된 무료 크레딧과 공고 분석 보관 1건 제한을 명확히 표시한다.
- AI 영상 면접처럼 많은 크레딧을 소비하는 기능은 시작 전 별도 확인을 제공한다.
- 구매 확정 후 기본 환불 불가 정책과 예외 조건을 결제 전 확인하게 한다.

정확한 크레딧 제공량과 차감량은 AI 모델과 운영 비용이 확정되기 전까지 바뀔 수 있으므로,
화면 문구는 정책 변경 가능성을 약관·도움말과 함께 안내한다.

### 6.13 관리자 화면

관리자 화면은 사용자 앱과 같은 디자인 토큰을 사용하되, 업무 효율을 위해 정보 밀도를 높일 수 있다.
사용자 기능이 완료될 때 관련 관리자 화면과 API도 함께 완료한다. 다만 고급 운영 분석, 대량 처리,
자동화 같은 관리자 고도화는 기본 운영 기능을 갖춘 뒤 확장한다.

```text
운영 지표 요약
사용자 검색과 상태 관리
AI 요청 실패와 비용 확인
프롬프트 버전 관리
결제·환불 처리
게시글·신고 처리
공지·FAQ·문의 관리
```

위험한 작업은 확인 단계를 제공하고, 변경 이력을 남긴다.

## 8. 디자인 시스템

### 8.1 토큰 원칙

색상, 타이포그래피, 간격, 반경, 그림자, 상태 색상을 토큰으로 관리한다. 화면별 임의 값을 최소화하여 사용자 화면과 관리자 화면의 일관성을 유지한다.

### 8.2 의미 기반 색상

| 의미 | 사용 예 |
| --- | --- |
| Primary | 주요 CTA, 활성 상태 |

| Warning | 보완 필요, 확인 필요 |
| Danger | 삭제, 오류, 크레딧 부족 |
| Neutral | 보조 정보, 비활성 상태 |

적합도나 준비도 점수는 성공·실패를 암시하는 강한 색상보다 중립적인 진행 상태로 표현한다.

```text
Tabs
Card
Badge
Progress
Tooltip
Dialog / Drawer
Toast
EmptyState
ErrorState
LoadingSkeleton
EvidencePanel
AiResultCard
CreditUsageNotice
```

`EvidencePanel`과 `AiResultCard`는 CareerTuner의 신뢰 경험을 만드는 핵심 공통 컴포넌트다.

## 9. 상태와 피드백 설계

모든 데이터 기반 화면은 다음 상태를 정의해야 한다.

| 상태 | 요구사항 |
| --- | --- |
| 최초 빈 상태 | 기능 가치와 첫 행동을 설명한다. |
| 로딩 | 진행 중인 작업과 예상 대기 형태를 알린다. |
| 부분 완료 | 완료된 결과와 아직 진행 중인 항목을 구분한다. |
| 성공 | 완료 결과와 다음 행동을 제시한다. |
| 오류 | 원인, 영향, 재시도 또는 대체 행동을 제공한다. |
| 권한 없음 | 필요한 권한과 해결 방법을 설명한다. |
| 크레딧 부족 | 필요한 양, 현재 잔액, 충전 경로를 표시한다. |

AI 분석은 시간이 걸릴 수 있으므로 화면을 차단하는 단일 로딩 스피너보다 단계별 진행 상태를 보여준다.

```text
공고문 확인
→ 요구사항 추출
→ 프로필 비교
→ 준비 전략 생성
→ 결과 저장
```

## 10. 접근성 기준

- 키보드만으로 주요 기능을 사용할 수 있어야 한다.
- 포커스 위치를 명확히 표시한다.
- 색상 외에 텍스트와 아이콘으로 상태를 구분한다.
- 입력 요소에는 시각적 라벨을 제공한다.
- 오류 메시지는 해당 입력과 연결한다.
- 모달과 Drawer가 열리면 포커스를 내부에 유지하고 닫은 후 원래 위치로 복귀한다.
- 차트에는 표 또는 텍스트 요약을 함께 제공한다.
- 음성 질문에는 텍스트를 제공하고, 영상 피드백은 선택적으로 사용하게 한다.
- 주요 터치 영역은 모바일에서 충분한 크기와 간격을 확보한다.

## 12. 화면 완료 기준

각 화면은 다음 조건을 충족할 때 구현 완료로 판단한다.

- 화면 목적과 주요 사용자 행동이 명확하다.
- 데스크톱과 모바일 레이아웃이 정의되어 있다.
- 로딩, 빈 상태, 오류, 권한, 크레딧 부족 상태가 처리되어 있다.
- 키보드와 스크린리더를 고려한 기본 접근성이 제공된다.
- AI 결과에는 근거, 확인 상태, 다음 행동이 포함된다.
- URL과 화면 상태가 일관되게 연결된다.
- 추적할 사용자 행동 이벤트가 정의되어 있다.

## 13. 우선 구현 순서

```text
1. 전역 디자인 토큰과 공통 레이아웃
2. 인증과 기본 내비게이션
3. 지원 건 목록과 생성 흐름
4. 지원 건 상세와 분석 결과 컴포넌트
5. 대시보드
6. 텍스트 기반 AI 면접과 리포트
7. 프로필 보완 흐름
8. 장기 분석
9. 커뮤니티와 결제
10. 관리자 화면 고도화
```

초기 구현에서는 장식적 완성도보다 지원 건 생성부터 분석, 면접, 개선까지 이어지는 흐름의 명확성을 우선한다.
각 사용자 기능의 필수 관리자 화면/API는 해당 기능과 동시에 완료하고, 대량 운영이나 자동화 같은 고도화는 후속으로 확장한다.
