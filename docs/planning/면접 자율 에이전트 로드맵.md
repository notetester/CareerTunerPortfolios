# 면접 자율 에이전트 로드맵 (D 도메인 · C/팀장 협업)

> 담당: D(가상 면접 / AI 면접관). 협업: C(홈·대시보드·취업 분석 / 팀장 · 공통 영역 Owner).
> 이 문서는 가상 면접을 "단발 LLM 호출"에서 "자율 에이전트가 스스로 계획·실행하고 그 과정을 사용자에게
> 투명하게 보여주는" 경험으로 끌어올리기 위한 공용 설계·분담 문서다. 개인 작업 메모(`todo.md`, git 미추적)와
> 별도로, 팀이 함께 읽고 합의하는 기준 문서로 관리한다.

---

## 1. 배경과 목표

### 1-1. 왜 바꾸나

현재 면접 답변 평가는 `Retriever → Evaluator → Critic` 3단계를 **고정 순서**로 실행하고, 그 결과를 답변 제출
이후에 **한 번에 조회**해 단순 리스트로 보여준다. 동작은 하지만 두 가지가 약하다.

- **자율성 없음**: 순서가 코드에 박혀 있어 분기·반복·재시도가 없다. "에이전트가 스스로 판단한다"는 느낌이 없다.
- **투명성 약함**: 단계가 사후에 한꺼번에 뜨고, `detail`(점수·판정 근거)은 저장만 되고 화면에 쓰지 않는다.
  "AI가 지금 무엇을 하는지" 실시간으로 지켜보는 체험이 없다.

목표는 사용자가 면접을 진행하는 동안 **에이전트의 작업 과정이 실시간으로 펼쳐지는** 화면을 만드는 것이다.
직관적으로 "자율 에이전트가 일하고 있다"가 읽혀야 한다.

### 1-2. 자체 LLM 방향 전환 (팀 회의 결정)

기존 합의는 학습 데이터를 모아 **OpenAI gpt-4o-mini 파인튜닝 API**를 호출하는 것이었다. 팀 회의에서 방향을
바꿨다. **오픈소스 모델을 유료 GPU 서버에서 직접 LoRA 파인튜닝**해, 자체 모델을 실제로 만들어 평가에 붙여본다.

- 목적은 최고 품질이 아니라 **"우리가 자체 모델을 직접 학습해 서비스에 연결해봤다"는 증거 확보**다.
- 품질이 OpenAI보다 낮아도 된다. 라이브 API는 폴백으로 유지한다.
- 데스크탑급 GPU 확보가 불가능하므로 **시간제 GPU 임대**를 전제로 한다(섹션 5에서 옵션 비교).

### 1-3. 목표 한 줄

> "AI 면접관이 스스로 계획하고 도구를 골라 쓰며, 그 판단 과정을 실시간으로 보여준다.
> 평가 모델은 우리가 직접 파인튜닝한 자체 LLM(폴백: OpenAI)을 쓴다."

## 3. 목표 아키텍처 — 자율 루프 (병행 모드)

자율 루프는 **두 모드를 병행**한다. 두 모드 모두 같은 trace 인프라(`interview_agent_step` + SSE)를 공유한다.

### 3-1. 기본 모드 — 동적 오케스트레이터 (운영 기본값)

오케스트레이터가 **상태를 보고 다음 액션을 규칙으로 결정**하는 루프. 액션 풀은 고정이라 데모가 안정적이다.

```text
상태(state): { question, answer, ragContext?, evalScore?, criticVerdict?, confidence, turn }
액션 풀(actions): RETRIEVE · EVALUATE · CRITIC · PROBE(꼬리질문) · REPORT · FINISH

루프:
  while not done:
    action = decideNext(state)        # 규칙 기반 정책
    result = run(action)              # 각 액션이 곧 하나의 에이전트 step
    logStep(action, result)           # interview_agent_step 적재 + SSE push
    state = update(state, result)

decideNext 규칙 예:
  - ragContext 없음 & RAG 활성  → RETRIEVE
  - 평가 전                      → EVALUATE
  - 평가 후 & 미검증             → CRITIC
  - Critic 점수 크게 조정(|Δ|≥N) → EVALUATE 재시도(1회)
  - 답변 모호(짧음/핵심 누락)    → PROBE(꼬리질문 생성)
  - 충분히 평가됨                → REPORT → FINISH
```

기존 고정 파이프라인은 이 루프의 한 경로(분기 없는 직선)로 포함된다. 즉 **기존 동작을 깨지 않고 확장**한다.

### 3-2. 시연 모드 — 풀 LLM 자율 루프 (토글)

`planner = llm`일 때, 매 턴 **LLM Planner가 다음 액션을 JSON으로 결정**한다(정석 자율 에이전트).

```text
Planner 프롬프트: "현재 상태 + 사용 가능한 도구 목록이 주어졌다. 다음에 어떤 도구를
                   어떤 이유로 호출할지 JSON으로 답하라." → { action, reason }
```

- 같은 trace 인프라를 그대로 쓴다(추가 UI 작업 없음). 각 Planner 결정도 `PLANNER` step으로 기록한다.
- 라이브 API 실패·비정상 응답 시 **3-1의 규칙 정책으로 자동 폴백**한다(데모 안정성).
- 시연·발표에서 "AI가 스스로 판단해 도구를 고른다"를 보여주는 용도.

### 3-3. 설정

```yaml
careertuner:
  interview:
    agent:
      planner: rule        # rule | llm  (기본 rule)
      max-turns: 6         # 무한 루프 방지
      reeval-threshold: 20 # Critic 점수 조정폭 임계치
```

## 4. 목표 아키텍처 — 마누스형 프론트 (실시간 trace UX)

직관적으로 "자율 에이전트가 일한다"가 읽히도록 프론트를 바꾼다. **이 영역은 C(팀장 · 공통 영역 Owner)와 협업**한다.

### 4-1. 핵심 변화

| 항목 | 현재 | 목표 |
| --- | --- | --- |
| 표시 시점 | 답변 후 배치 조회 | **SSE 스트리밍** — 단계가 하나씩 실시간 등장 |
| 단계 카드 | 배지 + summary 한 줄 | 아이콘 · 에이전트명 · action · **진행중→완료 상태 전이** · 소요시간 |
| detail | 미사용 | 펼치기(점수·판정·근거·RAG 출처 미리보기) |
| 진행감 | 없음 | "생각 중…" 인디케이터, 타임라인 패널 |

### 4-2. 컴포넌트 경계 (D ↔ C)

- **D 1차 구현**: `features/interview` 안에 `AgentTimeline` 패널을 만들어 SSE 구독 + 단계 렌더링.
- **C 공통화 검토**: 이 타임라인은 다른 AI 기능(C의 적합도 분석 진행 표시 등)에서도 재사용 가치가 있다.
  공통 컴포넌트(`app/components`)로 승격할지, 디자인 토큰·모션 톤을 어떻게 통일할지는 **C가 Owner**로 합의한다.
  (참고: C는 이미 `features/analysis/components/FitAnalysisProgress.tsx`로 단계 진행 표시를 갖고 있다 —
  톤을 맞추거나 공통 베이스를 함께 뽑는 것이 자연스럽다.)
- **대시보드 연결**: C의 대시보드 "최근 면접" 카드(`features/dashboard/components/RecentInterviewCard.tsx`)에
  "에이전트 활동 N단계" 같은 요약을 노출할 수 있다. 데이터 계약은 섹션 6 참고.

## 5. 목표 아키텍처 — 자체 LLM (오픈모델 + GPU 임대 LoRA)

### 5-1. 코드 추상화 (갈아끼움 포인트)

현재는 OpenAI에 100% 묶여 있다. **인터페이스를 끼워 자체 모델을 선택 가능**하게 만든다.

```text
AiEvaluatorClient (인터페이스)            ← interview/ai 에 신설
 ├─ OpenAiEvaluatorClient   (기존 로직 래핑, 폴백 기본값)
 └─ OssEvaluatorClient      (자체 파인튜닝 모델 호출: vLLM/TGI의 /v1/chat/completions)

선택: careertuner.interview.eval.provider = openai | oss   (기본 openai)
```

- `InterviewAgentOrchestrator`는 인터페이스만 호출하므로 **오케스트레이터 코드는 수정 없음**.
- 학습 데이터(JSONL)와 `interview_training_sample` 파이프라인은 **그대로 재사용**한다(Chat 포맷이 오픈모델 학습에도 호환).
- `FineTuneClient`(OpenAI FT API)는 자체 학습 잡 트리거/상태 기록 구조로 교체하거나 병행한다.

### 5-2. 학습 시나리오

```text
interview_training_sample → JSONL export → (GPU 서버) Qwen2.5 한국어 모델 LoRA 파인튜닝
  → 가중치 산출 → vLLM/TGI로 서빙 → OssEvaluatorClient 가 그 엔드포인트를 호출
```

- 후보 모델: Qwen2.5-7B-Instruct (한국어 양호, 7B는 단일 GPU LoRA 가능 / 데모 현실선).
  더 키우면 14B. 32B 이상은 비용·VRAM 부담이라 1차 범위 밖.
- 학습 프레임워크: Unsloth / LLaMA-Factory 등 LoRA 지원 오픈 도구.

### 5-3. GPU 서버 옵션 비교 (**결정 보류** — 팀과 확정)

| 옵션 | 형태 | 장점 | 단점 | 비고 |
| --- | --- | --- | --- | --- |
| **RunPod** (1순위 후보) | 시간제 GPU 임대(A100/H100 등) | 저렴($1~3/h대), 끄면 과금 중단, 셋업 단순 | 운영 직접, 스토리지 별도 | 데모용 "직접 돌린 증거" 확보에 적합 |
| Vast.ai | 시간제 GPU 마켓 | 더 저렴할 수 있음 | 가용성·신뢰성 편차 | RunPod 대안 |
| AWS SageMaker / Lambda | 매니지드 학습 | 운영 편함, 기존 배포가 AWS EC2라 인프라 통일 | 비용·세팅 진입장벽 | 인프라 통일 관점에서 검토 |
| 자체 서버 | 직접 보유 | 통제권 | **데스크탑 사양 확보 불가 → 제외** | 전제상 배제 |

> 결정 보류 항목: 실제 플랫폼·모델 크기·예산 상한은 팀과 확정한다. 플랜에는 RunPod 1순위로만 기재.

### 5-4. "이 정도까지 해봤다" 완료 기준 (최소선)

- [ ] 자체 모델 LoRA 파인튜닝 1회 실제 실행(작은 샘플이라도 무방)
- [ ] 산출 모델을 vLLM/TGI로 서빙해 `OssEvaluatorClient`로 평가 1회 성공
- [ ] OpenAI 평가와 자체 모델 평가를 **나란히 비교한 결과**(점수 차이·일관성) 1장 캡처/표
- [ ] 라이브 실패 시 OpenAI 폴백 확인

## 6. API 계약 (프론트 ↔ 백엔드)

### 6-1. 실시간 trace 스트림 (신규)

```text
GET /api/interview/sessions/{id}/agent-steps/stream      (SSE)
  event: step
  data: { id, questionId, stepNo, agent, action, status, summary, detail, elapsedMs, createdAt }
        status: running | done | failed
```

- 기존 `GET /sessions/{id}/agent-steps`(배치)는 **재진입·새로고침 시 복원용**으로 유지한다.
- 프론트는 답변 제출 직후 스트림을 구독하고, 단계가 끝나면 닫는다.

### 6-2. 대시보드 연결용 요약 (C 활용)

```text
GET /api/interview/sessions/{id}/agent-steps  의 응답을 C가 집계해
"에이전트 N단계 실행 / 마지막 액션 / 최종 점수" 형태로 대시보드 카드에 노출 가능.
별도 집계 API가 필요하면 D가 제공한다(계약은 C 요청 시 확정).
```

## 8. 단계별 로드맵

```text
Phase 1 — 자율 루프 백엔드 (D)
  · 동적 오케스트레이터로 리팩터(기존 경로 보존), 액션 풀·정책 구현
  · interview_agent_step 에 status/elapsed 보강
  · 단위 테스트(분기·재시도·폴백)

Phase 2 — 실시간 trace UX (D → C 협업)
  · SSE 스트림 API
  · AgentTimeline 패널(실시간 등장·상태 전이·detail 펼치기)
  · C와 디자인 톤 합의 → 공통화 여부 결정

Phase 3 — 풀 LLM 자율 토글 (D)
  · LLM Planner + 규칙 폴백, 시연 시나리오 정리

Phase 4 — 자체 LLM (D, 결정 보류 해제 후)
  · AiEvaluatorClient 추상화 → OpenAi/Oss 분기
  · GPU 임대로 Qwen2.5 LoRA 1회 학습 → vLLM 서빙 → 평가 연결
  · OpenAI vs 자체 모델 비교 결과 정리(완료 기준 5-4)

Phase 5 — 대시보드·발표 정리 (C 협업)
```

## 9. 비용·리스크

- **토큰 비용**: 자율 루프는 단계가 늘면 호출이 늘어 세션당 비용이 오른다. `max-turns` 상한 + `ai_usage_log`로 제한.
- **데모 안정성**: 풀 LLM 자율·Realtime·자체 모델은 라이브 실패 가능. 전 구간 **폴백(규칙 정책 / OpenAI / 목 모드)** 유지.
- **GPU 비용**: 시간제 임대는 끄면 과금 중단되나, 켜둔 채 방치 주의. 학습 후 즉시 인스턴스 종료.
- **자체 모델 품질**: OpenAI보다 낮을 수 있음 — 목표는 품질이 아니라 "직접 만든 증거"이므로 폴백 전제로 수용.
