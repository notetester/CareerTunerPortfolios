# CareerTuner · C 영역 통합 기획서

> **한 줄 정의** — C 영역은 "이 공고에 지원해도 되는가, 무엇을 보완해야 하는가, 다음에 어디로 가야 하는가"를 한 흐름으로 답해 주는 **취업 전략 분석·대시보드** 영역이다.
>
> **핵심 가치** — 채용공고의 요구 조건과 사용자 프로필을 비교해 **적합도 점수와 근거**를 제시하고, **부족 역량·학습·자격증**으로 보완 경로를 제안하며, 여러 지원 건을 누적해 **장기 취업 경향과 다음 지원 방향**까지 한 화면에서 연결한다. 점수 하나로 끝내지 않고 "왜 그 점수인지"와 "그래서 무엇을 할지"를 항상 함께 보여주는 것이 C 영역의 설계 원칙이다.

---

## 문서 목적과 범위

이 문서는 CareerTuner의 C 영역(홈 / 스펙 비교 / 취업 분석 / 대시보드 + AI 커리어 전략 추천)이 어떤 문제를, 어떤 화면과 데이터, 어떤 AI 구조로 풀어내는지 통합해 정리한 **기획·구현 설명서**다. 화면별 상세 캡처·주석·분기 상태는 스토리보드 산출물(`CareerTuner_C_Storyboard.pptx`, `storyboard.html`, `storyboard.pdf`)을 참조하고, 이 문서는 그 위에서 "왜 이렇게 기획·구현했는가"를 설명한다.

C 영역의 사용자 기능은 홈, 내 스펙과 공고 비교, 직무 적합도 점수, 부족 역량 분석, 추천 학습, 추천 자격증, 지원 전략, 취업 분석 대시보드, 장기 지원 경향 분석, 전체 준비 현황 대시보드로 구성된다. 여기에 대응하는 7개의 AI 기능(#12~#18)과 관리자 운영 화면이 같은 릴리스 완료 기준으로 함께 묶인다.

---

## 문제 정의

취업을 준비하는 사용자는 채용공고와 자신의 스펙 사이에서 다음 세 가지를 스스로 판단하기 어렵다.

1. **공고-스펙 격차를 모른다.** 공고에는 필수 조건과 우대 조건이 섞여 있고, 자신의 프로필이 그중 무엇을 충족하고 무엇을 빠뜨렸는지 한눈에 비교하기 어렵다. "지원해도 되는가"라는 가장 기본적인 의사결정조차 감에 의존하게 된다.
2. **무엇을 보완할지 모른다.** 부족하다는 느낌은 있어도, 지원 전 반드시 채워야 할 항목과 장기적으로 쌓아야 할 항목이 뒤섞여 우선순위가 서지 않는다. 학습 방향과 자격증 선택도 막연하다.
3. **다음 방향을 모른다.** 지원 건 하나하나는 보지만, 여러 번의 지원·분석·면접 결과가 누적되어 드러나는 "반복적으로 부족한 역량", "직무 선택 쏠림", "합격 가능성이 높은 방향" 같은 장기 패턴은 사용자 혼자 정리하기 어렵다.

C 영역은 이 세 가지 모름을 각각 **적합도 분석**, **부족 역량·학습·자격증 추천**, **장기 경향·다음 방향 추천**으로 대응한다.

## C 영역의 가치 제안

C 영역은 사용자의 질문을 세 축으로 나누어 답한다.

| 축 | 사용자 질문 | C 영역의 답 | 대표 AI 기능 |
| --- | --- | --- | --- |
| 지원해도 되는가 | "이 공고, 지금 지원할 수준인가?" | 공고-스펙 적합도 점수와 매칭/부족 근거, 그리고 "지원 가능 / 보완 후 지원 / 보류" 최종 판단 | #12 |
| 무엇을 보완하는가 | "지원 전에 뭘 채워야 하고, 뭘 길게 쌓아야 하나?" | 단기/장기로 구분한 부족 역량, 학습 로드맵, 실효성 있는 자격증 추천 | #13 #14 #15 |
| 다음에 어디로 | "여러 번 지원해 보니 나는 어디로 가야 하나?" | 누적 데이터 기반 장기 경향, 반복 약점, 다음 지원 방향, 대시보드 요약 | #16 #17 #18 |

## AI 기능 카탈로그

### AI 처리 원칙(전 기능 공통)

- 분석에는 **어떤 프로필 스냅샷과 어떤 공고 분석 결과**를 사용했는지 반드시 기록한다.
- 적합도 점수는 **설명 가능한 근거와 함께** 제공하고, 단일 점수만 단독으로 노출하지 않는다.
- 부족 역량 추천은 **필수 조건 미충족 / 우대 조건 보완 / 장기 성장 항목**을 구분한다.
- 학습·자격증 추천은 희망 직무·시간·기존 경험을 고려해 **과도한 추천을 억제**한다.
- 장기 경향 분석은 누적 데이터를 사용하되, **개별 원본 데이터의 소유권은 각 담당 영역**에 남긴다.

### 사용자 여정 흐름

```
로그인 → 홈(#18 AI 요약·지원 건별 적합도)
  └─ 지원 건 선택 → 지원 건 상세(B 셸, '적합도' 탭)
        └─ 적합도 분석(#12 점수·근거·지원 판단)
              └─ 부족 역량 → 학습·자격증(#13 #14 #15)
                    └─ 보완 후 → 지원 전략(#17 다음 과제)
                          └─ 경향 보기 → 장기 취업 경향(#16 반복 갭·패턴)
                                └─ 다음 지원 방향(#17) → 다음 지원 건으로 환류

홈 ──요약──> 취업분석 대시보드(#18 #12 #16) ──> 장기 취업 경향
홈 ──비로그인──> 로그인 필요 가드 → 로그인 후 원래 경로 복귀
```

이 여정의 핵심은 **점수 확인에서 끝나지 않는다는 점**이다. 적합도 점수 → 부족 역량 → 학습/자격증 → 전략 → 장기 경향 → 다음 지원 방향이 한 줄로 이어져, 사용자가 "지금 무엇을 해야 하는지"로 항상 귀결되게 설계했다. 홈과 대시보드의 'AI 요약'과 '오늘의 우선순위/이번 주 우선순위'는 이 긴 여정을 압축해 다음 한 걸음을 제시하는 진입점 역할을 한다.

## 데이터 모델 개요

C 영역의 데이터 소유는 `fit_analysis` 테이블을 중심으로 하고, 다른 영역이 만든 데이터는 **읽기 전용 기준 데이터**로 참조한다. 원본 데이터를 수정하지 않는 것이 핵심 경계 원칙이다.

### 소유 데이터: `fit_analysis`

적합도 점수, 매칭 기술, 부족 기술, 추천 학습, 추천 자격증, 지원 전략을 담는다. 실제 구현된 적합도 분석 결과는 다음 구조화된 필드를 포함한다(프런트 타입 기준).

- **점수 계열** — `fitScore`(0~100), `scoreBasis`(점수 산정 근거), `scoreBreakdown`(항목별 획득/만점·설명)
- **비교 계열** — `matchedSkills`, `missingSkills`, `conditionMatrix`(요구조건별 필수/우대 구분 · 충족/부분/미충족 판정 · 근거)
- **판단 계열** — `applyDecision`(APPLY / COMPLEMENT / HOLD + 근거 + 추천 행동), `analysisConfidence`(HIGH/MEDIUM/LOW + 근거)
- **추천 계열** — `gapRecommendations`(역량별 카테고리·우선순위·이유), `certificateRecommendations`, `learningTasks`(학습 과제·실습·예상 기간·완료 상태), `strategy`/`strategyActions`/`next24HourActions`
- **재현성 계열** — `sourceSnapshot`(사용한 공고 분석 ID·공고 revision·공고 분석 시점·프로필 갱신 시점·비교 입력 요약), `model`, `promptVersion`, `status`, `errorMessage`, `createdAt`

재분석 이력(`fit_analysis_history` 성격의 엔트리)은 직전 분석 대비 점수 변화(`scoreDelta`), 새로 획득한 역량, 해소된 갭, 새 갭을 포함해 사용자가 보완 효과를 추적할 수 있게 한다.

| `application_case` | B | 지원 상태와 지원 건 맥락을 참조 |
| `interview_session` | D | 면접 결과를 장기 경향 분석 입력으로 참조 |
| `interview_answer` | D/E | 답변 평가·개선 답변을 장기 경향 분석 입력으로 참조 |
| `user_profile`(스냅샷) | A | 프로필 스냅샷을 적합도 비교 기준으로 참조 |
| `ai_usage_log` | 공통 | C 영역 AI 요청의 사용량을 공통 규약으로 기록 |

### 원본 비수정 원칙

- A의 프로필 스냅샷과 B의 공고 분석 결과를 기준으로 분석하되 **원본 데이터는 수정하지 않는다.**
- B의 공고 조건이 바뀌면 **기존 적합도 분석과 새 적합도 분석을 구분**한다(`sourceSnapshot`의 공고 revision으로 추적).
- D의 면접 결과와 E의 첨삭 결과는 장기 경향 분석의 **입력으로만** 사용하고 원본 수정은 각 담당자에게 맡긴다.

## AI 아키텍처

### mock ↔ 실제 토글

C 영역의 AI 서비스는 **인터페이스 구현 교체** 방식으로 mock과 실제 모델을 전환한다. 도메인별로 `OpenAi*AiService`(실제 LLM 호출)와 `Mock*AiService`(결정적 mock 응답) 두 구현을 두고, `OPENAI_API_KEY`가 주입되면 실제 분석으로 전환된다. 프런트의 적합도 생성 API(`POST /fit-analyses/application-cases/{id}`)는 동일하게 유지되고, 백엔드 구현만 mock ↔ 실제로 바뀌므로 화면·계약을 건드리지 않고 전환할 수 있다.

이 구조 덕분에 데모 빌드는 `VITE_USE_MOCK=true`로 백엔드 없이 자체 완결되며(스토리보드 캡처가 모두 이 빌드 기준), API 키가 발급되면 같은 화면이 그대로 실제 분석을 표시한다.

### 설명가능성 (단일 점수 노출 금지)

C 영역의 가장 중요한 AI 설계 원칙은 **점수 하나만 보여주지 않는 것**이다. 적합도 결과 카드는 점수와 함께 다음을 항상 동반한다.

- **점수 구간 설명** — 숫자(예: 74점)에 더해 "지원 가능 구간입니다. 부족 역량 1~2개를 보완하세요" 같은 의미 안내
- **점수 산정 근거** — 매칭 근거, 부족 근거, 항목별 점수 구성(획득/만점)
- **요구조건-스펙 비교 매트릭스** — 공고 조건별 충족/부분/미충족 판정과 근거
- **신뢰도 배지** — 입력이 부실하면 'AI 제안 · 확인 필요 · 신뢰도 보통/낮음'을 표시해 점수 과신을 막음
- **지원 판단** — 점수와 부족 역량을 종합한 APPLY/COMPLEMENT/HOLD 결론과 추천 행동

### 실패 처리·사용량 로깅

- AI 호출 실패 시 **재시도 가능한 오류 상태**로 표시하고, 관리자 화면에서는 FAILED/FALLBACK으로 강등된 건을 별도 큐로 추린다.
- 모든 AI 요청은 공통 `ai_usage_log` 규약으로 사용량을 기록한다(모델·프롬프트 버전·상태 포함).
- 재생성/재분석은 크레딧 차감과 연동되며(예: 'AI 요약 재생성(크레딧 1)'), 실패 시 크레딧을 보호하고 버튼을 다시 활성화한다.

## 관리자 운영

C 영역은 사용자 기능과 같은 릴리스 완료 기준으로 **운영 가능한 관리자 화면**을 함께 제공한다. 관리자 기능은 빈 메뉴가 아니라 운영자가 실제로 데이터를 확인하고 조치할 수 있는 상태를 뜻한다.

### 분석 통계 (`/admin/analytics`)

전체 회원·분석 완료 지원 건·평균 적합도·이번 달 AI 크레딧을 핵심 지표로 집계하고, **적합도 점수 분포**(구간별 건수·비율)와 **반복 부족 역량**(빈도순 누적)을 시각화한다. 비정상 실행을 추리는 **분석 실패 큐**(FAILED/FALLBACK + 재시도 가능 배지)와 점수-근거 상충·자격증 과다 추천 같은 의심 분석을 추리는 **품질 검수 큐**를 함께 운영한다.

### 적합도 분석 관리 (`/admin/fit-analysis`)

지원 건별 적합도 분석 결과를 점수 구간·상태·재분석 필요·메모 보유 조건으로 필터링해 목록·상세로 조회한다. 선택한 분석의 **매칭/부족 역량, 추천 학습·자격증** 산출물을 검수하고, 점수 등급·상태/메모/재분석 배지로 운영 상태를 표시한다. 운영자는 과도한 추천·잘못된 분석·사용자 문의 대응을 **운영 메모**로 남길 수 있다.

### 프롬프트 운영 (`/admin/prompts/fit-analysis`, `/admin/prompts/analytics`)

적합도 계열 프롬프트(공고-스펙 채점·부족 역량 추천·지원 전략 생성)와 장기분석 계열 프롬프트(취업 경향·다음 액션·직무 준비도)의 **목적·버전·입출력 명세·품질 체크·위험 노트**를 읽기 전용으로 점검한다. 위험 노트에는 환각·과장 방지, 거짓 경력 유도 금지 같은 기준이 포함되어, 운영자가 AI 출력 품질의 기준을 일관되게 확인할 수 있다.

관리자 홈(`/admin/home`)은 위 화면들로 이어지는 **처리 대기 큐 중심 진입점**으로, 적합도 실패·강등·재분석·미분석·장기 분석 실패·최근 7일 신규 분석 건수를 한눈에 보여준다.

## 설명가능성·신뢰 원칙

C 영역은 AI 분석 결과를 사용자가 맹신하지 않고 검토하도록 설계했다.

1. **출처 기록** — 모든 적합도 분석은 어떤 **프로필 스냅샷**과 어떤 **공고 revision / 공고 분석 결과**를 입력으로 썼는지 `sourceSnapshot`에 남긴다. 사용자는 "이 분석은 어떤 데이터를 기준으로 만들어졌나요?"를 펼쳐 확인할 수 있고, 공고나 프로필이 그 이후 바뀌었다면 재분석을 안내받는다.
2. **단기/장기 보완 구분** — 부족 역량은 '필수 역량(지원 전 필수) / 우대 역량 / 장기 성장'으로 카테고리를 나눠, 당장 채워야 할 것과 길게 쌓을 것을 혼동하지 않게 한다.
3. **과도 추천 억제** — 학습·자격증 추천은 희망 직무·시간·기존 경험을 고려해 우선순위를 매기고, 가치가 낮은 자격증은 낮은 우선순위로 안내한다. 관리자 품질 검수 큐가 '자격증 과다 추천' 같은 패턴을 별도로 잡아낸다.
4. **신뢰도 경고** — 입력 데이터가 부실하면 신뢰도를 MEDIUM/LOW로 낮추고 그 이유를 표시해, 점수 자체보다 프로필·공고 입력 보강을 먼저 권한다.
5. **AI 생성물 명시** — 결과 카드에 'AI 제안 · 확인 필요' 배지를 붙여, 이 결과가 검토 대상임을 사용자에게 분명히 알린다.

## 향후 과제

- **실제 LLM 연동** — 현재 데모는 결정적 mock 기반이다. `OPENAI_API_KEY` 주입 시 `OpenAi*AiService` 구현으로 전환되도록 인터페이스를 갖춰 두었으며, 실제 키 발급 후 적합도·장기 경향·대시보드 요약을 실 모델로 전환하는 것이 다음 단계다.
- **분석 정밀도 개선** — 점수 산정 항목 가중치, 요구조건-스펙 매칭 판정의 정확도, 신뢰도 산정 기준을 실제 사용 데이터로 보정한다.
- **장기 경향 데이터 축적** — 장기 취업 경향·다음 지원 방향 추천은 누적 지원·면접·첨삭 데이터가 쌓일수록 품질이 올라가므로, 데이터 축적에 따른 추천 재현성과 안정성을 검증한다.
- **자체 LLM 운영 연계** — 팀 공통 자체 LLM 도입·운영 기준에 맞춰 C 영역 모델의 검증·fallback·산출물 품질 기준을 정렬한다.
