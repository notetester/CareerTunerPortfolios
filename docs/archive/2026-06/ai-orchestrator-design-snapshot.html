AI 오케스트레이터

사용자의 한 줄 요청을 받아 취업 준비 6개 영역(프로필·공고·적합도·자소서·면접·커뮤니티)을 한 번에 자동으로 돌리는 기능. 이 문서는 그 설계와 개발 가이드이고, 특히 F가 만들 인테이크 챗봇을 눈으로 보이게 정리했다.

예시 입력 — "네이버 백엔드 신입 통째로 준비해줘"  →  6개 AI가 알아서 준비 → 면접 시작

📌 이 문서는 2026-06 당시의 구현 상태와 구상을 구분한 기록이다. 구상은 예정 표시를 붙였다. 현재 문서와 같은 내용으로 간주하지 않는다.

1 한눈에 보는 큰 그림

오케스트레이터는 오케스트라 지휘자다. 직접 연주(AI 호출)는 각 담당자의 기존 도메인 서비스가 하고, 지휘자는 "누가 먼저, 누가 동시에, 누가 나중에"만 통제한다. 새 AI 로직을 따로 만든 게 아니라 기존 기능을 묶는다.

2 인테이크 챗봇 만들기 — F 가이드 ★

이 장만 읽어도 인테이크 챗봇을 시작할 수 있게 썼다. 목표 한 문장:

🎯
사용자와 자연어로 대화해서, 결국 AutoPrepRequest 하나를 완성해 POST /api/auto-prep/run/stream 으로 넘기면 끝. 두뇌·오케스트레이터는 그 그릇만 받으면 지금 코드 그대로 돈다. 즉 F와 D의 인터페이스는 AutoPrepRequest 하나로 고정이다.

2.3 두 가지 구현 경로 — 쉬운 길부터

경로 A · 되묻기 API 활용 (권장 시작점)
경로 B · 풀 LLM 에이전트 (고도화)

핵심: 백엔드 /api/auto-prep/intake가 "지금 뭘 더 물어야 하는지"(nextAsk)를 알려준다. 그래서 챗봇은 자체 판단을 거의 안 하고, 서버가 시키는 걸 자연어로 포장해 묻기만 하면 된다. 백엔드 추가 개발이 거의 없다.

// 인테이크 챗봇 한 턴 루프 (의사코드) — 시뮬레이터가 보여준 그 흐름
let req = { query: 사용자_첫메시지 };

while (true) {
  const res = await POST("/api/auto-prep/intake", req);  // AutoPrepIntakeResponse

  if (res.ready) {                          // 슬롯 다 참 → 실행으로
    await POST_SSE("/api/auto-prep/run/stream", req);
    break;
  }
  if (res.nextAsk === "CASE") {           // 서버: 지원 건이 필요해
    챗봇이_묻기(res.message, res.candidates);  // "어느 지원 건?" + 후보 칩
    req.applicationCaseId = 사용자가_고른_id;   // 슬롯 채움
  } else if (res.nextAsk === "MODE") {    // 서버: 면접 모드가 필요해
    챗봇이_묻기(res.message, res.modes);
    req.mode = 사용자가_고른_코드;
  }
}
💡
현재 프론트 AutoPrepChatModal이 정확히 이 방식이다. F의 인테이크 챗봇은 이 칩 선택을 자연어 대화 + 기존 챗봇 메모리로 감싸 더 똑똑하게 만드는 일이다. 출발은 이 루프면 충분하다.

핵심: 기존 커뮤니티 챗봇(CommunityChatAgent)과 똑같은 LangChain4j @AiService 패턴으로 슬롯 추출 에이전트를 새로 만든다. LLM이 대화에서 회사·직무·모드를 뽑고, 지원 건 조회는 @Tool로 한다.

// IntakeAgent — 기존 CommunityChatAgent 와 같은 구조 (예시 골격)
public interface IntakeAgent {
  @SystemMessage(fromResource = "prompts/intake-system.txt")
  String chat(@MemoryId Long conversationId, @UserMessage String userMessage);
}

// IntakeTools — 지원 건 후보를 LLM 에게 제공 (읽기 전용)
class IntakeTools {
  @Tool("사용자의 지원 건 목록을 조회한다")
  List<CaseHit> listCases(Long userId) { ... }
}

// 대화로 모은 슬롯 → AutoPrepRequest 로 변환 후 오케스트레이터 호출
var req = new AutoPrepRequest(query, caseId, mode, null, null);
orchestrator.runStream(userId, req);
⚠️
qwen3 JSON 스키마 함정 — 현재 챗봇이 String을 반환하는 이유가 있다. qwen3:8b는 JSON 스키마를 강제받으면 @Tool 호출을 건너뛰고 JSON만 뱉는 버그가 있다. 그래서 슬롯도 구조화 출력으로 강제하지 말고, String 답변을 받은 뒤 별도 상태 객체에 누적하는 방식이 안전하다.

2.5 단계별 체크리스트

  • 1. 경로 A 루프부터 동작시킨다 — /intake 호출 → nextAsk 분기 → ready면 /run/stream
  • 2. 챗봇 대화 UI에 후보 칩(CASE)·모드 칩(MODE)을 자연어 메시지와 함께 띄운다
  • 3. conversation_id로 대화·슬롯 상태를 이어간다 (새로고침해도 유지)
  • 4. (고도화) IntakeAgent로 "네이버 백엔드 압박면접"을 한 번에 파싱해 되묻기 횟수를 줄인다
  • 5. 자소서·첨부는 선택이니 나중에. 먼저 caseId+mode 확정 흐름을 완성한다

2.6 흔한 함정

🚫
모든 슬롯을 다 받으려 하지 마라. 시스템은 부분 슬롯으로도 돈다(4장). applicationCaseId와 (면접이면) mode만 확정하면 출발. 자소서·첨부까지 다 모으려고 대화를 길게 끌면 사용자가 이탈한다.
🚫
지원 건이 0개인 사용자는 candidates가 빈 배열로 온다. 이때 서버 message가 "채용공고로 지원 건을 먼저 만들어 주세요"다. 챗봇은 이 경우 지원 건 생성 화면으로 안내해야 한다(무한 되묻기 금지).
🏁
완성 기준 — "사용자와 대화해서 applicationCaseId(+면접이면 mode)를 확정 → /run/stream이 떠서 6파트 타임라인이 흐른다." 여기까지면 인테이크 챗봇 1차 완료다.

3 오케스트레이터는 뭘 하나 (참고)

F가 /run/stream을 부르면 그 뒤에서 벌어지는 일이다. 직접 만질 일은 없지만, 화면 흐름을 알면 인테이크를 어디서 끊어 넘길지 감이 온다. ▶ 재생을 눌러봐라.

⚙️ 6파트 의존 그래프 병렬 실행 (SSE)

의존 그래프 — 딱 두 줄

DEPS = {
  "FIT":       ["JOB"],
  "INTERVIEW": ["JOB"]
}

FIT·INTERVIEW는 공고 분석(JOB) 결과가 필요해 JOB 완료 후 시작. 나머지(PROFILE·WRITE·COMMUNITY·JOB)는 의존이 없어 동시 출발. 구현은 CompletableFuture.allOf().thenRunAsync().

SSE 이벤트 순서

plan실행계획 확정
part-start파트 시작
substep세부 작업 보고
part-done파트 완료
done전체 종료
🛟
한 파트가 죽어도 전체는 완주한다. 결과는 DONE / SKIPPED(입력 없음·비활성) / FAILED(예외) 셋. 각 파트를 try-catch로 감싸 예외가 옆으로 번지지 않는다. 이게 다음 장의 근거다.

4 "슬롯이 다 안 와도 작동한다"가 사실인 이유

설계 의도이고 코드로 보장된다. 안전장치가 두 겹이다.

① 인테이크 — 필요한 것만 되묻는다

CASE_REQUIRED = {"JOB","FIT","INTERVIEW"}

계획에 이 파트가 하나도 없으면 applicationCaseId 없이도 바로 ready. 면접이 없으면 mode도 안 묻는다. 필요 없는 슬롯은 처음부터 요구하지 않는다.

② 실행 — 못 채운 파트만 건너뛴다

각 핸들러가 자기 입력이 없으면 스스로 SKIPPED 반환(예: 자소서 없으면 WRITE만 SKIPPED). 한 파트의 SKIPPED/FAILED는 다른 파트에 영향 없음. "할 수 있는 것까지" 준비된다.

사용자가 준 것실행되는 파트건너뛰는 파트
query만, 지원 건 없음, 면접 미포함PROFILE 등 지원 건 불필요 파트JOB·FIT·INTERVIEW
지원 건 있음, 자소서 없음JOB·FIT·INTERVIEW·PROFILE·COMMUNITYWRITE
전부 다6파트 전체없음

5 AI 모델과 폴백 체인 (Haiku)

폴백은 1순위가 실패하면 자동으로 다음으로 넘어가는 안전망이다. 이상적 순서:

자체 LLM
파인튜닝·서빙 → 실패 시 → 1차 폴백 · Claude Haiku
claude-haiku-4-5 → 실패 시 → 2차 폴백 · OpenAI
gpt-5
💸
왜 1차 폴백이 Haiku인가 — Haiku는 소형·고속·저비용 모델이다. 작은 작업에서 큰 모델과 체감 차이가 작으면서 빠르고 단가가 매우 싸다. 그래서 자체 모델이 아직 없는 파트의 1차 폴백을 Haiku로 까는 방향을 잡는다 예정. 호출 비용은 D(정원일)가 부담한다.

할 일 예정 — 현재 Haiku 폴백이 실제로 들어간 건 면접(D) 도메인뿐이다. A(프로필)·C(회사분석)·E(자소서)·F(챗봇)의 OpenAI 전용 경로에 1차 폴백 Haiku를 추가해 "자체 모델이 비거나 죽어도 Haiku가 받치는" 일관된 안전망을 만든다. 자체 면접 생성 모델은 학습 데이터 보강 전까지 화이트리스트가 비어 있어 사실상 Haiku부터 시작한다.

6 세션 자동저장 — ChatGPT식 디벨롭 (C 구상)

원하는 것: "구글 면접 준비해줘"로 만든 준비안을 사용자가 나중에 다시 들어와 그대로 보는 것. ChatGPT가 지난 대화를 왼쪽에 쌓아두는 그것.

현재 상태

오케스트레이터(autoprep) 자체엔 세션 저장이 없다. 모달 닫으면 사라지고, 전용 세션 테이블도 없다. 하지만 F의 챗봇엔 이미 세션 영속화가 있다 — 이게 핵심.

이미 있는 기반

// 20260623_f_chatbot_memory.sql
chatbot_conversation_memory
  conversation_id  BIGINT PK   ← 세션 ID
  messages_json    JSON        ← 대화 전체
  updated_at       DATETIME

conversation_id가 곧 ChatGPT의 "대화 하나"와 같은 개념이다. 응답에 실려 나가고 다음 턴에 다시 보내 같은 세션을 이어간다.

🧩
방향 예정 — 세션 자동저장은 새로 만들 게 거의 없다. 챗봇의 conversation_id 구조를 오케스트레이터로 끌어오면 된다. ① 인테이크 챗봇이 만든 conversation_id를 준비안의 세션 키로 사용 ② 그 세션에 확정 슬롯 + 결과 요약 저장 ③ 재방문 시 복원 ④ 홈에 "최근 준비 세션" 목록. 결국 인테이크(F)·세션(C)·오케스트레이터(D)가 conversation_id + AutoPrepRequest 두 계약으로 맞물린다.
