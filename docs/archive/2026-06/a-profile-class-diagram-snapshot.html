운영 방식

관리자 페이지는 프롬프트 원문을 직접 편집하지 않고, 현재 적용된 평가 기준과 가중치, 검증 항목을 조회한다.

설명 포인트: 운영자는 품질 저하 원인을 프로필 입력 상태, 동의 상태, AI 사용 이력에서 추적관리자

설계 설명 - 회원/인증

Auth 책임과 흐름
F파트 설계보고서 형식처럼 클래스가 맡는 책임과 처리 흐름을 분리해서 설명한다.
책임회원가입, 로그인, 소셜 로그인, refresh token 재발급, 비밀번호 재설정, 휴면 해제, 로그인 감사 로그까지 사용자 인증 생명주기를 담당한다.

핵심 설계 포인트

JWT 구조 유지

기존 JWT 기반 인증 방식을 유지하면서 refresh_token 테이블로 세션 재발급과 폐기를 관리한다.

계정 상태 방어

ACTIVE, DORMANT, BLOCKED, DELETED 상태를 서비스 계층에서 확인해 잘못된 로그인 진입을 막는다.

감사 로그

로그인, 로그아웃, refresh, 비밀번호 재설정 요청을 user_login_history에 남겨 운영자가 추적할 수 있다.

소셜 계정 연결

소셜 제공자 ID와 이메일을 기준으로 기존 계정 연결 여부를 판단해 중복 계정 생성을 줄인다.

발표 설명: 인증은 단순 로그인 성공/실패가 아니라 계정 상태, 보안 정책, 감사 로그, 소셜 계정 연결까지 함께 처리하는 A파트의 기본 보안 영역이다.
설명 기준: AuthController → AuthServiceImpl → Mapper → Token/Log회원/인증 설명

책임사용자의 약관/개인정보/AI 데이터 동의 상태를 관리하고, 희망 직무·이력·기술·자기소개 등 프로필 데이터를 저장·조회한다.

핵심 설계 포인트

동의 게이트

AI_DATA 동의가 없으면 프로필 AI 분석을 실행하지 않도록 서비스 계층에서 강제한다.

사용자 친화 UI

DB에는 JSON으로 저장되더라도 화면에서는 학력, 경력, 활동, 기술을 카드와 태그로 보여준다.

이력 관리

동의는 현재 상태뿐 아니라 agreedAt, revokedAt, source를 남겨 언제 어디서 동의했는지 추적한다.

관리자 재사용

사용자 프로필 응답 구조를 관리자 프로필 조회에도 재사용해 운영 화면과 사용자 화면의 데이터 의미를 맞춘다.

발표 설명: 프로필은 AI 분석의 원천 데이터이므로 입력 편의성, 동의 통제, 관리자 조회 가능성을 함께 고려해서 설계했다.
설명 기준: ProfilePage → ProfileController/ConsentController → Service → Mapper동의/프로필 설명

직무군별 가중치

개발, 마케팅, 디자인, 사무, 교육 등 직무군마다 중요한 평가 기준의 비중을 다르게 둔다.

서버 검증

AI가 반환한 결과를 그대로 믿지 않고 서버에서 스키마와 점수를 검증해 안정성을 높인다.

Fallback

OpenAI 호출 실패, 설정 누락, JSON 검증 실패가 발생해도 규칙 기반 엔진으로 사용자 화면을 유지한다.

교체 가능한 인터페이스

ProfileAiService 인터페이스를 기준으로 추후 Qwen 파인튜닝 모델이나 별도 서빙 API로 교체할 수 있다.

발표 설명: 이 구조의 핵심은 “AI를 호출했다”가 아니라, CareerTuner의 평가 기준과 검증 규칙으로 AI 결과를 통제한다는 점이다.
설명 기준: Consent Gate → JobFamily → AI Service → Validator → Fallback → ai_usage_log프로필 AI 설명

핵심 실행 시퀀스

프로필 AI 분석 요청 기준
발표에서는 이 순서로 설명하면 A파트의 설계 의도가 가장 명확하다.
1
사용자 입력
ProfilePage에서 희망 직무, 학력, 경력, 활동, 기술, 이력서 원문, 자기소개를 입력하고 저장한다.
2
프로필 저장
ProfileController.save() → ProfileServiceImpl.save() → ProfileMapper.upsert() 순서로 user_profile에 저장한다.
3
AI 동의 확인
AI 실행 전 ProfileServiceImpl.requireAiConsent()가 ConsentService.hasCurrentConsent(userId, "AI_DATA")를 확인한다.
4
직무군/가중치
JobFamily.classify(profile)로 직무군을 분류하고 JobFamilyWeightPolicy.weightsFor()로 평가 기준별 가중치를 가져온다.
5
AI 분석
OpenAiProfileAiService가 구조화 JSON을 요청한다. OpenAI 설정이 없거나 실패하면 RuleBasedProfileAiService로 fallback한다.
6
서버 검증
ProfileAiJsonValidator가 summary, extractedSkills, criterionScores, 점수 범위, 평가 기준 누락 여부를 검증한다.
7
로그/응답
ProfileScoreCalculator가 최종 점수를 계산하고, ai_usage_log에 model, token, status, errorMessage를 기록한 뒤 프론트에 결과를 반환한다.
발표 핵심 문장: A파트 프로필 AI는 단순히 프롬프트를 보내는 챗봇 구조가 아니라, 사용자의 AI 데이터 동의를 확인하고, 직무군별 평가 기준과 가중치를 적용하며, AI 응답을 서버에서 검증한 뒤 실패 시 규칙 기반 엔진으로 대체하는 안정적인 평가 구조입니다.
전체 흐름: ProfilePage → ProfileController → ProfileServiceImpl → ProfileAiService → Validator/Calculator → ai_usage_log실행 시퀀스

발표 설명 포인트

처음 듣는 사람 기준 요약
클래스 다이어그램을 설명할 때 강조하면 좋은 설계 포인트

1. Controller와 Service를 분리한 이유

Controller는 HTTP 요청과 응답만 담당하고, 실제 정책 판단은 Service가 담당한다. 이렇게 나누면 화면이 바뀌어도 핵심 비즈니스 로직을 재사용할 수 있다.

  • AuthController: 로그인/회원가입 요청 접수
  • ProfileController: 프로필 저장/AI 분석 요청 접수
  • ServiceImpl: 상태 검증, 동의 확인, DB 저장, 로그 기록

2. AI 응답을 검증하는 이유

AI는 가끔 형식이 깨진 JSON이나 기준에서 벗어난 점수를 반환할 수 있다. 그래서 서버가 응답 구조와 점수 범위를 검증해야 한다.

  • 필수 필드 누락 확인
  • ScoreCriterion enum과 일치하는지 확인
  • rawScore가 0~100 사이인지 확인
  • 실패하면 RuleBased fallback으로 안정성 확보

3. 직무군별 가중치를 둔 이유

개발자와 마케터, 사무직, 의료/교육/물류 직무는 중요하게 보는 역량이 다르다. JobFamilyWeightPolicy는 이 차이를 점수 계산에 반영한다.

  • 개발/데이터: 직무 역량 적합도 비중이 큼
  • 영업/마케팅: 성과 근거와 경험 구체성이 중요
  • 일반 직무: 특정 직무군이 불명확할 때 기본 기준 적용

4. 관리자 화면을 함께 만든 이유

사용자 기능만 있으면 운영자가 품질 문제를 추적하기 어렵다. 관리자 화면은 사용자 프로필 입력 상태, 동의 상태, AI 평가 기준을 확인하는 운영 도구다.

  • 프로필 입력 누락 여부 확인
  • AI_DATA 동의/철회 상태 조회
  • 평가 기준과 가중치 확인
  • 운영 중 품질 저하 원인 추적
