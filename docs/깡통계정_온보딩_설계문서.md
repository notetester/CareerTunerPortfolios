## 0. 한 줄 요약

깡통계정(프로필·지원건 둘 다 없는 신규 유저)이 챗봇 대화만으로 프로필 작성 + 지원건 생성 → 기존 면접 준비 흐름에 합류하게 한다. **팀 결정 = 1안(챗봇 내 처리).** 목적은 *취업 처음인 사람의 막막함 제거* (폼은 빈칸만 보여주지 무엇을/얼마나 쓸지 안 알려줌 — 챗봇은 예시·범위를 주며 물을 수 있다).

## 1. 목표 / 범위

### 대상
- **깡통계정** = `user_profile` 행 없음(또는 비어있음) **AND** `application_case` 0건.
- 프로필만 있고 case 없는 유저, case만 있는 유저는 이 흐름 대상 아님(기존 흐름).

### 하는 것
1. 깡통 감지 → 온보딩 진입
2. 챗봇이 프로필(직무/기술/자소서) 대화로 수집 → A의 save 호출
3. 챗봇이 공고 받아 case 생성 → B의 createFromJobPosting 호출, 보정은 update
4. 둘 다 차면 → 기존 인테이크 면접 흐름(case 선택→mode→ready→autoPrep) 합류

### 안 하는 것 (명시적 제외)
- 프로필이 *이미 있는* 유저의 프로필 수정 (그건 /profile 폼)
- case별 프로필 변형 (측정 확인: case에 스펙 컬럼 없음, 프로필은 유저당 1개 공유)
- 온보딩 중간이탈 복원의 영속화 (MVP 제외 — 휘발 감수, 후속 과제)

## 2. 핵심 측정 사실 (설계의 근거 — 변경 금지 전제)

이 대화에서 코드/DB로 *측정 확정*한 것들. 설계는 이 위에 선다.

| 사실 | 측정 결과 | 설계 함의 |
|---|---|---|
| **user_profile 카디널리티** | **1:1** (user_id UNIQUE `uk_user_profile_user`, 단일 반환, 5행/5유저) | 게이트 `findByUserId==null` 유효. 다중 프로필 없음 |
| **프로필 저장** | `ProfileService.save` = **전체 덮어쓰기** (12컬럼 VALUES() upsert) | 부분저장 불가 → 모아서 한 번. 프로필 있는 유저 진입 시 기존값 날아감 → 게이트 필수 |
| **프로필↔슬롯 관계** | **별개 경로.** 인테이크 슬롯(caseId/mode/originalQuery)은 프로필 안 읽음. autoPrep가 user_id로 직접 조인 | 슬롯에 프로필 prefill 안 됨. 챗봇이 프로필 받아 *저장*해야 함 |
| **스킬추출 AI** | 입력 = AuthUser만 (자소서 텍스트 인자 **안 받음**). 이미 채워진 프로필에서 KNOWN_SKILLS *사전 매칭* | ⚠️ **깡통엔 무용** (빈 프로필 = 뽑을 소스 없음). "자소서 붙여넣기→스킬 자동추출" **불가** |
| **case 생성 입력** | 공고 **원문 텍스트**만 받음 (구조화 회사명/직무 불필요) | 챗봇이 공고 원문 던지면 case 생성됨 ✓ |
| **case 회사명/직무 파싱** | **규칙기반**(정규식+라벨). 실패 시 "기업명/직무명 확인 필요"로 남음 | 파싱 실패 = 유저가 채워야(결정성). AI에 떠넘기지 말 것 |
| **자소서(coverLetterText)** | **per-request.** case/프로필에 저장 안 함. autoPrep 요청에 실어보냄 | 온보딩에서 받아도 저장 고민 없음. 없어도 FIT/면접 돌아감 → 선택 항목 |
| **FIT 입력** | `job_analysis` + `user_profile`(skills/desired_job). 빈 프로필 → score 10 바닥(센티넬 아님, 0매칭 정상) | FIT 의미있으려면 **skills 필수**. 직무 없으면 confidence -7 |
| **인테이크 메모리** | `maxMessages=40` 슬라이딩 윈도우, 초과분 evict | 수집 데이터를 *메모리에 의존하면 밀려남* → **슬롯에** 모을 것 |
| **certificates(FIT)** | Mock 점수 미사용·실LLM만 프롬프트(신뢰도 -8). 로컬=Mock이라 무용 | 데모(Mock)에선 무의미 → 수집셋 제외 |
| **자소서 소비자** | WRITE는 coverLetterText만(UI 없어 항상 SKIP), RESUME 면접도 자소서 미로드, self_intro 미사용 | 자소서 = 기능적 소비자 0 → (c) 폐기 |
| **공고 파싱 타이밍** | createFromJobPosting=비동기(반환 시 회사명 DEFAULT). 워커 `@Scheduled` 5초 폴링. status QUEUED/RUNNING→SUCCEEDED | (d)는 직후읽기 X → status 폴링으로 완료 대기 후 읽기 |

## 3. 도메인 경계 / 합의 (★호출만, 코드수정 0)

온보딩은 **두 남의 도메인의 write 서비스를 호출**한다. 핵심 안전선: **남의 메서드를 *부르기만* 하고 *고치지 않는다.* 시그니처만 같으면 안 깨진다.**

| 도메인 | 소유자 | 챗봇이 부르는 것 | 합의 방식 |
|---|---|---|---|
| **프로필** | **A (신성륜)** | `ProfileService.save(authUser, request)` — 기존 메서드, 전체 저장 | **통보**(게으름): "온보딩 챗봇이 save 호출함. 네 코드 안 건드림, 기존 메서드만. 시그니처 바뀌면 알려줘" |
| **case** | **B (신상훈)** | `createFromJobPosting(userId, req)`, `update(userId, id, req)`, `getLatestJobPostingExtraction(userId, caseId)` (read) — 전부 @Service 내부호출 가능, userId 인자 | **협업메모**(친함): "case 온보딩에서 createFromJobPosting·update 부를게. 공고원문 넘겨 생성, '확인 필요' 뜨면 update로 채움. 네 공유경계(revision 등) 깨는 거 있으면 말해줘" |

> **B의 공유경계 책임**(문서 311-312): C·D·E가 참조하는 분석을 깨지 않게 지원건ID·공고 revision 유지 책임은 B. 챗봇이 case 만들 때 이 가정을 어기면 하류 분석이 깨짐 → B가 점검하게 메모로 알림.

> **schema.sql 안 건드림.** 팀장 영역 회피. (팀장이 영역침범 허락했으나 — 휘발로 충분하므로 *쓰지 않는다.* 컬럼 추가하면 A 도메인 SELECT* 2곳도 건드려야 함)

> **✅ C·D·E 공유경계 점검(2026-06-28, 측정·코드 변경 0)**: 챗봇이 만든 case는 폼 case와 **데이터 형태 동일**(같은 createFromJobPosting→워커→runAfterExtractionPass→revision 경로). C·D·E는 case 컬럼을 직접 가드하지 않지만 **트리거 시점으로 보호됨** — 자동 FIT/INTERVIEW/JOB/COMPANY는 전부 `runAfterExtractionPass` 안에서만 돌고, 워커는 **추출 SUCCEEDED+REVIEW아님**일 때만 이를 호출(`ApplicationCaseExtractionWorker`). 따라서 ⓐ추출 전 반쪽 case 자동노출 X, ⓑREVIEW_REQUIRED case는 자동분석 안 됨(사용자 리뷰 후), ⓒ**TEXT는 revision 안 올림**(`shouldSaveExtractedPosting=false`)→분석 일관. **함정(챗봇이 만든 게 아닌 기존 부채, (d)가 안 건드림)**: 수동 경로(`InterviewServiceImpl.generateQuestions`·`FitAnalysisServiceImpl.generate` 직접호출)는 가드 0 + DEFAULT 회사명을 LLM 프롬프트에 그대로(품질저하). → **(d)는 분석을 직접 트리거하지 않음**(자동 파이프라인/(f) autoPrep에 위임)으로 이 선을 안 넘는다. revision 무효화 없음은 B 기존 이슈(PDF 재업로드 시), TEXT 온보딩과 무관.

## 5. 단계별 설계

### (a) 게이트 / 진입

- **게이트 조건**: `ProfileMapper.findByUserId(userId) == null` (또는 행은 있으나 skills 빈값) **AND** `application_case` 0건.
  - findByUserId는 *행 존재*만 구분(측정). "행 있는데 빈" 케이스는 → /profile로 유도(이미 프로필 페이지 가봤다는 뜻).
  - **순수 깡통(행 없음 + case 0건)만 온보딩.**
- **진입점**: 기존 인테이크 라우터(UnifiedChatRouter)에서 깡통 감지 시 온보딩 분기.
- ⚠️ **결정성**: 깡통 판정은 코드가(DB 조회), 모델 아님.
- 🔧 **구현 시 측정**: findByUserId는 A 매퍼 — 챗봇이 read하는 신규 cross-domain. 호출 경로 깨끗한지 확인. case 0건 판정은 어느 서비스/쿼리로? (B의 list 또는 count)

### (b) 프로필 대화수집 ★핵심

**이 단계가 1안의 존재 이유.** 막막함 제거 = 질문이 예시+범위를 준다.

- **수집 항목 (영속 필요)**: `desiredJob`(직무), `skills`(기술스택). FIT가 이 둘 필요(측정).
- **질문 방식 — 막막함 제거형**:
  - "어떤 직무로 지원하세요? (예: 프론트엔드 개발자, 백엔드 개발자)"
  - "주로 다루는 기술을 알려주세요. 3~4개면 충분해요. (예: React, TypeScript, Spring)"
- **★결정성 선 (발견① 부류 — 위반 시 같은 버그 재생산)**:
  - ✅ 챗봇 = **질문을 친절하게**(예시·범위 안내). 유저가 막막함 덜고 *스스로* 답.
  - ✅ 유저 답 = **그대로 슬롯에 저장**. 챗봇이 가공/해석/확장 안 함.
  - ❌ 챗봇이 답 해석·부풀림 (유저 "리액트 좀" → 챗봇 "React, Redux, Next.js"). **금지.**
  - ❌ 스킬추출 AI로 답 파싱. **쓰지 마**(깡통 무용 + 사전매칭 한계 + 결정성 위반).
- **저장처**: 인메모리 슬롯(SlotState 확장 — IntakeSlotTrace에 desiredJob/skills 필드). **메모리(40창) 아님**(evict됨). 휘발 — 이탈 시 날아감(짧으니 감수).
- **저장 타이밍**: 매 항목마다 save 하지 말 것(전체 덮어쓰기라 앞값 날아감). 슬롯에 *모았다가* (e)에서 한 번.
- ✅ **구현·검증 완료(2026-06-28)**: SlotState 인메모리 확장(`onboardingStep/Job/Skills`) — schema 무관(persistSlot은 caseId-gated라 새 필드 미영속). 직무→기술 단계머신(route `④온보딩:직무/기술/수집완료/대기`), 유저 답을 `recordOnboardingJob/Skills`에 **그대로** 저장(echo 검증: "프론트엔드 개발자"·"React, TypeScript, Spring" 원문 그대로, 부풀림 0). 온보딩은 LLM 40창 미사용(순수 코드)→evict 구조적 무관. 비-깡통 회귀 0(user2 ①③ 정상). (e) save 전 이탈 시 휘발(수용).

### (c) 자소서 — ❌폐기 (2026-06-28, 측정 결과)

**측정으로 폐기 확정. 온보딩에서 자소서 안 받음.** 근거(코드):
- **자소서교정(WRITE) 소비자 0**: coverLetterText를 넘기는 UI 없음(AutoPrepLauncher는 query+첨부만) → WRITE 항상 SKIP. standalone 교정 화면 미구현(`frontend/correction/*`=.gitkeep, CorrectionController 프론트 호출 0).
- **self_intro는 WRITE가 안 읽음**(coverLetterText·첨부만) — 프로필에 저장돼도 교정에 안 흘러감.
- **"자소서 기반 면접(RESUME)"도 자소서 미사용** — 라벨+프롬프트 스타일 지시뿐, interview 도메인이 self_intro/resume_text 미로드(공고원문+모드만).
- **"나중에 /profile에서 교정" 안내는 거짓**(그 경로 미구현)→약속 불가.
- 결론: 온보딩 자소서 수집은 현 코드 기준 **흘려보낼 기능 0**. **수집셋 = 직무+기술 2개로 확정.** (자소서 기반 면접을 실제로 작동시키려면 D가 interview 질문생성에 자소서 주입 — 별도 과제.)

### (d) case 공고입력 + 보정

- **입력**: "지원할 공고를 여기 붙여넣어 주세요" → 공고 **원문 텍스트** 받음.
- **생성**: `createFromJobPosting(userId, {originalText, sourceType})` 호출 → case 자동 생성 + 추출 큐잉.
- **파싱 결과 처리 (★결정성)**:
  - 워커가 규칙기반으로 회사명/직무 파싱. 성공 → 자동 채워짐.
  - **실패 시** companyName = "기업명 확인 필요" / qualityStatus = REVIEW_REQUIRED.
  - 챗봇이 이걸 감지(`getLatestJobPostingExtraction` read 또는 companyName 기본값 체크) → **유저에게 물어서** `update(userId, caseId, {companyName, jobTitle})`로 채움.
  - ❌ AI가 회사명 멋대로 정하지 마(§4-1 부류). 파싱 실패 = 사람이 확정.
- **이게 챗봇이 진짜 가치 더하는 곳**: 공고 붙여넣기는 대화가 자연스럽고, "확인 필요" 보정도 대화로 매끄러움(폼 Step2 UI를 챗봇이 대체).

## 6. 지뢰밭 (측정으로 확인된 함정 — 어기면 이번 세션 실수 재생산)

1. **스킬추출 AI를 깡통에 쓰지 마.** 자소서 인자 안 받음, 채워진 프로필에서만 뽑음, 사전매칭. 깡통엔 무용. "AI 있으니 쓰자"는 측정 무시.
2. **AI에 결정 떠넘기지 마.** 프로필 답 해석, 회사명 파싱 확정 — 전부 발견①/§4-1 부류. 모델은 *질문을 친절히* 할 뿐, *확정은 코드/유저*.
3. **save는 모아서 한 번.** 전체 덮어쓰기라 항목마다 save하면 앞값 날아감.
4. **수집은 슬롯에, 메모리(40창) 아님.** 슬라이딩 윈도우라 초반 답 evict됨.
5. **남의 도메인은 호출만.** A·B 코드 수정 0. 시그니처만 의존.
6. **schema 안 건드림.** 휘발로 충분. 컬럼 추가하면 A SELECT* 2곳도 건드림 + 팀장 영역.
7. **게이트 = 깡통만.** 프로필 있는 유저 진입 시 save 전체덮어쓰기로 기존 프로필 소실.
8. **모르는 칸 채우지 마.** "구현 시 측정" 표시된 곳은 그 단계에서 먼저 측정. 추측으로 짜면 이번 세션 내내 막은 그 패턴.
