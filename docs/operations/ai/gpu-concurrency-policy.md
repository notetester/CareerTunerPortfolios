# GPU 동시성 정책 — 공유 4090 운용 결정과 근거

> **이 문서는 누구를 위한 것인가**
> - **비전공 관리자**: 1~3장만 읽으면 "무엇을, 왜 그렇게 정했는가"를 그림과 비유로 이해할 수 있습니다.
> - **엔지니어**: 4장 이후에 모든 수치·측정 방법·재현 절차·원자료 위치가 빠짐없이 들어 있습니다.
> - 이 결정들은 **모든 AI 도메인(A~F)이 공유하는 단일 GPU**에 영향을 주는 공통 정책입니다.

최종 갱신: 2026-07-06 · 작성 영역: C(적합도 분석) · 공통 인프라 결정(팀 승인)

---

## 1. 한눈에 (3줄 요약)

| 결정 | 값 | 한마디로 |
| --- | --- | --- |
| **서버 동시성** | `NUM_PARALLEL=2`, `MAX_LOADED_MODELS=8` | 여러 AI 모델을 동시에 올려두되(8), 모델당 동시 처리는 적당히(2). 무제한이 최고가 아님을 실측으로 확인. |
| **백엔드 GPU 게이트** | 기본 **꺼짐**(OFF) | GPU 과부하 방지용 안전밸브. 전 도메인에 배선해뒀지만 평소엔 꺼둠, 필요 시 켬. |
| **AI 호출 시간 상한** | 도메인별, 기본 **무제한**(신규) / C·E는 90초·30초 | 한 번의 AI 호출이 GPU를 무한정 붙잡지 않도록 하는 총 시간 상한. 켜고 끌 수 있음. |

**가장 중요한 발견 한 줄**: "설정을 전부 무제한으로 높이면 제일 빠르다"는 직관은 **틀렸습니다.** 실측 결과 특정 값(NP=2)을 넘어가면 GPU 메모리가 넘쳐서 오히려 **최대 5배 느려집니다.**

## 2. 배경 — 왜 이런 결정이 필요한가 (쉬운 설명)

우리 서비스의 여러 기능(이력서 첨삭, 면접 평가, 공고 분석, 적합도 분석 등)은 각각 **자체 AI 모델**을 씁니다. 그런데 이 모델들이 전부 **한 대의 GPU(RTX 4090)**를 나눠 씁니다.

이걸 **한 개의 주방을 6명의 요리사가 공유하는 상황**으로 생각하면 쉽습니다.

| 실제 용어 | 주방 비유 |
| --- | --- |
| GPU 메모리(VRAM, 약 21GB) | 조리대 공간 |
| AI 모델을 GPU에 올림 | 요리사가 조리대에 자기 도구를 세팅함 |
| `MAX_LOADED_MODELS`(MLM) | 조리대에 동시에 세팅할 수 있는 요리사 수 |
| `NUM_PARALLEL`(NP) | 요리사 한 명이 동시에 처리하는 주문 수 |
| 모델 축출 → 재로드("cold reload") | 자리가 부족해 쫓겨난 요리사가 다시 처음부터 세팅(느림) |

문제는 **조리대(GPU 메모리)가 한정**돼 있다는 점입니다. 요리사를 너무 많이, 혹은 한 명이 주문을 너무 많이 받게 하면 조리대가 꽉 차서 누군가 쫓겨나고, 쫓겨난 사람이 다시 세팅하느라 **전체가 느려집니다.** 그래서 "몇 명까지 세팅을 허용할지(MLM)"와 "한 명이 몇 개씩 처리할지(NP)"를 실측으로 정한 것이 이 정책입니다.

## 3. 세 가지 결정 (각각 쉬운 설명 + 핵심 근거)

### 결정 1 — 서버 동시성: `NUM_PARALLEL=2`, `MAX_LOADED_MODELS=8`

**쉽게**: 여러 요리사(모델)가 **동시에 조리대에 있도록 허용**(MLM=8)하되, **한 명이 한 번에 받는 주문은 2개로 제한**(NP=2)했습니다.

- **왜 모델 수는 넉넉히(8)?** 모델을 3개까지만 허용하면, 4번째 도메인이 일할 때마다 기존 모델이 쫓겨났다 다시 올라오길 반복합니다. 실측에서 이 "쫓겨남"이 요청의 56~66%에서 일어나 응답이 **14~22초**까지 느려졌습니다. 넉넉히 허용하면 이 문제가 사라져 **처리량이 약 7배** 좋아집니다.
- **왜 주문 수는 적당히(2)?** 여기서 직관이 뒤집힙니다. 한 명이 주문을 많이(8개씩) 받게 하면 그만큼 조리대 공간(메모리)을 더 차지합니다. 모델 5개 × 8개 주문분 = 메모리 초과 → 다시 쫓겨남 → 붕괴. 실측에서 **NP=8은 NP=4보다 5배 느렸습니다.**

**핵심 근거표** (모델 5개를 동시에 부하, 각 조합 120요청, 전 조합 오류 0):

| 모델당 주문(NP) | 처리량(초당 요청) | 응답 중앙값 | 쫓겨남 비율 |
| --- | --- | --- | --- |
| 1 | 3.9 | 2.6초 | 0% |
| **2 (채택)** | **6.4** | **1.6초** | **0%** |
| 4 | 6.9 | 1.4초 | 0% |
| 8 | 1.3 ⬇ | 3.2초 | **50%** ⬅ 붕괴 |

> **왜 NP=4가 아니라 NP=2인가?** 숫자만 보면 NP=4가 약간 빠릅니다(처리량 +7%). 하지만 NP=4는 GPU 메모리를 20.9GB(한계 21GB의 코앞)까지 씁니다. 위 테스트는 작은 모델 5개였고, 실제로는 6GB짜리 큰 모델(면접 평가용)이 섞이므로 NP=4는 그 순간 한계를 넘겨 붕괴합니다. **NP=2는 성능이 거의 같으면서(−7%) 메모리 여유(16.9GB)가 있어 안전합니다.**

### 결정 2 — 백엔드 GPU 게이트(옵션4): 기본 **꺼짐**, 켤 때 상한 `permits=12`

**쉽게**: 백엔드(서버 프로그램) 쪽에도 "동시에 GPU로 나가는 요청 수"를 제한하는 **안전밸브**를 만들어 전 도메인(A~F)에 달아뒀습니다. 다만 **평소엔 꺼두고**, GPU가 몰릴 때만 켤 수 있게 했습니다.

- 서버 설정(결정 1)과 **독립적인 2차 방어선**입니다. 서버가 이미 대기열로 잘 처리하므로 평소엔 불필요 → 기본 OFF.
- **켜고 끄는 것이 전 도메인 통일**: 전역 스위치 하나 + 도메인별 개별 스위치. 기본 OFF일 때는 **기존 동작과 100% 동일**(성능 영향 0).
- 왜 미리 달아뒀나: 나중에 필요할 때 코드를 안 고치고 설정만으로 켤 수 있도록. "한 도메인만 켜면 진짜 부하 테스트가 안 되므로 전부 연결" 방침.
- **켤 때 몇으로 제한하나(permits=12)**: 동시성 스윕(4-3장)으로 측정한 GPU의 "처리량 포화 무릎점"이 동시성 12입니다. 그 이상은 처리량이 안 오르고 **응답만 느려지므로**, permits=12는 정상 트래픽은 통과시키고 과부하 스파이크의 꼬리지연만 억제합니다. 게이트 자체가 "실제로 캡한다"는 동작은 통합 테스트로 별도 검증됨.

### 결정 3 — AI 호출 시간 상한(총 시간예산): 기본 **무제한**(끌 수 있음)

**쉽게**: 한 번의 AI 호출이 (재시도까지 포함해서) **GPU를 너무 오래 붙잡지 않도록** 총 시간 상한을 두는 장치입니다. 전 도메인에 넣었고, **0으로 두면 무제한(꺼짐)**, 양수면 그 시간까지만 허용합니다.

- C(적합도)·E(첨삭)는 원래 상한이 있었고(90초·30초) 유지. 나머지 도메인은 신규로 넣되 **기본 0=무제한**이라 지금 동작 변화는 없습니다.
- 목적: 공유 GPU가 느려졌을 때 한 요청이 무한정 점유해 다른 도메인을 굶기는 상황 방지. 필요 시 도메인별로 켜기.

### 4-1. NP × MLM 그리드 (2026-07-06, 최종 확정 근거)

- 부하: 서로 다른 도메인급 모델 **5개**(~2GB, C·E·B + correction 후보 2종)에 round-robin, 동시성 12, **120요청/셀**, 전 셀 **오류율 0**.
- `cold` = `/api/generate`의 `load_duration ≥ 1초`(모델이 축출됐다 재로드된 신호). `VRAM` = 측정 직후 `nvidia-smi memory.used`.

| 1 | 3 | 14132 | 27522 | 67 | 0.93 | 10237 |
| 2 | 3 | 13897 | 58320 | 72 | 0.70 | 11965 |
| 4 | 3 | 21797 | 68127 | 79 | 0.37 | 15421 |
| 8 | 3 | 14830 | 50778 | 70 | 0.76 | 22098 |
| 1 | 8 | 2582 | 7697 | 0 | 3.92 | 14732 |

**읽는 법**:
- **MLM=3 행 전체가 참혹함**(p50 14~22초, 처리량 <1): 5모델인데 3개만 상주 가능 → 상시 축출·재로드.
- **MLM=8 행에서 NP는 역U자**: NP1(슬롯 부족) → NP2·NP4(정점) → **NP8 붕괴**. VRAM 열이 원인을 확증 — NP8-mlm8은 22006 MiB로 한계 포화 → 축출(cold 60/120) 재발.

### 4-2. NP 단일모델 부하 (2026-07-06, NP 축 1차 검증)

- 부하: 단일 모델(careertuner-c) A-only 60케이스 × repeat2 = 120요청, 동시성 2·4, 전 조합 오류 0.

| 서버 설정 | c2 (p50 / p95 / rps) | c4 (p50 / p95 / rps) |

| baseline(미설정 → auto **NP=1**) | 1771 / 3949 / 1.03 | 3261 / 6096 / 1.13 |

- **무제약 baseline이 최악**: Ollama가 이 GPU에서 자동으로 NP=1을 골라(단일 슬롯) 대기열이 길어짐 → "무제약이 최선"이라는 가설 **1차 기각**. NP=2가 지연·꼬리지연 최우수.

## 5. 측정 방법 (재현 가능성)

- **하네스**: `ml/career-strategy-llm/scripts/run_concurrency_load_test.py`(단일모델), 멀티모델 드라이버는 부하 스크립트로 별도 실행. A-only 픽스처를 스레드풀로 재생, p50/p95/오류율/처리량/cold-reload 요약만 저장(요청별 raw 미저장 → 커밋 안전).
- **환경**: 공유 RTX 4090, Ollama 0.31.1, SSH 터널(`-L 11435:localhost:11434`)로 이 PC에서 직접 호출.
- **설정 전환**: 각 조합마다 서버 env(`OLLAMA_NUM_PARALLEL`/`OLLAMA_MAX_LOADED_MODELS`) 변경 + Ollama 재시작. 재시작 절차는 아래 6장 주의.
- **안전장치**: 그리드 실행 스크립트는 재시작 전 **비-Ollama GPU 프로세스(팀원 학습 등) 자동 감지 guard**를 돌려, 학습 중이면 중단하도록 함.
- **원자료·상세 문서**: CareerTunerAI(운영 artifact 저장소) `benchmarks/gpu-concurrency-loadtest/`
  (`np-mlm-grid.md`, `permits-concurrency-sweep.md`, `mlm-multimodel-probe.md`, `README.md`, `mmgrid_*.json`·`concsweep_*.json`), 운영 절차는 같은 저장소 `docs/ops/4090_{OLLAMA_CONCURRENCY_TUNING,GPU_BACKEND_PERMIT_GATE}.md`.

## 6. 운영 주의사항 (실측 중 발견한 함정)

1. **`llama-server.exe` 워커 VRAM 누수**: Ollama는 모델마다 별도 `llama-server.exe` 워커를 띄웁니다. 재시작 때 `ollama`/`ollama app`만 죽이고 이 워커를 안 죽이면 **고아 프로세스로 남아 VRAM을 계속 점유**합니다(GPU 사용률 0%인데 메모리만 소진). 실측 초기에 ~20GB가 좀비에 잠겨 "4모델만 상주"로 오측정됐습니다. **재시작 시 `taskkill /IM llama-server.exe`까지 필수.**
2. **재시작 전 학습 프로세스 확인 필수**: `/api/ps`(Ollama 유휴 확인)만으로는 **팀원의 별도 학습(python) 프로세스를 못 봅니다.** 재시작 전 반드시 `nvidia-smi --query-compute-apps`로 비-Ollama GPU 사용자를 확인하고, 있으면 재시작 금지.
3. **공유 GPU 재시작은 사용자 승인 후**: 사용 중인 4090을 예고 없이 재시작하지 않습니다.

## 7. 용어 사전 (비전공자용)

| 용어 | 뜻 |
| --- | --- |
| **GPU / VRAM** | AI 연산 전용 하드웨어와 그 전용 메모리. 여기 4090은 VRAM ~21GB 사용 가능. |
| **모델(model)** | 특정 작업(첨삭·면접평가 등)을 학습한 AI. 각 도메인이 자체 모델을 가짐. |
| **NP (NUM_PARALLEL)** | 모델 하나가 동시에 처리하는 요청 수. 높이면 메모리(KV 캐시)를 더 씀. |
| **MLM (MAX_LOADED_MODELS)** | GPU에 동시에 올려둘 수 있는 서로 다른 모델의 최대 수. |
| **KV 캐시** | 모델이 요청을 처리할 때 쓰는 임시 메모리. NP 슬롯 수만큼 곱해서 잡음. |
| **cold reload(축출·재로드)** | 메모리 부족으로 쫓겨난 모델을 다음 요청 때 처음부터 다시 올리는 것(수 초 소요). |
| **p50 / p95** | 응답 시간의 중앙값 / 상위 5% 느린 응답 기준값(꼬리 지연). |
| **처리량(rps)** | 초당 처리한 요청 수. 높을수록 좋음. |
