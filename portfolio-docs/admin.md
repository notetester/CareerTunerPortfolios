# 관리자 · 운영

CareerTuner의 관리자 기능은 `ADMIN`이라는 한 역할만 확인하지 않습니다. 역할은 관리자 영역 진입의 1차 경계이고, 실제 메뉴·라우트·버튼·API는 도메인별 CRUD 권한으로 다시 판정합니다. 사용자 기능을 만든 담당자가 대응 관리자 API와 화면까지 같은 릴리스에서 완성하는 것이 원칙입니다.

## 권한 모델

정규 권한은 다음 7개 도메인과 4개 동작을 조합한 28개 코드, 그리고 읽기 전용 감사 권한으로 구성됩니다.

| 도메인 | 관리 대상 |
| --- | --- |
| `USER` | 회원, 프로필, 지원 건, 동의 |
| `SECURITY` | 로그인 위험, 차단, 보안 운영 |
| `BILLING` | 결제, 크레딧, 보상, 요금제 |
| `CONTENT` | 커뮤니티, 공지, FAQ, 문의, 광고 |
| `AI` | 분석, 면접, 첨삭, AI 설정·사용량 |
| `POLICY` | 약관, 정책, 런타임 정책 |
| `ADMIN_PERMISSION` | 관리자 계정·권한 자체 |

각 도메인은 `READ`, `CREATE`, `UPDATE`, `DELETE`로 나뉩니다. `DELETE`도 제품 정책상 대부분 soft delete이며, 물리 삭제를 뜻하지 않습니다. 별도로 `AUDIT_READ`가 있습니다.

## 네 겹의 fail-closed 경계

1. `SecurityConfig`가 `/api/admin/**`를 익명·일반 회원에게 차단합니다.
2. 프런트 `AdminRouteBoundary`가 사용자 역할과 정확한 route policy를 확인합니다.
3. 권한 조회가 로딩·실패 상태이면 화면을 먼저 마운트하지 않고 금지합니다.
4. 백엔드 `@RequireAdminPermission`/interceptor가 같은 권한을 API에서 다시 검증합니다.

READ 권한이 없으면 왼쪽 메뉴 탭 자체가 보이지 않습니다. 주소를 직접 입력해도 경계에서 차단되고, CREATE/UPDATE/DELETE가 없으면 해당 버튼과 handler가 함께 닫힙니다. 프런트 숨김은 UX이고 백엔드 검증이 최종 보안 경계입니다.

`SUPER_ADMIN`은 전체 정규 권한을 암묵적으로 통과하며 다른 관리자에게 그룹·개별 권한을 부여하거나 회수합니다. 테스트와 초기 운영을 위해 여러 ACTIVE 최고 관리자 계정을 둘 수 있지만, 일반 관리자가 자기 권한을 상승시키거나 마지막 최고 관리자를 비활성화하는 변경은 guard가 막습니다.

## 관리자 정보 구조

55개 route를 평면 메뉴로 늘어놓지 않고 운영 목적별 7개 그룹, 검색, 아코디언, 모바일 drawer로 정리했습니다. 대표 화면은 다음과 같습니다.

- 운영 홈과 대시보드
- 회원·프로필·지원 건·동의
- 로그인 위험·MFA 정책·활동/이메일/보안 감사
- 결제·크레딧·리워드·요금제
- 공고·기업·적합도·면접·첨삭·AI 사용량·프롬프트
- 신고·공지·FAQ·문의·가이드라인·광고
- 약관·정책·런타임 설정·관리자 권한

공지/FAQ 작성처럼 목록과 mutation이 분리된 route도 CREATE 권한을 별도로 요구합니다. 회원 생성·수정·삭제 역시 서로 다른 exact 권한을 사용합니다.

## 공통 목록과 감사

관리자 목록은 공통 grid 계약으로 페이지, 정렬, 필터, bulk action, export를 일관되게 처리합니다. 서버는 정렬 컬럼과 한도를 화이트리스트로 제한하며, 프런트 입력만 신뢰하지 않습니다.

회원 상태, 관리자 권한, 결제·신고 조치처럼 민감한 변경은 행위자, 대상, 전후 값, 사유를 남깁니다. 회원을 비활성화하면 refresh token도 폐기해 다음 갱신을 기다리지 않습니다. 감사 조회는 업무 데이터 수정 권한과 분리된 `AUDIT_READ`를 사용합니다.

## DB와 마이그레이션

관리자 권한 카탈로그는 DB patch에서 canonical 29개를 검증하고 그룹 항목을 동기화합니다. 기존 legacy 행은 호환을 위해 즉시 물리 삭제하지 않지만, 권한 판정과 새 배정은 canonical 코드만 사용합니다. patch는 적용 이력과 checksum을 남겨 이미 배포된 SQL을 바꾸지 않습니다.

## 데모 경계

공개 mock은 사용자와 관리자 persona를 명시적으로 나눕니다. 정상 운영 장애 fallback에서 관리자 persona를 허용하지 않고, 공개 static demo일 때만 관리자 화면을 체험할 수 있습니다. mock API도 중앙 권한표를 거쳐 실제 route와 같은 방식으로 금지됩니다.
