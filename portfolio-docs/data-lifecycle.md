# 데이터 생명주기와 회색지대

CareerTuner의 핵심 단위는 지원 건이며, 프로필·공고 revision·분석·면접·첨삭·결제·파일 데이터가 그 주변에서 연결됩니다. 기능별 CRUD만 통과해도 연결 경계가 어긋나면 고아 파일, 오래된 분석, 중복 과금 같은 문제가 남습니다. 그래서 다음 생명주기를 함께 검증합니다.

## 생성과 연결

- 프로필 저장은 현재 행과 버전 스냅샷을 함께 갱신합니다.
- 공고 업로드는 파일 메타데이터, OCR 결과, 검수 상태와 분석 provenance를 분리합니다.
- 분석 결과는 사용한 프로필·공고 revision을 기록해 입력이 바뀐 뒤 오래된 결과를 구분합니다.
- 면접과 첨삭은 지원 건과 원본 답변을 소유권 검증된 식별자로 전달합니다.
- AI 사용량은 결과 저장과 과금 정산을 요청 키로 묶어 재시도 시 중복 차감을 막습니다.

## 수정과 재실행

- OCR 검수 후 확정된 텍스트가 downstream 분석 입력이 됩니다.
- 이미 성공한 추출도 사용자가 다른 모델을 선택해 strict 재추출할 수 있습니다.
- 면접 질문 재시도는 최초 모델을 기본으로 제시하되 모델 선택을 잠그지 않습니다.
- 모델, 실제 provider, fallback 경로를 결과와 함께 남겨 같은 화면의 결과 차이를 설명합니다.

## 삭제와 복구

- 사용자·관리자 데이터 삭제는 원칙적으로 soft delete이며 기본 조회는 활성 행만 반환합니다.
- 재가입·복구는 삭제 상태를 무시한 중복 행 생성 대신 기존 계정의 명시적 재활성화 경로를 사용합니다.
- 파일 삭제는 DB 메타데이터와 저장소 객체의 책임을 나누고, 임시·고아 면접 미디어 정리 경로를 둡니다.
- 챗봇 대화는 목록·불러오기·명시적 삭제를 제공하며 계정 범위를 넘지 않습니다.

## DB 변경

운영 DB 변경은 `schema.sql`을 통째로 덮는 대신 순서가 있는 patch로 적용합니다. 적용 이력과 SHA-256 checksum을 기록하므로 한 번 적용된 patch의 내용을 몰래 바꾸지 않고 새 patch로 정정합니다. 빈 DB 생성, 기존 DB 증분 적용, 재실행 멱등성을 각각 검사합니다.
