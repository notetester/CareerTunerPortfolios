# C_FIT_EXPLAIN 평가 결과 (1차, 2026-06-22)

> 4090 실측. 골든셋 12건. **LoRA**(`careertuner-c-career-strategy-3b`) vs **base**(`qwen2.5:3b-instruct`), 동일 system 프롬프트(`FIT_EXPLAIN_SYS`)·동일 하니스. 결과 원본 JSON은 `out/eval/`(미커밋), 여기엔 요약만.

## 0. ★한 줄 결론 (정직하게)
- **contract(기계검증 계약) 기준 base가 LoRA를 근소 우위**(12/12 vs 10/12). LoRA 실패 2건 = **콜드스타트 타임아웃 1(인프라)** + **CJK 누출 1(품질)** → 모델 품질 실질 11/12.
- 즉 **파인튜닝이 contract 준수를 개선하지 못했고, CJK 누출을 1건 유발했다.** "우리 LoRA가 더 낫다"는 주장은 이 데이터로 불가.
- ★ 단, **이 평가는 contract만 측정**한다. LoRA가 학습한 **설명 품질(한국어 도메인 적합성)은 미측정** — 그건 LLM-judge 정성평가가 필요(다음 라운드). 그 전엔 LoRA의 가치/무가치를 단정하지 않는다.
- latency 2배 헤드라인은 **콜드스타트 artifact**다(아래 §3).

## 3. latency 해석 — 콜드스타트 artifact (헤드라인 보정)
LoRA 케이스별(ms): **120,041 · 102,136** · 26,231 · 24,814 · 23,948 · 27,172 · 24,037 · 28,629 · 19,829 · 19,635 · 30,869 · 21,497
- **첫 2건(120s 타임아웃·102s)이 콜드 모델 로드**(C 모델이 GPU에 상주 안 한 상태에서 첫 호출이 로드+생성; D `interview-3b` 6.4GB와 VRAM 경합 정황). 나머지 10건(warm)은 **~20-31s**.
- base는 10,475~26,758ms로 콜드 spike가 작다(LoRA 타임아웃이 이미 GPU/Ollama를 데워둔 영향 가능).
- **warm 비교**: LoRA median ~25s vs base ~18s → LoRA가 ~30-40% 느림(2배 아님). 잔차 원인 후보: 출력 길이(파인튜닝이 더 장황) 또는 quant 차이. → **토큰수 포함 격리 재측정 필요.**
- ※ 3B가 4090에서 warm ~20s 자체가 느림 → **공유 GPU 경합/모델 스왑 정황**(C+D 동시 상주). 단독 로드 시 더 빠를 것.

## 4. CJK 누출 (LoRA 1/12, base 0)
- `case-it-clear-gap-001`. 파인튜닝이 base의 한국어 청결도를 미세 저하시킨 정황(오프도메인 스모크에서도 CJK 관찰됨). 추정 원인: 합성데이터 미세 오염 또는 LoRA가 base 분포를 약간 이동.
- **다음 라운드에서 하니스가 원본 출력을 저장**하도록 보강해 실제 누출 토큰을 확인·디버그한다.

## 5. 정직한 재프레임 (포트폴리오)
이 결과는 약점이 아니라 **성숙도 신호**로 쓴다:
> "파인튜닝해서 더 좋다"가 아니라, **"자체 파인튜닝·배포·정량평가까지 수행했고, 측정 결과 이 bounded task에선 강한 instruct base + 정밀 system 프롬프트가 이미 계약을 충족함을 확인했다. 따라서 신뢰성의 레버는 모델 lift가 아니라 모델-무관한 **뉴로-심볼릭 경계 + OSS→OpenAI→Mock 폴백**임을 데이터로 보였다."**
- 이건 측정하고 oversell 하지 않는, senior한 결론이다.
- 미해결 가치 질문: **LoRA가 설명 품질(유용성·구체성·톤)에서 base보다 나은가?** → §6-1로 판정.

## 6. 다음 액션 (우선순위)
1. **LLM-judge 정성평가** — 동일 케이스에서 LoRA vs base 설명을 pairwise(유용성·구체성·근거충실·한국어 톤). + 하니스에 **원본 출력·토큰수 저장**. → LoRA의 실제 가치 판정(이게 LoRA 존속의 근거).

3. **CJK 누출 디버그** — 원본 출력 확보 후 데이터/프롬프트 원인 추적.
